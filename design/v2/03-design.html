A · The system

A1 What & why — goals

The Investigation Workbench — a governed, in-house system for production operations. At its core, one engine runs a versioned playbook through gated, audited phases; it builds a graph of the situation, reasons over it with the model, calls governed capabilities (never tools directly), and a human approves every write. The first domain is incident triage; the same engine runs other domains by loading a different playbook.

GoalHow the design meets it
No vendor lock-inthe engine calls abstract capabilities; a source can change without touching the engine (Part C).
No domain lock-ineverything domain-specific is in the playbook (types · phases · outputs); the engine, shapes, registry, governance, storage and console are universal (A2).
Governed at the boundaryevery call passes allow / ask / deny; every write is human-approved; closing is human (B5, C3).
Durable + reversiblecheckpointed, exactly-once, resumable; every write carries a rollback (B5, E4).
Evidence-based + auditableevery fact carries an evidence_ref; the Step journey is the audit trail (D, F2).
GradualAmelia runs alongside, wrapped as a governed capability, dialled down per proven slice (G1).

B · The engine on LangGraph
This is the heart of the design. The engine is a LangGraph application. It (B1) loads a playbook and compiles it into a state machine, (B3) runs each phase as a model-driven agent loop that (B4) calls the capability layer in the loop and folds results into the graph, (B5) pauses at the gate and checkpoints, and (B6) moves between phases by conditional edges — looping for confidence and backtracking on failed recovery. B7 is a full trace.

B1 Loading the playbook → compiling the run

On startup (and on every playbook publish), the engine loads the playbook row from Postgres and compiles it into a LangGraph StateGraph. The playbook is data; the compiled graph is what runs.

  1. Parse + validate the front-matter: phases[] (each with id · effect · needs · output · min_confidence?), graph_schema (node_types, edges, facts, labels, conventions), output_schemas, defaults, unknown_access, error_handler.
  2. Register the domain types from graph_schema (so the graph builder + UI know the vocabulary) and the output_schemas (JSON Schema, used to validate each phase output, D3/E).
  3. Build the state machine: one node per phase, edges in declared order, conditional edges for loop/backtrack (B6), interrupt_before on writes (B5), and the checkpointer.
def compile_run(playbook):                          # playbook = the Postgres row, pinned @version
    pb = parse_frontmatter(playbook.body_md, playbook.phases, playbook.schemas)
    validate(pb)                                     # phases, graph_schema, output_schemas well-formed
    # ── COMPILE-TIME INVARIANTS (fail at LOAD, not mid-run) ──────
    # 1. every write phase MUST set gate_writes=true (else invoke mid-phase would see
        #    access=ask but no interrupt_before to pause → Denied exception uncaught)
    for p in pb.phases:
        if p.effect == "write" and not p.gate_writes:
            raise ValueError(f"write phase {p.id} must set gate_writes=true")
    # 2. every phase.output name must be a registered output type (else mid-run validation fails)
    for p in pb.phases:
        if p.output not in OUTPUT_TYPES:
            raise ValueError(f"phase {p.id}: unknown output type {p.output}")
    ──────────────────────────────────────────────────────────
    register_types(pb.graph_schema)                  # node_types / edges / facts → graph builder + UI
    register_schemas(pb.output_schemas)              # JSON Schema per phase output
    layer.unknown_access = Access(playbook.unknown_access)  # reconcile playbook policy into layer

    g = StateGraph(RunState)
    for ph in pb.phases:
        g.add_node(ph.id, make_phase_node(ph, pb))   # B3 — the agent loop for this phase
    g.set_entry_point(pb.phases[0].id)               # assess
    wire_edges(g, pb.phases)                          # sequential + conditional (B6)
    gate_writes_phases = [p.id for p in pb.phases if p.gate_writes]
    return g.compile(checkpointer=PostgresSaver(...), # B5 — durable, resumable
                     interrupt_before=gate_writes_phases or None)  # pause before each gate_writes phase

Because the phases, types and contracts all come from the playbook, compiling a different playbook yields a different engine behaviour with zero code change — the domain-neutral promise, realized at compile time.

B2 The run state

One run = one LangGraph thread_id. The state is the working set every phase reads and writes; LangGraph checkpoints it after each step.

class RunState(TypedDict):
    subject:        SubjectRef            # {domain, id, kind} — the thing under investigation
    playbook_ref:   str                   # "incident-triage@1.0.0" (pinned for reproducibility)
    graph:          InvestigationGraph    # nodes / edges / facts — SHARED across all phases (Part D)
    phase_records:  Annotated[list, append]   # one PhaseRecord per phase entered (D2)
    messages:       Annotated[list, append]   # the conversation (operator + agent)
    current_phase:  str
    pending_decision: dict                # {decision, actor} — carried into resumed gate phase; consumed once recorded
    status:         str                   # running|waiting_approval|waiting_input|done|failed — routed by B6 router

The graph is the shared world (it persists and grows across phases); each PhaseRecord is one phase's journey + typed output. On phase completion the engine writes through to the Mongo incident document (D6); the live state stays in the checkpointer until close.

B3 The phase loop — plan → execute

Each phase node is a model-driven agent loop, bounded by the phase's needs (which intents it may use) and effect (read-only or write). The model plans, then loops — pick a need, call a capability, fold the result into the graph, log a Step, update the output — until the typed output is sufficient (its required fields are present + validate).

def make_phase_node(ph, pb):
  def node(state):
    rec = PhaseRecord(id=f"{state.subject.id}:{ph.id}:{attempt(state, ph)}",
                      phase=ph.id, goal=ph.goal, plan=model.plan(state, ph), steps=[])
    while not sufficient(rec.output, pb.output_schemas[ph.output]):
        need, args = model.next_action(state, rec, allowed=ph.needs)   # 1 · model picks an INTENT
        result     = capability_layer.call(need, args, effect=ph.effect, state=state)  # 2 · Part B4 / C
        fold_into_graph(state.graph, result)                            # 3 · nodes / facts / edges + side-effects
        rec.steps.append(Step(kind="tool_call", capability=result.cap_id,
                              input=args, result=result.data,
                              touched=result.node_ids, evidence=result.evidence))
        rec.output = model.update_output(rec, state.graph)              # 4 · build the typed output incrementally
        if model.wants_operator(rec): return pause(state, "waiting_input")   # ask a question
    validate(rec.output, pb.output_schemas[ph.output])                  # 5 · contract check (E)
    state.phase_records.append(rec); write_through_to_mongo(state)
    return state
  return node

Steps are emergent, not scripted — the planner (model) decides the next need each iteration given the current graph state and phase output. The loop ends when sufficient(output, output_schema) is true (all required fields present + validates). The output is refreshed after each tool call via planner.update_output(). Steps record tool calls (kind=tool_call), reasoning (kind=reasoning), gate decisions (kind=decision, recorded FIRST on re-entry if approved), and other events — the Step list is the audit trail.

B4 Running the capability layer in the loop

Step 2 above — capability_layer.call(need, …) — is where intents become governed, real calls, every iteration of the loop. The layer never lets the model touch a tool directly.

def call(need, args, effect, state):
    caps = resolve_intent(need, effect)        # C4 — match DeclaredCapabilities by intent + effect
    cap  = rank(caps)[0]                        # by trust, freshness, fit
    dec  = govern(cap, playbook_of(state))      # C3 — effect × access (allow | ask | deny)
    if dec.access == "deny":  raise Denied(cap)             # model must find another path / escalate
    if dec.access == "ask":   return request_gate(cap, args, state)   # B5 — pause for approval
    key = input.get("idempotency_key") if isinstance(input, dict) else None
    if key is not None and key in cache: return cache[key]  # exactly-once on WRITE
    adapter = adapter_for(cap.provider)        # bind by provider id
    if adapter.kind != provider(cap).kind: raise Denied(f"adapter kind mismatch")  # C1 binds by kind
    raw = adapter.invoke(cap.id, input)        # native MCP | dynamic-MCP | A2A dispatcher
    if key is not None and effect == Effect.write: cache[key] = raw  # cache ONLY writes
    return Result(cap_id=cap.id, data=raw,
                  node_ids=raw.touched,         # which graph nodes this read/wrote
                  evidence=raw.evidence_refs)   # deep links — fold into Fact.evidence_ref + output
In the loopWhat happens
resolve_intentneed (e.g. metrics) → DeclaredCapabilities tagged with that intent whose effect is compatible with the phase's effect. The filter uses the authoritative effect — policy.effect ?? effect_hint ?? unknown, the SAME value govern() enforces — not the raw effect_hint. So a capability whose hint says read but whose policy corrects it to write is still excluded from a read-only phase: a read phase provably never even ranks a write/unknown cap (FR12/AC1).
governCalls govern(cap, playbook): resolves cap.effect_hint × CapabilityPolicy.access at call time. read → usually allow (invoke, audit); write → ask (pause the run, wait for operator decision); unknown effect → playbook.unknown_access policy (allow|ask|deny per policy, reconciled into layer at compile B1).
invokeIf access=ask and not yet approved: raise NeedsApproval (pause, no invoke); if deny: raise Denied; else dispatch. invoke() is the exactly-once chokepoint (FR5): a write carrying an idempotency_key is dispatched at most once — a retry or crash-replay with the same key returns the cached result, never double-applies. The bound adapter's kind must match its provider's declared kind (native MCP · dynamic-MCP-wrapped API · A2A specialist agent), else Denied; it returns the raw result with evidence_refs.
foldEach capability result is mapped to graph mutations (nodes/facts/edges) by a per-capability fold-adapter registered in FoldRegistry. The adapter is the ONLY place a source's shape meets graph schema. Fold updates the in-memory graph and records touched node ids + evidence refs in the Step.

B5 The gate, checkpoint, resume

The gate. At compile, every gate_writes phase (effect=write, gate_writes=true) is named in interrupt_before. When the run enters such a phase, the LangGraph interrupt pauses BEFORE the phase node executes, leaving next=[phase_id] visible to the API (status=waiting_approval). The engine returns the phase spec + output so far; the operator approves | refines | denies via the API.

Resume path. The operator's decision {decision, actor} is posted to the API, which calls engine.resume(thread_id, decision). The engine:

  1. Updates the checkpoint with pending_decision: {decision, actor} (idempotent on decision content + idempotency_key).
  2. Invokes the LangGraph thread, which re-enters the paused phase.
  3. The phase node's FIRST step records Step(kind=decision, decision=…, actor=…) before any tool calls — the most accountability-critical event (D2).
  4. Then the phase proceeds with the approved writes; pending_decision is consumed/nulled after the Step is appended.
  5. LangGraph checkpoints after every step (durable + resumable).

Exactly-once. The decision is idempotent — a double-approve on the same gate with the same decision_id re-runs deterministically (same inputs → same result). Crash mid-phase = resume from the last checkpoint; transient failures within the phase retry per defaults.retry.

B6 Phase transitions — conditional edges

Phases run in declared order, but the edges are driven by each phase's own metadata — never hardcoded phase ids. This is what makes the run adaptive and re-enterable, and what keeps the engine domain-neutral: a differently-named playbook (different phases, different loop/backtrack points) compiles to the right graph with zero code change.

assess→root-cause↺ if conf < min→remediation→verify↺ if not recovered → root-cause→END
A phase that declares……compiles to this edge (the contract)
effect: write
+ gate_writes: true		Pauses before phase executes. The phase id is added to interrupt_before at compile. When the run reaches this node, LangGraph halts with next=[phase_id] visible to the API. On resume with the operator's decision, the phase re-enters, records the decision Step (D2 audit: who approved + when), then proceeds with tool calls. Every write phase MUST declare gate_writes=true — validated at compile; otherwise writes would trigger access=ask mid-node, uncaught.
min_confidence: x
(a loop phase)		Re-enter the same phase while top_confidence(state) < x; else advance. Re-entries are capped at 5 attempts — once hit, advances anyway (inconclusive evidence proceeds; next phase may escalate) rather than infinite recursion. A phase with min_confidence: null never loops.
backtrack_to: <earlier-phase>
(a backtrack / terminal phase)		If the phase's recovered output field is true → END (close); else re-enter the named earlier phase to gather more evidence. backtrack_to must reference a phase that appears earlier in the phases list — validated at Playbook.model_validator (compile load time), so a forward or missing target fails immediately, never silently mid-run.
neitherAdvance sequentially to the next declared phase in the phases list; if last, route to END.

Halt on pause / failure. The route_phase() function (the compiled conditional-edge router) checks status first: if status ∈ {waiting_input, failed} it returns halt → END, surfacing that status to the API. Only if status is running does routing proceed to loop/backtrack/advance logic. This guard ensures a paused or failed phase never silently advances. The router returns exactly one token — halt · loop · advance · backtrack · close — which LangGraph maps to a destination node (END, self, next, or backtrack_to).

A re-entered phase opens a new attempt (…:root-cause:2); the graph and prior records are retained as context, so the second pass is better-informed. A human performs the final close after the terminal phase reaches END.

Related incidents — human-guided, one focus. A run investigates one incident in its own focus. On launch (and as it looks out during its cycle), the engine checks for an existing session on a similar incident; if it finds one it asks the operator — link this as related (into a cluster via member_of), or open a new triage. It never auto-merges and never runs "at cluster scope" — each incident keeps its own session, phases, and closure; a linked cluster just lets the shared cause/fix propagate.

B8 The live session — shared, multi-user, on stateless servers

Two layers, not one. The record (graph + phase records + outputs) is the durable audit + display — the Mongo read-model (Part D). The live chat session (the conversation + the in-flight run) is a runtime the data model deliberately left to "LangGraph owns it" (D4). This section designs that runtime. The rule that keeps it simple: state lives in the stores; the session is a thin layer = a lock + a polled event log.

B8.1 · Lifecycle — free chat → shared session

free chatidentity + subject →related-incident check→create or join→shared session
StateWhat it is · cost

B8.2 · Problem 1 — the live run on stateless, load-balanced servers

One incident = one run, but requests land on any server and operators share it. The crux: a step that is in-flight (the agent mid-thought, streaming, calling a tool) lives in one server's memory — it isn't checkpointed yet.

MechanismHow
Serialize the runadvancing a session takes a per-session lock — a lease with heartbeat (a Postgres advisory lock on the subject id in production, or in-memory in-process for mocks). The holder is the transient run-owner and drives the agent loop one step at a time. No second run can start for the incident. The lease has a 30s TTL; a slow tool keeps it by calling heartbeat(session_id, token, ttl=30); a dead owner's lease expires and is stolen by another server, which resumes from the checkpoint.
Input during an in-flight stepa 2nd operator's message is queued (a per-session input queue), drained after the current step or at the next gate — merged into the one run, never spawning a second.
Reads & chat go anywhereonly advances need the lock; reads (render graph/phases) hit the Mongo read-model and chat is polled from the event log (B8.3) — both from any server. No sticky load-balancing.
Crash mid-stepthe lock lease expires → another server resumes from the last checkpoint; the lost step re-runs (reads idempotent, writes idempotency-keyed). No work lost, no double-write.

In-memory = only the active step, on the owner, transiently. The durable truth is the LangGraph checkpoint + the read-model — so any server takes over between steps.

B8.3 · Problem 2 — syncing the chat across users

Several operators, possibly on different servers, must see the same conversation, the agent's live stream, and the gate — in real time.

MechanismHow
One event log per sessionan append-only log keyed by the subject. Every event — a user message, an agent message, a gate prompt, a gate decision, a phase/graph delta — is appended with a per-session sequence number (starts at 0, increments per append, returned to the caller). The log is queryable by since(session_id, after_seq) — returns all events with seq > after_seq (exclusive lower bound); snapshot_seq(session_id) returns the highest seq so far (the checkpoint for join-mid-flight).
Append, don't pushthe agent appends its output to the log; any connected client (on any server) sees it on its next poll — not just the operator who triggered the step.
Clients = snapshot + polled deltasa client renders the heavy state from the read-model (the graph/phases snapshot) and applies the deltas it polls from the log. The log carries the conversation + change events; the read-model carries the structured truth.
Join mid-flight / reconnectfetch the snapshot (read-model + recent chat) and poll from the last seq → no lost or duplicated messages.

Backend: an append-only event log in the read-model store (Mongo, or Postgres); clients poll since(seq) for correctness, and may upgrade to SSE (/stream, Last-Event-ID; B8.5) for live delivery over the same seq interface. No push bus — no Redis, no WebSocket fan-out: every server answers a poll/stream statelessly from the shared store, so there is no cross-server fan-out. (A WebSocket + a bus like Redis / Postgres LISTEN/NOTIFY remains a later option behind the same seq interface; correctness never depends on it.)

One ordered stream. Chat, graph deltas, and phase deltas all ride the same log under one sequence; the read-model snapshot is consistent to a seq, so a client renders the snapshot and applies only the deltas after it — no cross-path races between "the graph changed" and "the agent said…". Scaling: serialize within a session (one run-owner), run across sessions in parallel (different owners) — the system scales by incident count, not by serializing everything.

The whole session, in one line: a lock serializes the one run, a polled event log syncs the many clients, and the durable stores hold the truth — so any server serves any request, and the chat is shared and live without push infra, a fourth store, or sticky routing.

B8.5 · Roles, the pen, and the chat UI (MVP)

The pen. One member holds the pen (the single human writer) at a time — only the pen-holder may send a message or approve a gate; everyone else is a read-only viewer. (The pen is the human write-token — who may act; it sits above the run-owner lock of B8.2, which serialises the engine's one run — different mechanisms, same "one at a time" spirit.) The pen is handed over (take-pen / release-pen, one holder at a time; the creator starts with it). Pen liveness: a holder's activity (message / gate decision) refreshes a heartbeat; an idle holder past PEN_TTL (60s) can be force-taken by any other member via take-pen(), so a disconnected operator never permanently blocks writes and gate approvals. The agent runs one unit of work to completion, then takes the next message — strictly sequential. Approval is single — the pen-holder's OK is enough; no dual approval (a maker-checker rule is future). Roles are enforced in the UI and re-checked at the server on every write.

The chat UI. The chat is a heterogeneous stream of typed events ordered by seq; each renders via a widget registry (kind → component: text · agent step / tool-call · table · image · graph · sandboxed-iframe HTML for untrusted/agent-generated UI). A new widget type = one renderer, no core change; untrusted HTML is isolated in a sandboxed iframe (the safety seam — aligns with the open MCP-Apps pattern). Transport: SSE (GET /stream, resume via Last-Event-ID) for the live app, polling as the simple fallback — both over the one event log, so a widget is just a richer event payload, independent of the transport.

B9 The investigation graph — runtime, tools & library

The engine owns the graph; the LLM acts on it through tools. The graph is state — it lives as an in-memory working model (the engine's), persisted to the stores, never as text the LLM holds. The model reasons over a rendered slice and builds via governed tools. (Chosen over LLM-held text — fragile, no audit, can't be shared — and over a shared graph DB — heavy, concurrent-write contention; a per-incident graph, reused across incidents via the learning loop, gives determinism + audit + bounded context.)

B9.1 · Representation & library

A per-incident, in-process graph: nodes by id (facts embedded), typed edges (out + in adjacency), per Part D. It is small (the working slice, not the estate) — so an in-memory structure is enough; no graph DB.

ChoiceStance
Librarynetworkx (Python, in-process) — mature, cycle-safe traversal (descendants, shortest_path, subgraph). A custom typed adjacency structure is the fallback only if footprint/perf demands; per-incident the graph is small, so networkx is the KISS default.
Where it livesthe hot working copy on the run-owner (B8.2) — loaded from the durable graph when a step starts, mutated, checkpointed. Not a persistent in-memory server.
Durable formthe Mongo incident document (the graph as JSON, D6) + the LangGraph checkpoint. The in-memory copy is transient.

B9.2 · The tool surface — how the LLM acts on the graph

The LLM never receives the raw graph; it gets a small, governed tool surface (capabilities, gated like any other, Part C). Mostly read (traverse/expand) + one write (annotate a finding, with provenance). Bulk graph-building is automatic — the engine folds each tool result (B9.4); the LLM only queries + records explicit findings.

ToolEffect · what it does
graph.get(id)read · a node + its facts. Unknown id → {id, status: "unknown"}; never invents.
graph.neighbours(id, edge?, dir?)read · adjacent nodes — expand the frontier. Optional edge filter by edge type; dir = "in" | "out" (default). Caps at expand_cap=12 per node; returns {id, dir, neighbours:[stub…], more: N} where more is the overflow count (omitted if 0).
graph.walk(from, edges[], dir, until?)read · traverse a path (e.g. depends_on/hosted_on downward to the cause). With until (a predicate dict), BFS the edge-type-filtered subgraph for the NEAREST matching node, returning {from, path, nodes:[stub…], reached: true|false}. Without until, a greedy single-path frontier walk, returning reached: null. Max depth 12; cycle-safe.
graph.find(predicate)read · nodes matching a structured predicate. Recognized keys: label · layer · kind · type · id · name · unhealthy · impacted; an unrecognized key → match fails (never silent match-all). Empty predicate {} matches all. Returns {predicate, matches:[stub…], count}.
graph.blast_radius(id)read · the impacted set (walk affects/depends_on).
graph.path(from, to)read · the shortest causal path.
graph.annotate(target, key, value, evidence_ref)write · the agent records a finding (a fact or label like suspect) — validated + provenance-stamped (which step). Requires a non-empty evidence_ref — a finding without evidence is rejected with ValueError. Target must exist (fold/add it first). Returns {target, label|fact, evidence_ref, by_step}. This is "build as it goes," audited not gated.

Read tools resolve effect: read-only (allowed); annotate is a write but low-risk — it changes the agent's own working graph, not the world — so it is audited, not gated like a remediation.

B9.3 · Rendering — the slice the LLM sees

Each turn a deterministic render-slice projects the graph into the context — bounded regardless of size:

  • In full: the subject node, the current cause path, flagged nodes (suspect/impacted) + their facts, and the 1-hop frontier (adjacent un-expanded nodes, as stubs the agent can expand). Neighbours are capped per-node at expand_cap=12 and the frontier is globally capped at frontier_cap=20; overflow neighbours fold into the collapsed count, so the invariant full + frontier + collapsed = total always holds (the slice stays ~10–30 nodes regardless of graph size).
  • Collapsed: healthy / ruled-out / resolved → a count + one-line summary ("135 healthy · collapsed").

So even a 147-node incident renders ~10–30 nodes; the graph carries the memory, the slice shows only what's live. This is the compaction (B3) — the agent sees the distilled world + prior phase outputs, never the raw history.

B9.5 · Concurrency & reuse

Concurrency: single writer per session (the run-owner lock, B8.2) → no concurrent mutation → no merge conflict. Reads (the UI Graph pane) hit the Mongo read-model from any server. Reuse across incidents: the graph is per-incident; cross-incident knowledge comes via related/similar (the learning loop — the suggestions field), not a shared graph. A persistent estate graph, if ever added, is a read-mostly source the engine pulls topology from — never the mutable investigation surface.

B9.6 · Edge cases

CaseHandling
Unknown node ida tool references an id not in the graph → return {id, status: "unknown"}; the agent adds it (fold/annotate) or picks another path. Never silently invent.
Conflicting factstwo sources disagree on a node's health → keep both (source + confidence + observed_at); the disagreement stays visible; reconcile by confidence/recency. The graph never silently overwrites.
Graph grows largethe render-slice collapses aggressively (path + frontier only); the agent expands on demand; in-memory stays small (per-incident).
Expand too wideneighbours/walk cap depth + breadth → "N more, collapsed"; the agent narrows.
Cycles (depends_on loops)traversal is cycle-safe (visited set; networkx handles it).
Orphan node (an alert not yet linked)allowed; the agent links it (annotate) once it learns the relation.
Idempotent fold (crash re-run)folding is idempotent — keyed by (node id, fact key, source, observed_at) — so a replayed step never duplicates.
Annotate hallucinationannotate is validated vs the Part D type system and requires an evidence_ref — a finding with no evidence is rejected; bad annotations are auditable + reversible.

C3 govern() — resolved at call time, every time

Hard deny on missing capability: if the capability is not in the registry, govern() returns deny immediately and never invokes — a missing/unregistered capability is never allowed to run.

if cap not in registry: return Decision(cap_id, unknown, deny, "not in registry")
effect = policy(cap).effect ?? cap.effect_hint ?? "unknown"
access = policy(cap).access ?? (playbook.unknown_access if effect=="unknown"
                                else DEFAULT[trust(cap)][effect])

C4 Intent resolution

def _effect_of(cap, registry):                     # the AUTHORITATIVE effect govern() will use
    pol = registry.policy(cap.id)
    return (pol.effect if pol else None) or cap.effect_hint or Effect.unknown

def resolve_intent(need, phase_effect, registry):
    return rank([c for c in declared_capabilities
                 if need in c.intents                       # satisfies the intent
                 and compatible(_effect_of(c, registry), phase_effect)])  # POLICY-CORRECTED effect, not hint

The playbook names needs:[metrics, logs, …]; resolution turns each into the concrete, allowed capability. The effect boundary is enforced here at POLICY-CORRECTED effect — the resolver mirrors govern()'s effect calculation, so if a policy overrides a mistyped hint, the resolver uses the corrected effect. A read-only phase thus provably cannot select a write, even if a hint was wrong.

C5 Autonomy ladder & feedback

Operator feedback (D5), kept separate from the run, drives an autonomy ladder per phase × node-type × severity: a proven low-risk action can move ask → allow (earned autonomy); a declared failure/correction moves it back. Autonomy is narrow, evidence-based, and always reversible — the same CapabilityPolicy row is what changes. Implementation note: the autonomy unit is the whole capability row, so an earned allow currently applies engine-wide; finer per-phase × node-type × severity scoping (re-keying the policy + the govern/resolver lookups) is a tracked refinement, not yet built.

D · The data model
Two schemas. The Investigation Graph — the per-incident world (Node · Fact · Edge), shared by all phases. The Phase Record — the per-phase journey (PhaseRecord · Step) + a typed output. Every field is here; NEW marks a field the phase-output sufficiency check added.

D1 The investigation graph

The incident & cluster rule. A graph has exactly one incident node — the subject of this investigation. Other incidents are referenced and actively used (correlate · carry knowledge · link), never loaded as full nodes — they live in their own documents; related_to holds an id + kind. When incidents share a cause, the engine surfaces the match and the operator may link them into a cluster (member_of) — a human-confirmed grouping, not a separate run; each incident keeps its own session, phases, and closure. One playbook (incident); no "cluster playbook."

E4 Errors, retries, recovery

In the phase loop (B3): a capability invocation raises TransientError or PermanentError (B4 classify, not the capability — the layer wraps raw exceptions). Transient → retry per defaults.retry (max attempts; no backoff in-process); permanent → record the error Step (kind=tool_call, result={error: …}) → check on_failure:

on_failure modeBehaviour
run-remainingfinish any remaining independent steps in the phase; then set state=blocked (partial) and close. The run continues to the next phase; the escalation lives in the audit (the error Step is recorded; no separate error_handler fires).
default (halt)fire the playbook's error_handler immediatelyappend a reasoning Step (result={escalated: true, error: …}) with the handler's action/to/via, set state=failed, close the phase, halt the run (do not advance). The error-handler is an escalate Step — a persistent audit event, not config code that runs.

Exactly-once & checkpointing: every write carries an idempotency_key; on resume after a crash, the layer checks seen and returns the cached result instead of re-invoking. The run state (graph + messages + phase records) is checkpointed after every step (LangGraph), so a second server resumes from the last checkpoint, re-runs any in-flight step (reads idempotent, writes guarded by idempotency_key).

Backtrack: Verify's recovered field determines the edge (B6). If false, the run backtracks to Root cause as a new attempt (same session, new phase_records entry with attempt++); the graph + prior outputs stay (context for the retry). A halted or denied run ends (status=failed or denied); human closes the incident.

F · Operations

F1 The console

The only operator UI — a React single-page app, in the Claude-Desktop three-pane shape: Incidents (left) · Triage — the whole incident's chat with the approval gate inline (center) · Phases & Steps with a Graph tab (right). It evolves as autonomy is earned (the autonomy badge).

Live sync & message flow

The UI runs a polling loop (or SSE for liveness) to fetch /poll every ~2 seconds (with after_seq to resume from the last event). Each poll returns: the read-model snapshot (incident + graph + phases) + new events since the client's seq + the current status + the pen_holder. Messages are typed by the UI user (pen-holder only) and sent via POST /messages, enqueued in the session, then drained into the run on the next /advance. So a message does NOT block — the operator sends it, sees it appended to the event stream, then a separate /advance call drives the run forward. (In future, a single /message + auto-advance endpoint may conflate these.)

The gate, inline: when the run pauses (status=waiting_approval), the gate prompt appears in the chat with the proposed action + expected_effect + blast_radius + rollback. The pen-holder clicks approve | refine | deny; the UI calls POST /gate {gate_id, decision, actor}. On approve, the server resumes the run. On deny, the run halts with status=denied (terminal). On refine, the gate stays open — the operator can send a follow-up message (via /messages) asking for a different action; the agent loops and proposes again. The pen is held until explicitly released or the incident closes.

Roles: the session has a pen_holder (the only writer) and viewers. The UI renders /poll's role field (writer|viewer) to gate the send-message + approve buttons. A revoke (via require_member re-check at the server) drops the operator from the next event poll — they lose access to chat + incident doc immediately, without requiring a page refresh.
