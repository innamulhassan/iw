The Investigation Workbench — The Capability Layer

A visual architecture: where it sits, the path of a call, and why it is built this way.

Infrastructure Resilience & Operations · governed investigation automation · June 2026

Own the domain, rent the orchestrator — applied to tools. The agents never call Datadog, ServiceNow, or a browser directly. They call abstract capabilities (observe_service, restart_workload, verify_user_journey). One governed layer maps each to whatever concrete tool implements it today — an existing MCP server, an internal API, one of our own skills, or Playwright. Swap what is underneath; the engine never changes. Reuse before we build; govern every capability the same way.

1 Where the Capability Layer sits

An incident engine is only as good as what it can see and do — and it sees and does everything through tools. This is that layer: one place between the engine and everything it touches.

  • The engine calls verbs, not tools. It asks for restart_workload, never "POST to the k8s API."
  • One layer, four sources. Registry + uniform contract + governance, fed by existing MCP servers, internal APIs, our own skills, and Playwright.
  • Reuse three, build one. We connect to what already exists and build only our own domain skills.
  • Why: one catalogue and one way to act is what keeps a bank's blast radius governable — and lets us swap any source without touching the engine.

2 The path of a single call

Every action — a metric query, a restart, a browser click — takes the same path. That is where the control lives.

  • Reads flow straight through. Observe-tier calls (triage, root cause, verify) run freely.
  • Writes meet the dial, then the gate. Actuate-tier calls consult the per-phase / per-service / per-severity policy; if approval is required the engine pauses and resumes — checkpointed, so a pause survives a restart.
  • Everything is audited. Who, what, when, inputs, result, approver — immutable.
  • Why: a prompt can be talked around; a boundary check cannot. This is the control a regulator wants to see.

3 Why it is built this way

  • No vendor lock-in. Sources are swappable adapters; the engine is portable. Replacing Datadog with Splunk is a config change, not a rewrite.
  • Safe by construction. The read/write line is the gate line — autonomy is a dial you turn up per phase as trust is earned.
  • Playwright where it is uniquely right. User's-eye health checks, actions through consoles with no API, end-to-end verification, and screenshot evidence — run isolated and origin-allow-listed, credentials out-of-band.
What I recommend. Build the read-only path end-to-end first (Triage + Root cause, observe-only), prove the audit trail, then enable the first gated write. Reuse before you build; treat Playwright as the bridge-and-verifier, not the default actuator; and insist on one registry — if a capability is not in the catalogue, the engine cannot use it.
