Incident Triage — Low-Level Design (LLD)

The implementation: concrete schemas, the LangGraph run loop, govern(), intent resolution, the phase executor, the gate, and the engine API.

Shapes and fields are defined in the Data-Model doc; here we fix the physical design — collections, tables, the state machine, the resolution functions, and the contracts a developer implements against. 1 Stores — physical schema

PostgreSQL — config plane (normalized)

CREATE TABLE playbook (              -- the versioned process (body_md + machine front-matter)
  id            text, version text,    -- PK = (id, version): every edit is a new immutable row

CREATE TABLE provider (
  id text PRIMARY KEY, name text, kind text,            -- skill|mcp_local|mcp_remote|a2a_agent|api
  connection jsonb, trusted bool, status text, last_synced timestamptz );

CREATE TABLE declared_capability (
  id text PRIMARY KEY,                                  -- provider__action
  provider text REFERENCES provider(id),
  description text, input_schema jsonb, output_schema jsonb,
  effect_hint text, intents text[], last_synced timestamptz );  -- intents[] = which needs it satisfies

CREATE TABLE capability_policy (                    -- human-owned, sticky
  capability_id text PRIMARY KEY REFERENCES declared_capability(id),
  effect text, access text,                             -- allow | ask | deny
  status text, reviewed_by text, reviewed_at timestamptz );

The LangGraph checkpointer uses its own Postgres tables (PostgresSaver); we do not hand-model them. thread_id = the run id.

2 The run loop — LangGraph state machine

One graph; the run is a thread_id; state is checkpointed after every step. The durable incident document is written through to Mongo on phase completion + close.

# the run State (reducers omitted)
class RunState(TypedDict):
    subject:        SubjectRef            # {domain, id, kind} — NOT incident_id
    playbook_ref:   str                   # "incident-triage@1.0.0" (pinned)
    graph:          InvestigationGraph    # nodes / edges / facts — shared across phases
    phase_records:  list[PhaseRecord]
    messages:       list[Msg]             # the conversation
    current_phase:  str
    status:         str                   # running | waiting_approval | waiting_input | done | failed

# nodes + edges
graph = StateGraph(RunState)
graph.add_node("assess",      run_phase("assess"))
graph.add_node("root_cause",  run_phase("root-cause"))
graph.add_node("remediation", run_phase("remediation"))   # interrupts before each write (the gate)
graph.add_node("verify",      run_phase("verify-close"))
graph.add_edge("assess", "root_cause")
graph.add_conditional_edges("root_cause", lambda s: "root_cause"     # loop if < min_confidence
        if top_conf(s) < min_conf(s) else "remediation")
graph.add_conditional_edges("verify", lambda s: "root_cause"         # backtrack if recovery didn't hold
        if not recovered(s) else END)
app = graph.compile(checkpointer=PostgresSaver(...), interrupt_before=["__write__"])

3 The phase executor — the per-phase agent loop

def run_phase(phase_id):
  def _run(state):
    rec = PhaseRecord(id=f"{state.subject.id}:{phase_id}:{attempt}", phase=phase_id, plan=plan(state))
    while not output_sufficient(rec.output, schema_of(phase_id)):
        need   = model.next_need(state, rec)                 # intent the model wants next
        caps   = resolve_intent(need, effect_of(phase_id))   # §4 — registry lookup, policy-gated
        result = call(caps, args, state)                     # §5 — govern() runs inside; writes interrupt
        fold_into_graph(state.graph, result)                 # nodes / facts / edges + side-effects
        rec.steps.append(Step(kind="tool_call", capability=caps.id,
                              input=args, result=result,
                              touched=result.node_ids, evidence=result.evidence))
        rec.output = model.update_output(rec)                # build the typed output incrementally
    validate(rec.output, schema_of(phase_id))                # §6
    state.phase_records.append(rec); write_through_to_mongo(state)
    return state
  return _run

Read-only phases pass effect="read-only" to resolve_intent, so no write capability can be selected — the effect boundary is enforced at resolution, not just at the gate.

4 Intent resolution & govern()

def resolve_intent(need, phase_effect):
    caps = [c for c in declared_capabilities
            if need in c.intents                       # matches the intent
            and compatible(c.effect_hint, phase_effect)]  # read-only phase ⇒ no write caps
    return rank(caps)                                  # by trust, freshness, fit

def govern(cap, playbook):                             # resolved at call time, every time
    effect = policy(cap).effect or cap.effect_hint or "unknown"
    access = policy(cap).access or (playbook.unknown_access if effect=="unknown"
                                    else DEFAULT[trust(cap)][effect])
    return Decision(effect, access)                    # allow → invoke · ask → gate · deny → refuse
accessbehaviour
allowinvoke directly, then audit.
askpause (interrupt) → operator approve / refine / deny → on approve, invoke; record the decision.
denyrefuse; the model must find another path or escalate.

5 Writes — gate, idempotency, rollback

A write capability (effect=write) always resolves access=ask in remediation (gate_writes:true). The LangGraph interrupt_before pauses; the API surfaces the proposed action; the operator decides; the run resumes.

action = { action_id, kind, technique, target, expected_effect, blast_radius,
           rollback, temporary, revert_when, idempotency_key, gated:true }
# on resume(approve):
if seen(action.idempotency_key): return cached_result   # exactly-once
result = provider.invoke(action)                         # A2A / MCP / dynamic-MCP
record Step(kind="decision", decision="approve", actor=…); record action.result
on permanent failure → error_handler (escalate); transient → retry per defaults.retry

6 Output validation

Each phase output is validated against the playbook's output_schemas[phase.output] (JSON Schema, stored in playbook.schemas) on produce — required fields present, types correct. A failed validation keeps the phase open (the loop continues) rather than emitting a malformed contract.

7 Capability registry sync

  1. For each provider, pull its manifest → upsert declared_capability rows (last_synced stamped); tag each with the intents[] it satisfies.
  2. A newly-seen capability gets a capability_policy row at status=pending_review, access=deny — it cannot be used until a human reviews.
  3. A reviewer sets effect + access (allow/ask/deny) → status=active. Policy is sticky across syncs (declarations refresh; the grant does not).

9 Error handling, retries, backtrack

  • Transient capability failure → retry per defaults.retry (max + backoff); permanent / blocked → error_handler (escalate to on-call).
  • on_failure: run-remaining — independent steps in a phase finish before the phase reports.
  • Backtrack — Verify failing recovery routes back to Root cause (a new attempt); the graph + prior records are retained as context.
  • Crash recovery — the run resumes from the last checkpoint; in-flight writes are guarded by the idempotency key.
