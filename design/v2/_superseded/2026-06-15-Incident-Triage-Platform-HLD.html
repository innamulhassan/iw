Incident Triage — High-Level Design (HLD)

The architecture: the components, how they fit, and the cross-cutting design. A governed, in-house, domain-neutral investigation engine; incident triage is the first domain.

Infrastructure Resilience & Operations · v1.0 · companion to the Spec (scope), the LLD (implementation), the Data-Model doc (schemas), and the Runbook (operations)

This HLD sits between two companions: the Spec (what we're building + scope, one page) above it, and the LLD (exact schemas, internals, contracts) below it. Field-level schemas live in the Data-Model doc; this document is the architecture — components, interactions, and the cross-cutting concerns (governance, audit, hardening, autonomy).

1 Context & goals

Production incident support today leans on a vendor engine (Amelia) and manual triage. The goal is a governed, in-house engine that diagnoses and (under approval) remediates incidents, with a regulator-grade audit trail and reversible, earned autonomy. Design goals:

  • No vendor lock-in — the engine calls abstract capabilities, never tools directly; a source can change without touching the engine.
  • No domain lock-in — everything domain-specific lives in a versioned playbook; the engine, shapes, registry, governance, storage and console are universal.
  • Governed at the boundary — every capability call passes an allow / ask / deny gate; every write is approved by a human; closing is a human act.
  • Durable + reversible — checkpointed, exactly-once, resumable; every write carries a rollback.
  • Gradual — Amelia runs alongside and is dialled down only as in-house actions prove themselves.

2 Architecture overview

One engine drives the work and calls abstract capabilities; one governed capability layer sits over every source; the operator works in one console. Two data planes (Mongo / Postgres) plus the LangGraph run state.

4 The engine & run loop

Assess→Root cause→Remediation→Verify & close↺re-enterable

The engine loads the playbook (pinning playbook@version) and seeds the graph from the incident source. It then drives each phase:

  1. Plan — the model proposes how to satisfy the phase goal, bounded by the phase's needs + effect.
  2. Loop — resolve a need → a governed capability, call it, fold the result into the graph, log a Step; repeat until the typed output is sufficient. Steps are emergent, not scripted.
  3. Gate — a write pauses for human approve / refine / deny; read-only phases cannot write (enforced by effect).
  4. Checkpoint — state is persisted (LangGraph) after each step; the run is resumable across pauses and crashes; writes are exactly-once (idempotency key).
  5. Re-enter — Verify can backtrack to Root cause with new evidence; a phase re-runs as a new attempt.

A human closes; the engine never auto-closes. The Step journey is the audit trail; the typed output is the contract the next phase, the UI, the audit and the learning loop read.

5 The capability layer & governance

The playbook names intents; the layer turns an intent into a real, allowed call. Onboarding priority per source:

native MCPelse →register API as dynamic MCPelse →A2A specialist agent
Registry entityRole
Providera source (kind: skill · mcp_local · mcp_remote · a2a_agent · api); connection + trust + status; synced.
DeclaredCapabilityone action a provider offers (provider__action); input/output schema; effect_hint.
CapabilityPolicyhuman-owned: effect × access (allow / ask / deny); a new capability lands pending_review · deny.

govern() resolves at call time: effect = policy.effect ?? declared.effect_hint ?? unknown; access = policy.access ?? (effect==unknown ? playbook.unknown_access : DEFAULT[trust][effect]). A playbook need resolves to DeclaredCapabilities by intent + effect; policy gates each; the call is logged as a Step. The playbook never names a tool — only the registry does.

Autonomy ladder. Feedback (separate from the run) promotes/demotes a capability's access per phase × node-type × severity: proven low-risk reads/actions can move ask → allow; a declared failure moves it back. Autonomy is earned narrowly and is always reversible.

6 The data model (high level)

Each run builds one investigation graph — typed Nodes (systems, the incident, changes, alerts), Edges (depends_on, hosted_on, suspected_cause…) and Facts (health, latency, impact_state, each with an evidence_ref). One subject node; other incidents are referenced and actively used. Each phase records a PhaseRecord (plan · ordered Steps · a typed output) and writes its output into the graph.

PlaneHolds · shape
MongoDB (data)the incident document — graph + phase outputs + feedback; denormalized, one per subject; one read.
PostgreSQL (config)the playbook (md + version + fields) + the capability registry; normalized, versioned, governed.
LangGraph (run)the live checkpoint + conversation; the engine's mechanism, materialized into the incident document on close.

Every field — Node, Edge, Fact, PhaseRecord, Step, the four outputs, the registry, feedback — is defined in full in the Data-Model doc.

7 The operator console

One workbench, three panes: Incidents (left) · Triage — the whole incident's chat with the approval gate inline (center) · Phases & Steps with a Graph tab — the phases + typed outputs, or the investigation graph with the cause path highlighted (right). It evolves as autonomy is earned (an autonomy badge). Worked example: INC-4821 — a failed disk → RAID rebuild → DB I/O → checkout latency. See the Presentation for the rendered console.

8 Cross-cutting concerns

ConcernDesign
Auditthe Step journey is append-only and is the audit trail; every tool-call carries an evidence deep-link; approvals record actor + decision + time.
Security / hardeningsecrets / KMS, encryption at-rest + in-transit, least-privilege capability scopes, audit immutability — in place before any write ships (gates M2).
Exactly-onceevery write carries an idempotency key; a replay after a crash does not double-apply.
DR / resiliencecheckpointed run state (resumable); RTO/RPO targets; the engine survives source outages (a need with no healthy capability degrades gracefully + asks).
Feedback / learningoperator verdicts (outcome / failure / correction), separate from the run, feed the autonomy ladder and the similar-incident suggestions.
Observabilitythe engine emits its own metrics (phase latency, gate decisions, autonomy mix) for the autonomy dashboard.

9 Delivery (summary)

M0 Foundation→M1 Assisted→M2 Remediation→M3 Coverage→M4 Autonomy

Five milestones over ~14 two-week sprints; each gated, audited, reversible; a parallel hardening track gates production writes (M2); Amelia runs alongside and ramps down per proven slice. The by-sprint plan + timeline is the Delivery Plan.
