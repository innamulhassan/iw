The Investigation Workbench — Runbook

The first use case — incident triage — end to end: what each phase asks, the capability intents it needs and what they gather, the steps, and how the investigation graph + record build up — phase by phase.

Infrastructure Resilience & Operations · the operational detail behind the Design doc + the Data-Model doc · worked on INC-4821 / payments-api (a storage root-cause incident)

How to read this. Each phase has four parts: the questions it must answer · the capability intents (needs) it calls and what each gathers (with the access it needs) · the steps · and the data it adds — the typed output and the graph it touches. Phases name intents, never tools; the engine resolves each to a concrete capability from the live registry, bounded by effect. Reads flow through; the one write (remediation) stops at the approval gate.
Two things grow as the run proceeds. The graph — one per incident, shared by all phases (nodes · facts · edges). The record — AssessResult → + RootCauseResult → + RemediationResult → + VerifyResult. Each phase reads what's there and adds its typed output; a backtrack re-enters a phase with more context. The subject is {domain, id} = {app-incident, INC-4821} — domain-neutral.

1 Assess — what broke, how bad, the context

Questions

  • Which node(s) are affected — one app, or several across layers (app · DB · network · storage)?
  • How bad — impact × urgency → priority, and the business impact?
  • What changed recently (deploys / config / flags)? (changed:[] = checked & clean.)
  • What's the blast radius, and which neighbours show an impact_state?
  • Are there related or similar priors — and what fixed them?

Needs (intents) — and what each gathers

Need (intent)What it gathersEffect
incident-sourcethe incident record — symptom, reporter, the firing alert(s)read
topologythe affected node + its neighbourhood (depends_on / hosted_on / runs_on), owner / on-callread
telemetrylight health on the symptom path — golden signals + impact_state per noderead
change-historyrecent deploys / changes on the node + its dependenciesread
similar-incidentsrelated / similar priors — and their resolution (becomes a suggestion)read

Steps

  1. Seed the graph from incident-source — the incident node (the subject) + the firing alert.
  2. Load the topology (the affected app + neighbourhood) and overlay light health.
  3. Compute the blast radius in business terms; classify incident_type.
  4. Pull recent changes — record changed:[] as a positive "checked & clean" if none implicate.
  5. Link related / similar priors and use them — a prior's resolution → a suggestion.
  6. Set an initial suspected_locus to give Root cause a head start.
Data after Assess — AssessResult (+ the seeded graph)
{ incident_type:"performance",

  changed:["chg:deploy-rev47"],                       // present, but will be ruled out in root-cause
  time_factor:null,

2 Root cause — the probable cause, with evidence and a basis

Questions

  • What is the probable cause — and the ranked alternatives?
  • How confident, on what basis, and with what evidence?
  • Does it sit below the unhealthy node (a deeper layer), or in a change / a zone / a time factor?
  • What can we rule out, with the evidence that kills it?

Needs (intents) — and what each gathers

Need (intent)What it gathersEffect
traceswhere the latency actually originates across the path (here: 75% is the DB call)read
metricsgolden signals for the suspect nodes around onset — incl. the layer below (DB I/O, storage latency)read
logserror lines / signatures for the windowread
change-historythe diff of any suspect change (rev47), to confirm or rule outread
layer-deep-divethe per-layer specialist (DB / storage / network agent) for the deep signalread

Steps

  1. Walk the graph from the symptom toward suspected_locus — and below it if that layer reads slow.
  2. Correlate traces / metrics / logs; follow the path app → DB → storage.
  3. Form ranked candidates, each with a path and a confidence{value, basis} — the basis states why.
  4. Rule out alternatives and record the killing evidence; pivot if the health-walk dead-ends.
  5. If top confidence < min_confidence → gather more, or ask the operator.

3 Remediation — the safest fix, gated

Questions

  • What's the safest fix — reverse, mitigate, or escalate?
  • What will it do (expected_effect), who does it touch (blast_radius), how do we undo it (rollback)?
  • Is it temporary — and when must it be reverted (revert_when)?
  • Is it approved — and did it actually run?

Needs (intents) — and what each does

Need (intent)What it doesEffect · access
remediation-actionwrites: the chosen fix (here: failover db:payments-ora → standby); returns the resultwrite · ask (gated)
escalationwrites: hand a durable fix to the owning team (replace the failed disk) — a ticket / pagewrite · ask

Steps

  1. Propose the fix from the selected cause — here a mitigate / failover; state expected_effect · blast_radius · rollback, and temporary + revert_when.
  2. Approval gate — the operator approves / refines / denies inline (recorded as a decision step).
  3. Execute with an idempotency key (exactly-once); capture the result + evidence.
  4. On failure — retry transient errors per the playbook retry; permanent / blocked → error_handler (escalate).
  5. Record followups for the durable fix owned elsewhere.
Data added — RemediationResult
{ actions:[
    { action_id:"a1", kind:"mitigate", technique:"failover", target:"db:payments-ora",
      expected_effect:"DB served by standby; I/O normal", blast_radius:"payments read path",
      rollback:"fail back once disk replaced", temporary:true, revert_when:"incident_close",
      idempotency_key:"INC-4821-a1", gated:true,
      approval:{ decision:"approve", actor:"j.rivera", at:"14:21" },
      result:"applied; I/O 28ms→4ms", status:"done" } ],
  followups:[{ detail:"replace disk 1.4.7", basis:"escalate", owner:"storage" }],
  status:"applied" }

4 Verify & close — confirm recovery, then a human closes

Questions

  • Did it recover — from the user's side (before / after)?
  • Does the user journey work (synthetic replay)?
  • Is every temporary action reverted-or-scheduled?
  • What actually fixed it (the resolution)? Safe to close? (a human decision)

Needs (intents) — and what each gathers

Need (intent)What it gathersEffect
telemetrybefore / after on the symptom — p99, error rateread
synthetic-replayruns the key user journey and returns pass / fail + evidenceread

Steps

  1. Re-check the symptom + replay the journey; capture before / after.
  2. Confirm recovery from the user's side (not just "the metric came back").
  3. Confirm every temporary action is reverted-or-scheduled — none may dangle.
  4. Record the resolution (what the next incident will learn from). If it didn't hold → backtrack to Root cause.
  5. The operator confirms and closes — closing is a human act, never automatic.
Data added — VerifyResult
{ recovered:true,
  before_after:"p99 4.2s→260ms, errors 18%→0.2%",
  watch_window:"15m", recovery_confidence:0.95,
  resolution:"failover to standby; durable fix = disk replacement",
  temporary_actions_status:[{ action_id:"a1", status:"scheduled" }],
  residual:[{ item:"replace disk", owner:"storage" }],
  closed_by:"j.rivera", status:"closed" }

5 What the triage needs — intents, resolved by the registry

Phases name intents; the registry resolves each to a concrete capability and the human-owned policy gates it (allow / ask / deny). Onboarding priority per source: native MCP → else register the API as a dynamic MCP → else A2A for the per-layer specialists.

6 The playbook that drives this

The process above is a versioned playbook (Markdown + YAML front-matter) — tool-agnostic, so it never names a tool; change it without changing the engine, and a different playbook runs a different domain. The full file + the graph_schema are in the Data-Model doc (§1); the essentials:

---
id: incident-triage
version: 1.0.0
domain: app-incident                         # one engine, many playbooks
defaults: { on_failure: run-remaining, retry: { max: 3, backoff: exponential } }
unknown_access: ask
error_handler: { action: escalate, to: on-call, via: pagerduty }
phases:                                        # TOOL-AGNOSTIC — intent (needs) + effect, never a tool
  - { id: assess,       effect: read-only, output: AssessResult,
      needs: [incident-source, topology, change-history, telemetry, similar-incidents] }
  - { id: root-cause,   effect: read-only, output: RootCauseResult, min_confidence: 0.7,
      needs: [metrics, logs, traces, topology, change-history, layer-deep-dive] }
  - { id: remediation,  effect: write, gate_writes: true, output: RemediationResult,
      needs: [remediation-action, escalation] }
  - { id: verify-close, effect: read-only, output: VerifyResult,
      needs: [telemetry, synthetic-replay] }
output_schemas:
  AssessResult:      { required: [incident_type, symptom, affected, impact_assessment, changed, suggestions] }
  RootCauseResult:   { required: [candidates, selected, status] }
  RemediationResult: { required: [actions] }
  VerifyResult:      { required: [recovered, resolution] }
---
## assess        Build the picture (read-only); use similar priors; changed:[] is "checked & clean".
## root-cause    Ranked candidates with a path + a confidence basis; rule out; pivot if health reads green.
## remediation   Safest fix — reverse | mitigate | escalate; every write gated; expected_effect · blast · rollback.
## verify-close  Confirm recovery from the user's side; revert temporaries; record resolution; a human closes.
