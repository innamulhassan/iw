The Investigation Workbench — Data Model

Domain-neutral by design — one engine, many playbooks. The shapes, the run loop, the registry, feedback and storage are universal; each playbook supplies its own domain (types · phases · outputs). Shown end-to-end with the Incident-Triage playbook. Every schema is defined → explained → shown as JSON → tabled.

v1.0 · consolidated from the HLD + LLD + the 14-incident validation (G1–G25) · coverage-checked (Appendix) · written so a human can approve each schema and understand every field — no ambiguity.

1 · The Playbook

The self-sufficient domain reference — process + types, tool-agnostic

We start here because this is the part we agree on — and because this file is the only place anything domain-specific lives. Front-matter the engine executes — phases (intent + effect, no tools), graph_schema (the domain type system), output_schemas — and a body it follows. Swap this file → new domain, same engine (§2 shows exactly how). Stored as a config row in §6.1; every type and field defined in §3–§4.

---
id: incident-triage
version: 1.0.0
domain: app-incident

defaults:
  on_failure: run-remaining                  # finish independent steps, then report
  retry: { max: 3, backoff: exponential }    # TRANSIENT errors only; permanent → error_handler
unknown_access: ask
error_handler: { action: escalate, to: on-call, via: pagerduty }

phases:                                        # tool-agnostic: intent (needs) + effect, never a tool name
  - { id: assess,       effect: read-only, output: AssessResult,
      goal: "Know what's affected, how bad, what changed — surface impact + early suggestions.",
      needs: [incident-source, topology, change-history, telemetry, similar-incidents] }
  - { id: root-cause,   effect: read-only, output: RootCauseResult, min_confidence: 0.7,
      goal: "Find the most probable cause — ranked, with evidence; pivot if the health-walk dead-ends.",
      needs: [metrics, logs, traces, topology, change-history, layer-deep-dive] }
  - { id: remediation,  effect: write, gate_writes: true, output: RemediationResult,
      goal: "Apply the safest fix — reverse, mitigate, or escalate; stop the bleeding.",
      needs: [remediation-action, escalation] }
  - { id: verify-close, effect: read-only, output: VerifyResult,
      goal: "Confirm recovery from the user's side; revert temporaries; a human closes.",
      needs: [telemetry, synthetic-replay] }
# the engine maps `needs` (intents) → concrete capabilities from the LIVE registry (§6.2), bounded by `effect`.

graph_schema:                                  # THE DOMAIN TYPE SYSTEM — incident-triage's vocabulary (defined in §3)
  node_types:
    system: { business: [business_service], app: [app, service, component], database: [database],
              network: [network_device], storage: [storage], compute: [compute],
              location: [zone, region], external: [external_service], any: [generic] }
    incident: [incident, cluster]              # exactly ONE `incident` node per graph (the subject); see §3 intro
    change:   [change]
    alert:    [alert]
  edges: [part_of, depends_on, runs_on, hosted_on, routes_to, located_in, changed, affects,
          suspected_cause, related_to, member_of, observed_on, correlated_into]
  facts:  [health, error_rate, latency, sessions, deadline, impact_state]
  labels: [suspect, policy_block, central, pci, tier-0]
  conventions: [policy_block, realized_by, alert_noise]

output_schemas:                                # this playbook's phase output contracts (every field in §4.4)
  AssessResult:      { required: [incident_type, symptom, affected, impact_assessment, changed, suggestions] }
  RootCauseResult:   { required: [candidates, selected, status] }
  RemediationResult: { required: [actions] }
  VerifyResult:      { required: [recovered, resolution] }
---

## assess
Build the picture — read-only. From the incident-source seed the incident + alert nodes; from topology
load the affected service(s) + neighbourhood; overlay light health on the symptom; pull recent changes;
link related / similar priors and USE them. Cluster if incidents converge on one upstream node or share a
fingerprint in a window. Emit the impact_assessment + early suggestions (from similar past incidents).
`changed:[]` is a first-class "checked-and-clean" assertion; record a `time_factor` for time-triggered contributors.

## root-cause
Ranked candidates — never one asserted — each with a `suspected_cause.path` (victim → … → cause) and a
confidence whose **basis** is stated. Walk the graph; rule out alternatives with evidence. **Fallback:**
if no node is unhealthy (data / async-lag / expiry read green on health), pivot — data-quality alerts,
`changed:[]` + `time_factor`, write-edges, and `impact_state`. Cause may be a node, change, zone, or time_factor.

## remediation
The safest fix — every write gated. `kind`: reverse · mitigate (technique; temporary ones carry
`revert_when`) · escalate (names the team). Each action carries `expected_effect`, `blast_radius`, and
`rollback`. Idempotency key (exactly-once); permanent failure → error_handler. `followups[]` for durable fixes.

## verify-close
Confirm recovery from the user's side (before/after + synthetic replay). Confirm every `temporary` action
is reverted-or-scheduled before close. Record the **resolution**. If it didn't hold, backtrack to root cause.
Human closes.
2 · Domain-neutral by design universal engine

One engine, many playbooks — no domain lock-in

The model has three levels. Only the middle one is domain-specific — so the same engine runs incident-triage today and provisioning or capacity tomorrow, by loading a different playbook.

LevelWhat it isDomain-specific?
Enginethe shapes (Node·Edge·Fact·PhaseRecord·Step), govern(), the run loop, gates, retries, clustering, the registry, feedback, the two storage planesNo — universal. Never changes between domains.
Playbookthe type system (graph_schema), the phases + their needs/effect, and the output contracts (output_schemas)Yes — this is the only domain layer. One file per domain.
Runtimethe concrete nodes/edges/records of one investigationInstances — conform to whatever playbook is loaded.

How the subject is identified — domain-neutral (never a bare incident_id)

Why a subject reference, not incident_id: the PhaseRecord (§4), Feedback (§5), the run, and the read-model must point at the thing under investigation without assuming it is an incident. So they carry a SubjectRef — the pair {domain, id} (plus the subject's kind). The unique key is (domain, id): incident-triage fills it {app-incident, INC-4821}, provisioning {provisioning, REQ-123}. This is what makes the neutrality real, not just claimed.

{                                  // SubjectRef — used by PhaseRecord (§4.1), Feedback (§5), the run, and the read-model (§8)
  "domain": "app-incident",        // = Playbook.domain — which world this is
  "id":     "INC-4821",            // the subject's identity IN that domain (provisioning: "REQ-123")
  "kind":   "incident"             // the subject node's kind (provisioning: "provision_request")
}                                  // unique key = (domain, id) — the engine never assumes the word "incident"
Proof — the same shapes carry a different domain. A second playbook changes only the green/right column above. Everything else is identical. 
id: provisioning-request          # a DIFFERENT investigation playbook — SAME engine, SAME shapes
domain: provisioning
phases:
  - { id: validate,     effect: read-only, output: ValidationResult, needs: [request-source, quota, policy] }
  - { id: allocate,     effect: write, gate_writes: true, output: AllocationResult, needs: [provision-action] }
  - { id: verify-close, effect: read-only, output: VerifyResult,     needs: [resource-health] }
graph_schema:
  node_types:                       # totally different vocabulary — but still Nodes
    request:  [provision_request]
    resource: [vm, database, bucket, network]
    actor:    [requester, approver]
    quota:    [quota]
  edges: [requires, consumes, approved_by, blocked_by, allocated_to]   # still Edges
  facts: [quota_available, cost_estimate, policy_check]                # still Facts
output_schemas: { ValidationResult: {...}, AllocationResult: {...}, VerifyResult: {...} }
The engine never learns the word "incident" or "vm" — it only sees nodes with a kind from the loaded playbook's vocabulary. That is what "no domain lock-in" means: a library of playbooks, one engine.
3 · The Investigation Graph Mongo · data plane

The per-investigation world: Node, Fact, Edge

One graph per investigation — the shared world all phases read and write. The shapes here are universal (§2); the type/edge/fact values shown are incident-triage's, from its graph_schema. NEW = a field the phase-output sufficiency check added.

The incident & related-incidents rule — one place, stated once. A graph has exactly one subject node — in incident-triage, the incident node (provisioning's would be a provision_request): the subject of this investigation. But other incidents are an active part of the work — during Assess we pull in related and similar priors and use them: to correlate (do they share a fingerprint or converge on one node?), to carry knowledge (a similar incident's resolution becomes a candidate suggestion), and to link them (related_to, kind = child \| related \| caused_by \| chained \| similar). They are referenced, not duplicated — we hold an id + kind + the few bits we actually use (its resolution, fingerprint, cause); the full incident stays in its own document. When the engine spots incidents that share a cause, it surfaces them to the operator, who may link them into a cluster (each via member_of) — a human-confirmed grouping, not a separate run. Each incident keeps its own session, phases, and closure; the cluster just lets the shared cause/fix propagate. So incident: [incident, cluster] = the one subject, plus the optional cluster it may be linked into. One playbook (incident); no "cluster playbook."

3.1 · Node — a thing in the investigation

Definition. A typed vertex in the graph. The shape is universal; kind/type/layer draw their values from the loaded playbook's node_types (here, incident-triage's).

Why this shape: one node shape for every kind so the engine walks a single uniform graph in any domain. kind → type → layer is the only branching the engine needs (it tells the deep-dive which tools apply). labels and props stay open so a new source or attribute needs no schema change.

{
  "id":     "app:payments-api",        // kind:name — stable; everything refers to a node by id
  "kind":   "system",                  // value from the playbook's node_types (incident-triage: system|incident|change|alert)
  "type":   "app",                     // a type within the kind (or "generic")
  "layer":  "app",                     // system only: business|app|database|network|storage|compute|location|external
  "name":   "Payments API",
  "labels": ["pci","tier-0","suspect"],  // open tags; some (suspect/policy_block/central) drive triage
  "props":  { "owner_team":"payments-sre", "on_call":"j.rivera" },
  "facts":  [ /* Fact[] — observations made during the run; see 3.3 */ ],
  "summary": "degraded since 14:02; DB-backed latency",
  "sources": ["servicenow-cmdb","appd"]   // which tools contributed this node
}

3.3 · Fact — an observation on a node

Definition. A single timestamped, sourced observation embedded on a node (its health, a metric, an alert reading).

Why a property on the node, not its own node: an observation has no identity worth a node — embedding it keeps the whole picture of a node in one place to read. evidence_ref makes it auditable (a deep link, not hearsay). There is no ttl — freshness is judged at read time. impact_state is separate from health so a healthy-but-impacted node still counts toward blast radius.

{
  "key":          "health",                  // fact key from the playbook (incident: health|error_rate|latency|sessions|deadline|...)
  "value":        "degraded",
  "source":       "appd",                    // the tool that observed it
  "evidence_ref": "appd://payments-api/bt?t=14:02",  // deep link to backing data — auditable
  "observed_at":  "2026-06-18T14:02Z",       // NO ttl — freshness judged at read time vs what the reader needs
  "confidence":   0.9,                       // 0–1; discounted for noisy/suppressed sources
  "impact_state": "degraded"                 // erroring|degraded|stalled|lagging|wrong-data|ok — SEPARATE from health
}
FieldWhat it means
key · valueWhat's observed + the value — the keys are the playbook's (health, error_rate, latency…).
sourceThe tool that observed it.
evidence_refA deep link to the backing data (log query · trace id · dashboard) — auditable, not hearsay. (This is why no separate Evidence entity is needed — see Appendix.)
observed_atNo ttl. Freshness is judged here, at read time, against what the reader needs.
confidence0–1; discounted for noisy / suppressed sources.
impact_state NEWDistinct from health. A health:ok node can still be impacted — lagging, wrong-data — and counts toward blast radius.

3.4 · Edge — a typed relationship

Definition. A directed, typed link between two nodes — dependency, containment, causation, correlation.

Why typed edges + open props: the type carries the meaning the engine reasons over (walk depends_on for blast radius, suspected_cause for the cause path); props carry the per-type extras without exploding into dozens of subtypes. Direction is meaningful — the two walks go opposite ways.

{
  "type":    "suspected_cause",         // edge type from the playbook (see 3.5)
  "from":    "INC-4821",
  "to":      "stor:pay-vol",            // direction matters: blast-radius and cause-finding walk opposite ways
  "props": {
    "confidence": { "value":0.9, "basis":"evidence-complete + matches INC-4820" },
    "rank":       1,
    "path":       ["app:payments-api","db:payments-ora","stor:pay-vol"]   // the causal chain victim→…→cause
  },
  "sources": ["otel","netapp"]
}
FieldWhat it means
typeThe relationship (full list + meanings in 3.5).
from · toThe two node ids. Direction matters — blast-radius walks edges one way, cause-finding the other.
props{}confidence · last_seen · via (cmdb \| trace) · kind (related_to) · rank · realized_by · path (causal chain on suspected_cause).
sources[]Tools that asserted the edge.

3.6 · Conventions & clustering — reasoning rules on top of the shapes

Why conventions, not new fields: these are reasoning rules the engine applies to the shapes above — they need only a label or a prop, so the model stays small.

RuleWhat it means
policy_blockA node healthy on its own signals but the cause (a firewall enforcing a bad rule, an expired cert, a node serving bad data). Rule: a changed-edge / fact / time_factor on a healthy node on the victim's path → treat it as suspect despite health.
realized_bydepends_on.props.realized_by:[net…] names the physical path that implements a logical dependency — so blast-radius (logical) and fault-localization (physical) reconcile.
alert_noiseNoisy / suppressed alerts count toward clustering (volume + simultaneity) but are discounted in root-cause confidence — a storm can't outvote a high-confidence fact.
cluster (operator-linked)When incidents share an alert fingerprint AND their walks converge on one node in a window, the engine surfaces the match to the operator → who links them into a cluster (member_of). Never auto-merged; nothing runs "at cluster scope" — each incident keeps its own session.
4 · The Phase Record Mongo · data plane

The per-phase journey: PhaseRecord, Step, and the four outputs

One record per phase entered — the durable account of what the agent did, written into the subject document on close (the live run state itself stays in LangGraph, §7). The PhaseRecord and Step shapes are universal; the four outputs are incident-triage's (another playbook defines its own). Output contracts are enforced: each phase output is validated as a Pydantic model (AssessResult, RootCauseResult, RemediationResult, VerifyResult) — the output_schemas in the playbook are descriptive; these models are the enforced contract.

4.1 · PhaseRecord — the per-phase journey

Definition. The plan, the ordered steps, and the typed output for one phase of one investigation.

Why separate from the graph: the graph is the shared world; the PhaseRecord is the journey (what the agent did this phase). Same world, many journeys. steps[] is the audit trail; output is the typed contract the next phase reads. They connect through each step's touched[] (→ graph) and evidence[] (→ output).

{
  "id":          "INC-4821:root-cause:1",   // subject.id:phase:attempt
  "subject":     { "domain":"app-incident", "id":"INC-4821", "kind":"incident" },  // SubjectRef (§2) — the one incident this record belongs to
  "phase":       "root-cause",
  "goal":        "Find the most probable cause — ranked, with evidence.",  // from the playbook
  "state":       "done",                    // active|waiting_input|waiting_approval|blocked|done|failed
  "plan":        "Walk below the app; correlate vs rev47; rank with a path.",
  "steps":       [ /* Step[] — the ordered, timestamped journey; see 4.2 */ ],
  "output":      { /* the typed phase output — points back into the graph; see 4.4 */ },
  "summary":     "Cause = pay-vol disk failure (storage), conf 0.9.",
  "opened_at":   "14:04",  "closed_at": "14:07"   // closed_at = null while open
}
FieldWhat it means
idsubject.id:phase:attempt. The attempt number lets a phase be re-run after a backtrack.
subjectthe SubjectRef {domain, id, kind} (§2) — the one incident this record belongs to; never a bare incident_id. (Each incident has its own records — no "cluster scope".)
phase · goalWhich phase · its goal (carried from the playbook).
stateactive · waiting_input · waiting_approval · blocked · done · failed.
plan · steps[] · outputWhat it set out to do · the ordered journey (4.2) · the typed result (4.4).
opened_at · closed_atISO8601 timestamps. opened_at set when the record is created; closed_at set on every terminal state (done, failed, blocked) — null only while active.
summaryRolling summary of the phase's work.

4.2 · Step — one entry in the journey

Definition. One ordered event in a phase — a tool call, a reasoning note, a suggestion, the gate decision, or operator input.

Why one shape with a kind discriminator: the journey is a single ordered log — so the audit reads top-to-bottom with no joins. This deliberately keeps as one record what the HLD split into three tables (Action + HumanInteraction + RunEvent); the gate decision is just a step of kind:"decision". touched[] and evidence[] tie a step to the graph and to the output.

{
  "seq":        2,  "at": "14:04:30",
  "kind":       "tool_call",               // tool_call | reasoning | suggestion | decision | user_input
  "capability": "traces",                  // resolved from a `need` via the registry (§6.2), NEVER a hardcoded tool
  "input":      { "trace":"checkout" },
  "result":     { "db_span_ms":3140, "pct_of_total":"75%" },
  "touched":    ["svc:checkout","db:payments-ora"],  // → graph node/fact ids read or written
  "evidence":   ["otel://trace/9af3"],     // → deep links that flow up into the output
  "note":       "75% of latency is the DB call — not app code"
}
// other kinds: suggestion → { proposal, rationale }   decision → { decision, actor }   user_input → { message }
FieldWhat it means (by kind)
seq · at · kind(all) order · ISO8601 timestamp (every step) · the kind ∈ {tool_call, reasoning, suggestion, decision, user_input}.
capability · input · resulttool_call — the capability invoked (resolved from a need, never a hardcoded tool) + args + what came back. tool_call steps MUST name the capability id.
touched[]tool_call — graph node/fact ids read or written → links the step to the graph.
evidence[]tool_call — deep links to backing data → flows into the output.
proposal · rationalesuggestion — what's proposed + why.
decision · actordecision (the gate) — approve \| refine \| deny + who.
message · noteuser_input — what the operator said · (any kind) a short reasoning note.

5 · Feedback Mongo · data plane (separate from the run)

The operator's verdict — kept apart from the run record

Definition. A human's judgement about an investigation or run — an outcome, a declared failure, or a correction. Universal across domains.

Why separate from the run: the run record says what the engine did; feedback says what the human thought of it — mixing them would taint the audit. Its own small document, added after close, queried on its own. It feeds two loops: outcome confirms the resolution the next incident reads; failure/correction lower autonomy on the capability involved (the autonomy ladder).

  "actor":       "j.rivera",
  "kind":        "outcome",          // outcome (did it work) | failure (engine wrong) | correction (right answer)

  "note":        "disk replaced next morning, no recurrence",
  "at":          "2026-06-19T09:00Z"

6 · Config Postgres · config plane

Shared, versioned, governed: the Playbook and the Capability registry

Config is not per-incident — it is shared across every investigation, versioned, and changed under governance. So it lives normalized in Postgres, edited on its own lifecycle. An incident only references it (by playbook@version and capability id). All of this is universal — the registry serves any playbook.

6.1 · Playbook — the stored, versioned process

Definition. The row form of §1 — the process as markdown plus the machine-readable front-matter, immutably versioned. One row per playbook version; a library of them serves many domains.

Why md + version + fields, normalized (not denormalized): the playbook is the human contract — so its body stays markdown (body_md, exactly the §1 text), readable and diffable. A few structured columns (version, domain, status, owner) make it queryable; the machine parts (phases, schemas, defaults) ride along as JSONB. PK is (id, version) — every edit is a new immutable row, so a run pins playbook@version and old incidents stay reproducible. It is not copied into the incident doc — the incident stores only the pin.

{
  "id":       "incident-triage",
  "version":  "1.0.0",                  // PK = (id, version) — every change is a new immutable row
  "domain":   "app-incident",           // lets ONE engine carry a library of playbooks (incident, provisioning, capacity…)
  "status":   "active",                 // draft | active | deprecated
  "owner":    "sre-platform",
  "body_md":  "## assess …  ## root-cause …  ## remediation …  ## verify-close …",  // the §1 markdown
  "phases":   [ /* front-matter phases: id, effect, needs, output, … */ ],   // jsonb
  "schemas":  { /* output_schemas: AssessResult, RootCauseResult, … */ },          // jsonb
  "defaults": { "on_failure":"run-remaining", "retry":{"max":3,"backoff":"exponential"} },  // jsonb
  "unknown_access": "ask",  "changelog": "1.0.0 — initial graph-based playbook"
}
FieldWhat it means
id · versionThe composite key. version is semver; a run pins id@version so it is reproducible.
domainWhich problem domain it serves — lets one engine carry many playbooks (the heart of §2).
status · ownerdraft · active · deprecated (only one active version dispatched) · the owning team.
body_mdThe markdown body (the §1 prose) — the human-readable process the agent follows.
phases · schemas · defaultsJSONB — the front-matter the engine executes (phases + needs/effect, the output contracts, the defaults).
unknown_access · changelogWhat to do for an un-policied capability (ask) · why this version exists.

6.2 · Capability registry — how intents become governed calls

Definition. Three small tables that turn a playbook need (an intent) into a real, allowed tool call: Provider (where capabilities come from), DeclaredCapability (what each can do), CapabilityPolicy (whether it's allowed).

Why this is config, not code: the playbook names intents (needs:[metrics, logs, …]), never tools. The registry resolves an intent → matching DeclaredCapability (by intent + effect), and CapabilityPolicy decides allow / ask / deny. A human owns the policy — so what the agent may do is governed data a reviewer edits, not a code change. Shared by every playbook.

Provider — a source of capabilities

{
  "id":        "appd",
  "name":      "AppDynamics",
  "kind":      "mcp_remote",   // skill|mcp_local|mcp_remote|a2a_agent|api
  "connection":{ "url":"…" },  // jsonb — varies by kind
  "trusted":   true,
  "status":    "connected",    // registered|connected|disabled|error
  "last_synced":"2026-06-18T09:00Z"
}

DeclaredCapability — one action a provider offers

{
  "id":          "appd__get_health",  // provider__action
  "provider":    "appd",
  "description": "Health of an app / tier / node",
  "input_schema": { /* JSON Schema */ },
  "output_schema":{ /* JSON Schema */ },
  "effect_hint": "read",         // read | write | unknown
  "intents":     ["telemetry","metrics"]   // which playbook `needs` it satisfies — the resolver matches on this (03-design C2/C4)
}

CapabilityPolicy — the human-owned gate (the heart of governance)

{
  "capability_id": "bladelogic__restart_service",
  "effect":        "write",        // read | write | unknown
  "access":        "ask",          // allow | ask | deny  ← what the agent may do

TableFields
Providerid · name · kind (skill \| mcp_local \| mcp_remote \| a2a_agent \| api) · connection (jsonb) · trusted · status · last_synced. Where capabilities are discovered + synced from.
DeclaredCapabilityid (provider__action) · provider · description · input_schema · output_schema · effect_hint (read \| write \| unknown) · intents[] (which playbook needs it satisfies). The synced contract of one action.
CapabilityPolicycapability_id · effect · access (allow \| ask \| deny) · status (active \| pending_review) · reviewed_by · reviewed_at. The sticky, human-owned decision; a new capability starts pending_review + deny until a human rules.

The link back to §1: phase needs:[metrics] → registry finds DeclaredCapabilities tagged metrics with a matching effect → CapabilityPolicy gates each → the chosen call is logged as a Step{kind:tool_call, capability} (§4.2). The playbook never names appd or bladelogic — only the registry does.

7 · Storage — the two planes

Where everything lives — data in Mongo, config in Postgres, run state in LangGraph

Now that the entities are defined, here is where each lives. Three concerns, three homes — kept apart so each is shaped for how it's read.

PlaneStoreHoldsShape
dataMongoDB (primary)the subject document — the graph (§3), the phase records + outputs (§4), and feedback (§5)Denormalized, one doc per subject — the whole subject reads back in a single query (§8). One collection per domain (here incidents).
configPostgreSQLthe playbook (§6.1) and the capability registry (§6.2)Normalized, versioned, orderly — config is shared across all incidents and governed independently.
runLangGraph checkpointer (on Postgres)the live checkpoint + the conversation — the in-flight run stateThe engine's mechanism, not a schema we model. Materializes into the Mongo doc on close.
What we purge, and what we trust. We do not hand-model a bespoke run-state schema (no Session/Run/PhaseRun/Action/RunEvent/Checkpoint tables) — LangGraph's checkpointer owns the live run state, and the durable journey is already captured by the Phase Record (§4), written into the subject document on close. We keep exactly what we trust: the subject document (data) and the playbook + registry (config). The planes link by the subject — playbook@version on the subject doc, and the {domain, id} SubjectRef from the run.
The live session is runtime, not a schema. The conversation (the live chat) and the in-flight run are a runtime layer, not durable shapes — designed in the Design doc · B8: free chat → shared session; a per-session lock serializes the one run across stateless servers; a per-session channel syncs the chat across users. The data model holds only the durable record (graph + phase records + outputs); the chat history is persisted with the run and archived to the subject document on close.

Why split this way: the subject is read hot and whole during the investigation → denormalized Mongo (one read). Config is shared, versioned, governed → normalized Postgres (one source of truth). Run state is transient and recovery-critical → LangGraph already solves it; re-modelling it would be duplication to keep in sync.
