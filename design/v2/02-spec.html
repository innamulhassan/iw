Investigation Engine — Project Spec

The clean, single capture — what we're building, how it works, and what to build. Consolidates the HLD + LLD against the settled model; the full data model is the companion doc.

Infrastructure Resilience & Operations · v1.0 · for human + machine scoping · KISS · domain-neutral

Plan in a playbook · reason over a graph · act through governed capabilities · a human at every write.
1 · What we're building

A governed, in-house investigation engine

One engine runs a versioned playbook through gated, audited phases. It builds a graph of the situation, reasons over it with the model, calls governed capabilities (never tools directly), and a human approves every write. The first domain is incident triage; the same engine runs other domains (provisioning, capacity…) by loading a different playbook.

Two kinds of "no lock-in". No vendor lock-in — the engine calls abstract capabilities, so a source can be swapped without touching it. No domain lock-in — everything domain-specific lives in the playbook (types · phases · outputs); the engine, the graph shapes, the registry, governance, storage and the console are universal.

2 · How it works — the agent loop

The plan is the playbook; the model drives a governed loop; the human closes

Assess→Root cause→Remediation→Verify & close↺re-enterable
  1. Load the playbook (the run pins playbook@version) and seed the graph from the incident source.
  2. Per phase, the model plans, then loops: resolve a need → a governed capability, call it (gated by policy), fold the result into the graph (nodes · facts · edges) and log a Step — repeat until the phase's typed output is sufficient. Steps are emergent at runtime, not scripted.
  3. Every write pauses at the gate (allow / ask / deny); the operator approves / refines / denies inline. Read-only phases cannot write.
  4. Durable — checkpointed (LangGraph), exactly-once (idempotency key), resumable across pauses and crashes.
  5. Re-enterable — Verify can backtrack to Root cause with new evidence; a phase can re-run (a new attempt).
  6. A human closes — the engine never auto-closes. Feedback (separate from the run) feeds the learning loop and the autonomy ladder.

Every capability call passes govern() — effect (read/write) × access (allow/ask/deny), resolved at call time; unknown-effect → ask. Everything is logged: the journey IS the audit trail.

3 · Architecture & the capability layer

One engine, one governed layer over every source

The engine calls abstract capabilities; a governed layer binds each source by its kind, in a clear onboarding priority. Every call — whatever the kind — passes the same gate and is audited.

native MCPelse →register API as dynamic MCPelse →A2A specialist agent
Kind (priority)When · examples
native MCPthe tool already speaks MCP → use it directly.
dynamic MCP (API)a REST tool with an OpenAPI spec → register it as a dynamic MCP server, no per-endpoint code (how most tools onboard).
A2A agentdeep, per-layer specialists — App / DB / Network root-cause agents; Amelia / BladeLogic wrapped as governed capabilities.

The registry makes this concrete: Provider (a source) → DeclaredCapability (one action, synced) → CapabilityPolicy (human-owned allow/ask/deny). A playbook need resolves to capabilities by intent + effect, the policy gates each, and the call is logged as a Step. A new capability lands pending_review · deny until a human rules.

4 · The data model

A graph per run, a record per phase — domain-neutral

PieceIn brief
Investigation graphNode (kind·type·layer·labels·props·facts) · Edge (depends_on, hosted_on, suspected_cause, related_to…) · Fact (health, latency, impact_state, with an evidence_ref). One graph per run; one subject node, other incidents are references actively used.
Phase RecordPhaseRecord (plan · steps · typed output) + Step (the single ordered journey log — tool_call · reasoning · suggestion · decision). touched[] ties a step to the graph, evidence[] to the output.
Phase outputsAssessResult · RootCauseResult · RemediationResult · VerifyResult — typed per playbook; e.g. confidence carries a {value, basis}, causes carry a path, writes carry expected_effect · blast_radius · rollback.
Subject identitySubjectRef {domain, id} — domain-neutral; never a bare incident_id.
Storage (3 homes)MongoDB — the incident document (denormalized, 1/incident) · PostgreSQL — the playbook + capability registry (normalized) · LangGraph — the live run state (checkpoint + conversation), kept separate.

Every node, edge, fact, record and output — each defined → explained → JSON → tabled — is the companion Data-Model doc (incident-triage-data-model.html).

Three realistic runs — incident, related incidents, and a different domain

1 · Deep incident — cross-layer root causeincident
INC-4821 · checkout p99 0.4s→4.2s, errors 18% · P1, revenue path.
assesstopology + health + changes (rev47) + similar prior INC-4820 → AssessResult: perf, P1, suspected_locus = the DB, suggestion = failover.
root-causewalk app → DB → storage: trace (75% is the DB) → DB I/O 28ms → pay-vol disk reconstructing. → RootCauseResult: stor:pay-vol 0.9 (basis: evidence-complete + matches INC-4820), path app→db→stor; rev47 / gemini / network ruled out.

Shows: the graph walk below the unhealthy app, confidence with a basis, a gated reversible write, the learning loop reusing a prior's fix.
2 · Related incidents — detected, linked under human controlrelated
INC-4821 is already under investigation (its own session, its own focus). A second incident — INC-4830 on orders-api — pops up, looking similar (shared fingerprint; its walk converging on the same zone).
on launchthe operator opens a triage for INC-4830. The engine checks for an existing session on a similar incident → finds INC-4821's, and asks the operator: "looks related to INC-4821 — link as related, or open a new triage?"
if linkedINC-4830 joins a cluster with INC-4821 (member_of, operator-confirmed). INC-4821's cause/fix propagates as context — INC-4830 doesn't re-derive from zero. Each keeps its own session, phases, and closure.
if newthe operator declines — a fresh, fully independent triage runs. Human decides, every time.
Shows: one playbook, one incident = one focus; the engine looks out for related incidents and surfaces them — but linking into a cluster is human-controlled. No auto-merge; nothing runs "at cluster scope."

8 · Delivery

Phased — value early, autonomy earned, the incumbent retired gradually

M0 Foundation→M1 Assisted (read-only)→ M2 Remediation (gated writes)→M3 Coverage→M4 Earned autonomy

Each milestone is gated, audited, reversible; Amelia runs alongside and is dialled down only as in-house actions prove themselves — no big-bang cutover. Full by-sprint plan + timeline: the companion Delivery Plan.
