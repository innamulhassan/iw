Incident Triage — Runbook

What each phase asks, which tools it calls and what they collect, the steps, and how the incident record builds up — phase by phase.

Infrastructure Resilience & Operations · the operational detail behind the HLD playbook (§3) + per-phase models (§5.6) · worked on INC-4821 / payments-api

How to read this. Each phase below has four parts: the questions it must answer · the tools it calls and what each collects (with the access it needs) · the steps · and the data it adds to the incident record (shown growing, on the running example). Every step is a governed capability call that records what it did + the evidence (§5.5). Reads flow through; the one write (remediation) stops at the approval gate.
The record grows as the run proceeds: AssessResult → + RootCauseResult → + RemediationResult → + VerifyResult. Each phase reads what's there and adds its own typed output; a backtrack re-enters a phase with more context.

1 Assess — what broke, how bad, the context

Questions

  • Which service(s) are affected — and is it one app or several (with dependencies)?
  • How bad — impact × urgency → priority, and severity?
  • What changed recently (deploys / config)?
  • What's the blast radius, and which dependencies are unhealthy?
  • Are there related or prior incidents (same service / same change / an open major)?

Tools — and what each collects

CapabilityWhat it collectsAccess
servicenow.incident.readthe incident record — title, severity hint, reporter, affected CI, descriptionread · allow
cmdb.readthe service's CIs + dependency relationships (the topology), owner / on-callread · allow
appd.flowmap.readthe live flow map — service → dependencies, with health + golden signals per noderead · allow
slo.readSLO status + error-budget burn rate (1h / 6h / 3d)read · allow
changelog.readrecent deploys / changes (CHG records, deploy events) on the service + its depsread · allow
incidents.searchrelated / prior incidents — same service, same change, any open majorread · allow

Steps

  1. Read the incident + the triggering signal.
  2. Resolve affected_services[] from the alert / CI (primary + dependents).
  3. Load the topology (CMDB + flow map) and check health across nodes.
  4. Compute blast radius + pull SLO burn.
  5. Pull recent changes on the service and any unhealthy dependency.
  6. Search related / prior incidents and link them.
  7. Derive priority (impact × urgency) and set an initial suspected_locus.

2 Root cause — the probable cause, with evidence

Questions

  • What is the probable cause — and what are the ranked alternatives?
  • How confident are we, and what's the evidence?
  • Does it correlate with a recent change or an unhealthy dependency?
  • What can we rule out?

Tools — and what each collects

CapabilityWhat it collectsAccess
appd.metrics.readgolden-signal time series for the suspect nodes (latency, errors, saturation) around onsetread · allow
logs.searcherror log lines / stack traces for the window, grouped by signatureread · allow
traces.readdistributed traces showing where latency / errors originate across the topologyread · allow
changelog.readthe diff / details of the suspect change (CHG-77 → deploy v812)read · allow

Steps

  1. Walk the topology from the symptom toward suspected_locus.
  2. Correlate metrics / traces / logs with the recent change + unhealthy deps.
  3. Form ranked candidates; score confidence (keep the tool's native scale in confidence_raw).
  4. Rule out alternatives and record why.
  5. If top confidence < min_confidence → gather more, or ask the operator.
Data added — RootCauseResult
{ candidates:[
    {cause:"deploy v812 — connection-pool exhaustion", type:"change",
     confidence:0.86, confidence_raw:"0.86 (Watchdog)", critical_failure:true,
     method:"change-correlation + trace fault path"},
    {cause:"payments-db slow queries", type:"infra", confidence:0.31},
    {cause:"auth latency", type:"dependency", confidence:0.08} ],
  status:"found",
  evidence:[{kind:"change", ref:"CHG-77"},{kind:"logs", ref:"sig:pool-timeout x2.4k"},
            {kind:"trace", ref:"trace:payments-api→payments-db p99 9s"}],
  suspect_changes:[{change_id:"CHG-77", score:0.86, time_frame:"-12m"}],
  alternatives_ruled_out:["payments-db (no query-plan change)","auth (latency normal)"] }

3 Remediation — the safest reversible fix, gated

Questions

  • What's the safest reversible fix (often: roll back the change)?
  • What's the blast radius and the rollback plan?
  • Is it approved — and did it actually run?

Tools — and what each collects / does

CapabilityWhat it does / collectsAccess
k8s.rollbackwrites: roll the deployment back to the last healthy release (v811); returns the rollout statuswrite · ask (gated)
k8s.restart · k8s.scalewrites: restart / scale as alternatives; gated the same waywrite · ask
runbook.runwrites: a reviewed runbook step; one governed callwrite · ask
telemetry.readreads health right after the action to confirm it tookread · allow

Steps

  1. Propose the fix from the root cause (here: k8s.rollback payments-api → v811); compute blast radius + rollback plan.
  2. Approval gate — the operator reviews the plan and approves / refines / denies (recorded on the HumanInteraction).
  3. Execute with an idempotency key (exactly-once); capture evidence.
  4. On failure — retry transient errors per the playbook retry policy; permanent / blocked → error_handler (escalate / manual).
Data added — RemediationResult
{ action:{capability:"k8s.rollback", input:{service:"payments-api", to:"v811"}},
  blast_radius:"1 service", rollback_plan:"redeploy v812",
  approval:{interaction_id:"hi_91", decision:"approved", actors:["payments-oncall"]},
  execution:{status:"done", idempotency_key:"act_5f3", evidence:[{kind:"rollout", ref:"rs/payments-api-811"}]},
  on_failure_policy:"run-remaining", error_handler:null }

4 Verify & close — confirm recovery, then close

Questions

  • Did it recover — is the symptom cleared and the SLO recovering?
  • Does the user journey work (synthetic replay)?
  • Safe to close? (a human decision)

Tools — and what each collects

CapabilityWhat it collectsAccess
telemetry.readbefore / after health — error rate, latency, SLO burn trendread · allow
synthetic.replayruns the key user journey and returns pass / fail + evidenceread · allow

Steps

  1. Re-check health + replay the journey.
  2. Confirm symptom cleared + SLO recovering; capture before/after.
  3. If it didn't hold → backtrack to Root cause (with the new evidence).
  4. The operator confirms and closes — closing is a human act, never automatic.
Data added — VerifyResult
{ verification_outcome:"recovered",
  before:{error_rate:"4.2%", p99:"9s"}, after:{error_rate:"0.1%", p99:"240ms"},
  synthetic_result:"pass", symptom_cleared:true, slo_recovered:"recovering",
  resolution:"rolled back deploy v812", close_code:"resolved-by-rollback" }

6 The playbook that drives this

The process above is a versioned playbook (Markdown + YAML front-matter) — change it without changing the engine. A realistic form:

---
id: incident-triage
version: 1.2.0
domain: app-incident
status: active
owner: iro-sre
defaults:
  on_failure: run-remaining
  retry: { max: 3, backoff: exponential }   # transient errors only
unknown_access: ask                          # default gate for unknown-effect capabilities
phases:
  - id: assess
    name: Assess
    goal: Identify the affected service(s), how bad, what changed, and the context.
    capabilities: [servicenow.incident.read, cmdb.read, appd.flowmap.read,
                   slo.read, changelog.read, incidents.search]
    output: AssessResult
  - id: root-cause
    name: Root cause
    goal: Find the probable cause, with evidence and a confidence score.
    capabilities: [appd.metrics.read, logs.search, traces.read, changelog.read]
    output: RootCauseResult
    min_confidence: 0.7
  - id: remediation
    name: Remediation
    goal: Apply the safest reversible fix.
    capabilities: [k8s.rollback, k8s.restart, k8s.scale, runbook.run, telemetry.read]
    output: RemediationResult
    gate_writes: true
  - id: verify-close
    name: Verify & close
    goal: Confirm recovery from the user's side; the operator closes.
    capabilities: [telemetry.read, synthetic.replay]
    output: VerifyResult
schemas:
  AssessResult:      { type: object, required: [affected_services, impact, priority] }
  RootCauseResult:   { type: object, required: [candidates, status] }
  RemediationResult: { type: object, required: [action, approval, execution] }
  VerifyResult:      { type: object, required: [verification_outcome, resolution] }
---
## assess
Affected service(s); severity/impact + blast radius; recent changes; dependencies and which
are unhealthy; related/prior incidents; owner. Build the topology and set a suspected locus.
## root-cause
Correlate metrics/traces/logs with recent changes + dependency health; rank candidates; score
confidence. Below min_confidence, gather more or ask.
## remediation
Propose the safest reversible fix (often: roll back the change); show blast radius + rollback.
Writes are gated; retry transient failures, escalate permanent ones.
## verify-close
Re-check health and replay the user journey; confirm recovery. The operator confirms and closes.
