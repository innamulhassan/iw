Incident Triage AIOps Platform — Design

A reusable, declarative, human-governed automation platform for production support — incidents first.

Infrastructure Resilience & Operations · AIOps Incident Automation · June 2026 · High-Level Design (HLD) · v1.3 (draft) — developer detail in the companion LLD; revision history in §7.5

In one line. A self-investigating, human-governed incident engine that earns more autonomy as it proves itself — supervised-autonomy, closed-loop AIOps. The AI does the toil (gather, correlate, propose); the human owns the judgment; nothing touches prod by surprise; everything is auditable.

1 Goal & principles

1.1 What it is

A governed engine that takes an incident through four re-enterable phases — assess → root cause → remediation → verify — doing the investigative toil and proposing fixes, with a human owning every consequential decision. When an incident fires, the engine opens a working session with a chat, runs the defined phases, calls whatever capabilities each phase needs (telemetry, CMDB, specialist database/network agents, runbooks), and works alongside the Major Incident Management (MIM) team — proposing, pausing for approval, resolving. It replaces a blunt vendor auto-heal with a transparent, auditable one that scales, with no vendor lock-in.

1.2 One platform, many automations

The engine, capability layer, governance, registry, human-in-the-loop, and audit are domain-agnostic. Incident-triage is the first application; provisioning, patching, capacity, and cost follow on the same core — configuration, not a rebuild.

1.3 The autonomy ladder

The engine starts assisted and is dialed up per phase × service × severity — never globally, always behind the gate + audit — as evidence accrues:

  • Assisted — the AI investigates and proposes; a human acts. (where every phase starts)
  • Augmented — the AI acts, but a human approves each action. (the default for writes)
  • Autonomous — for low-risk, high-confidence, proven actions, the AI acts within bounds and a human supervises (human-on-the-loop).

That governed progression is how a blunt vendor auto-heal is replaced safely — autonomy is earned and reversible, not switched on. Promotion is by evidence, tracked via Feedback (§5.5): a capability climbs a rung only after a run of clean incidents (no regression) for that phase × service × severity — and a single bad outcome demotes it. The thresholds (how many, what error-rate ceiling) are an org policy, set deliberately, not guessed.

1.4 Principles

  • Own the domain, rent the orchestrator — phases, incident model, capabilities, and governance stay framework-agnostic, so the orchestrator (LangGraph today) stays replaceable.
  • Governance at the boundary, not in the prompt — the host enforces allow/ask/deny on every capability call; the model can't talk its way past it.
  • Declared vs granted — providers declare what they can do; humans grant access; anything ungranted defaults to ask.
  • Goal-driven phases — each phase works toward a goal using available capabilities, not a fixed script.
  • Earned, reversible autonomy — the ladder above, dialed narrowly, always behind the gate + audit.
  • Delegation narrows, never widens — a specialist agent (database, network) or an incumbent (Amelia, BladeLogic) is just an A2A/API capability a phase calls: it gets only the scope granted, its consequential actions surface to the same gate, and its steps nest in the audit. Delegation is a governed capability call, not an exception.

2 Components

The building blocks and what each is responsible for. Actions (§3) and Interactions (§4) reference these by name.

2.1 Phased engine — drives the four phases as a stateful, checkpointed graph (LangGraph) with durable execution + recovery (§4.6); a modular monolith scaled by stateless replicas (§7.1). It calls abstract capabilities, never tools directly.

2.2 Capability layer — the registry (the catalogue), per-kind adapters (skill · mcp · a2a · api), and the govern() boundary every call passes through. One uniform contract over five sources.

2.3 Playbook — the declarative, versioned process (Markdown + YAML front-matter) that drives both the engine and the workbench; change the process by editing the playbook, not the code. It holds only the workflow — access lives on the capability, engine config is runtime, RBAC is org config.

2.4 Governance — the dial (the autonomy level), the gate (allow / ask / deny, host-enforced), and the immutable audit.

2.5 Workbench — the operator UI, our own console (not Amelia's): the incident queue, the triage console (the run, cited evidence, and the inline approval surface), and the governance / playbook / dashboard screens. Full design in §6.

2.6 Stores — Postgres (control plane: registry, governance, engine state, audit) + MongoDB (data plane: incidents, evidence, timeline). A thin repository layer hides the split. Detail in §5.

3 Actions

What the system does — the operations the components perform.

3.1 The four phases (re-enterable)

Four goal-driven phases. Each has a goal; it works toward it with available capabilities; when the goal is met and the user approves, it advances, carrying prior phases' context forward. The order is a default, not a rigid chain — see re-enterability below.

  • Assess — what broke, how bad, and the context around it: the affected service; severity/impact + customer blast radius; recent changes/deploys; dependencies and which are unhealthy; related/active incidents; owner & on-call. (Intake + Impact, merged — one picture of the situation.)
  • Root cause — probable cause(s) with evidence and a confidence score: ranked candidates, never one asserted cause; correlate metrics/traces/logs against recent changes + dependency health.
  • Remediation — the safest reversible fix (often: roll back the change); blast radius + rollback plan; executed behind the prod gate, with an on-failure policy.
  • Verify & close — confirm recovery from the user's side (health + journey replay); resolve and close.

Re-enterable, not a rigid chain. Google SRE puts mitigation above all else — so Remediation can run before Root cause completes (mitigate now, diagnose after), and any later phase can backtrack to an earlier one to fill a gap (Verify fails → reopen Root cause; Remediation reveals the cause was wrong → back to Root cause). The phase order is the happy path; the state machine (§5.7) governs the real transitions.

The phases are declared in the playbook — the minimal form (defaults do the rest):

---
id: incident-triage
version: 1.0.0
phases:
  - id: assess
    name: Assess
    goal: Identify the affected service, how bad it is, what changed, and the context around it.
    capabilities: [servicenow.incident.read, cmdb.read, appd.flowmap.read, slo.read, changelog.read]
    output: AssessResult
  - id: root-cause
    name: Root cause
    goal: Find the probable cause, with evidence and a confidence score.
    capabilities: [appd.metrics.read, logs.search, traces.read, changelog.read]
    output: RootCauseResult
    min_confidence: 0.7         # below this -> gather more, or ask
  - id: remediation
    name: Remediation
    goal: Apply the safest reversible fix.
    capabilities: [k8s.rollback, k8s.restart, k8s.scale, runbook.run]
    output: RemediationResult
    gate_writes: true
    on_failure: run-remaining   # or fail-immediately
  - id: verify-close
    name: Verify & close
    goal: Confirm recovery from the user's side, then close.
    capabilities: [telemetry.read, synthetic.replay]
    output: VerifyResult
---
## assess
Affected service; severity/impact + blast radius; recent changes; dependencies and which are unhealthy; related incidents; owner.
## root-cause
Correlate metrics/traces/logs with recent changes + dependency health; rank candidates; score confidence. Below min_confidence -> gather more or ask.
## remediation
Propose the safest reversible fix (often: roll back the recent change); show blast radius + rollback. Writes are gated.
## verify-close
Re-check health and replay the user journey; confirm, then close.

The playbook evolves — it doesn't have to be complete on day one. It wraps whatever capabilities the enterprise already has, discovered through the registry (§3.3) rather than predefined — we don't need to know them all up front. It starts assisted and evidence-based (the AI proposes, a human acts, every step carries a citation), then earns its way up the autonomy ladder (§1.3): more capabilities wired in → broader coverage → autonomous for the low-risk, proven actions. Because the playbook is versioned (§3.5), all of this is just new versions — the engine never changes. This is the crawl → walk → run path from a vendor auto-heal we don't control to a governed engine we do.

For a general application, each phase is a small set of questions answered with capabilities; what it produces is the per-phase model in §5.6. Context accumulates and the suspected location narrows as it goes — the engine walks the topology (§5.5), it doesn't run a fixed script.

How context builds. Assess establishes the topology + affected services; Root cause walks that topology, narrowing suspected_locus with evidence; Remediation acts on the located cause; Verify confirms from the user's side. Each step's tool call + evidence (§5.5) accrues into the shared context — so a backtrack (§5.7) re-enters an earlier phase with more context, not less, and a multi-app incident is worked by walking across apps and layers until the locus is found.

3.2 The in-phase loop (identical every phase)

Propose → approve/refine → execute → report → approve. During execute, each step: a permission gate (Allow / Allow-always-this-session / Deny — reads usually auto, writes ask); the run (often a sub-agent); a possible mid-run pause for your input (the session is stateful); and a capture of result + citation + evidence (screenshots/summaries included). If it can't complete, it re-plans → you approve → resumes. Every plan — first, refined, or re-planned — passes through your approval before it runs. All captured to the DB as JSON for handoff + audit.

3.4 The governance decision

Governance at the boundary, not in the prompt. For every call the host reads the capability's effect and access, applies the dial, and decides allow / ask / deny — reads usually flow through, writes meet the gate, everything is audited. How access resolves from the policy and the defaults is in §5.3; the call sequence is §4.1.

3.5 Playbook authoring, validation & lifecycle

A playbook is two halves in one file: YAML front-matter — the machine contract the engine executes (phases, the capabilities each may use, gates, thresholds, output schemas) — and a Markdown body whose ## <phase-id> sections are the agent's guidance for that phase: plain-language intent the model follows while pursuing the phase goal. Front-matter is what is allowed and required; the body is how to think about it. (Full schema in §5.4.)

How a phase advances. A phase is goal-driven, not a script: the engine works the goal with the phase's capabilities until it is met, the operator approves, and any advance_when / min_confidence condition holds — then it carries context to the next phase. If a step fails, the engine first retries transient errors (per the playbook's retry policy — up to max, with backoff); a permanent error, or exhausted retries, then hits on_failure (stop, or finish the independent steps then report) and the error_handler (escalate / hand to manual). A phase may fire a benign on_complete hook (e.g. attach_session); closing the incident is a human action, not a hook — in Verify & close the operator confirms the fix held and closes it, so an incident is never auto-closed. Phases are re-enterable, so a later phase can backtrack to an earlier one — the full transition set is the state machine (§5.7).

Lifecycle. One playbook.schema.json validates every playbook on author (a draft saves with lint warnings), on publish (rejected if it names an unknown capability or a missing schema), and on load. Published = immutable; changes are new versions, change-controlled (reviewed before publish) with a changelog entry per version. Each incident pins playbook@version and the capability manifest it ran against, so opening a past incident replays the exact process and tools → regulator-grade audit. A new playbook can be AI-drafted from a reference → review → publish.

The complete reference playbook — the contract (front-matter) + the guidance (body), all four phases. §3.1 is the minimal form; this is the full one:

---

## assess

(services, users, customers). Read the SLO + burn rate. List changes in the last 24h and
the dependencies that are unhealthy. Link related / similar / caused-by incidents.

and the SLO is recovering. Resolve, then close.

3.6 Worked example — a driven capability (AppD browser)

End to end for the generic/driven case — and a validation of the model: Provider (status registered → connected) → discovered DeclaredCapability rows → CapabilityPolicy by default (read→allow, write→ask, overridable) → governed invoke.

  • Register — one provider row (appd-browser: command · target_url · auth_ref); auth_ref is a vault reference, never the secret.
  • Discover — on register the engine spawns the MCP and tools/list registers the capabilities; it re-syncs on tools/list_changed · periodically · on-demand.
  • Grant — default policy by trust × effect (read→allow, write→ask); a human can change it later.
  • Drive — a trusted login uses auth_ref (the AI never types creds), then the engine loops snapshot (read) → act (write, asks in chat) → capture (read → Evidence) until the goal; then close + audit.

Concretely — the one provider you author, and the calls the engine makes:

// Provider — the only row you write
{ "id":"appd-browser", "kind":"mcp_local",
  "connection": { "command":["npx","@playwright/mcp@latest","--headless","--isolated"],
                  "target_url":"https://appd.internal", "auth_ref":"vault:appd_svc" },
  "trusted": true }

// discovered + granted:  navigate·snapshot·screenshot → read·allow   |   type·click → write·ask

// the engine drives (autonomous, gated per write):
login(auth_ref)                                               # trusted · vault · AI never types creds
invoke("appd-browser__browser_navigate", {url:"…/services"})         read  → allow
invoke("appd-browser__browser_snapshot", {})                         read  → allow   # SEE the refs
invoke("appd-browser__browser_type",     {ref, text:"payments-api"}) write → ASK
invoke("appd-browser__browser_click",    {ref})                      write → ASK
invoke("appd-browser__browser_take_screenshot", {})                  read  → allow → Evidence

Local vs remote MCP — same flow, one line of difference. The driven flow above is mcp_local (the engine spawns the server). A hosted MCP is mcp_remote — identical protocol (initialize → tools/list → tools/call), capabilities, governance, and call path; only the provider's connection changes:

// mcp_local — spawn it on the box
{ "id":"appd-browser", "kind":"mcp_local",
  "connection": { "command":["npx","@playwright/mcp@latest","--headless"] } }

// mcp_remote — connect to a hosted server (same tools, same calls)
{ "id":"appd-mcp", "kind":"mcp_remote",
  "connection": { "url":"https://mcp.internal/appd", "auth":{"type":"bearer","ref":"vault:appd_mcp"} } }
// discovery + invoke unchanged: tools/list → capabilities; tools/call over HTTP/SSE instead of stdio

When to use which: mcp_local for the simplest path (spawn on the engine box, no extra service); mcp_remote when the MCP is a shared/hosted service — scaled, patched, and owned independently, reachable by many engine replicas. The engine, the capabilities, and the gate are identical either way.

3.7 Worked example — predefined Playwright scripts in a Python module

What. A reviewed Python module of Playwright scripts, bundled in the engine (e.g. skills/replays.py). Each function is a parameterized flow — service_name in, evidence out — that runs as one governed call.

How it registers & discovers. You mark each function a capability; on load the engine imports the module, scans for the marked functions, and introspects the typed signature into the input_schema — no hand-written schema, and adding one never touches the engine.

# skills/replays.py — reviewed Playwright scripts, bundled in the engine
@capability(effect="read")                        # marks it a capability + declares the effect
async def appd_service(service_name: str, window: str = "1h") -> dict:
    """Log into AppD, open the named service, capture the memory + error panels."""   # → description
    async with browser_context(storage_state=vault.get("appd_svc")) as page:          # vault login
        ...                                        # navigate · screenshot
        return { "memory_mb": ..., "screenshots": [save_blob(png)] }

@capability(effect="read")
async def appd_dependencies(service_name: str) -> dict: ...

Data model — discovered from the module:

// Provider — one row, points at the module
{ "id":"appd-replay", "kind":"skill", "connection": { "module":"skills.replays" }, "trusted": true }

// DeclaredCapability — REGISTERED by introspection (one per @capability function)
{ "id":"appd-replay__appd_service",
  "description":"Log into AppD, open the named service, capture the memory + error panels.",
  "input_schema": { "type":"object", "required":["service_name"],
                    "properties": { "service_name":{"type":"string"}, "window":{"type":"string","default":"1h"} } },
  "effect":"read" }                              // CapabilityPolicy default → allow
// also discovered: appd-replay__appd_dependencies

Use. The engine makes one governed call:

invoke("appd-replay__appd_service", { "service_name":"payments-api" })
   → read → allow → the function runs the reviewed flow (login · navigate · capture)
   → { memory_mb, screenshots } → Evidence + RootCauseResult + audit

Adding a script later = add a decorated function to the module → it auto-registers on next load (a build, since it's bundled); the engine never changes. Driven (§3.6) vs predefined (§3.7): driven = many primitive calls, gated per write (exploratory); predefined = one call, the loop is inside the reviewed function, gated once (deterministic) — same registry, same invoke contract.

3.8 Registering a skill — the Agent Skills (SKILL.md) standard

We don't invent a skill format — we adopt the Agent Skills open standard (agentskills.io; Anthropic's Skills). A skill is a folder with a SKILL.md (YAML frontmatter + instructions) plus optional bundled scripts/resources. This is the standard packaging around the bare module of §3.7 — same scripts, now wrapped with discovery metadata, instructions, and a tool scope.

The SKILL.md spec (verified): required name (≤64 chars, lowercase/digits/hyphens, not "anthropic"/"claude") and description (≤1024 chars, says what it does + when to use it); optional allowed-tools and disable-model-invocation.

incident-replays/                      # the skill folder (the unit you register)
├── SKILL.md
└── scripts/
    └── appd_replay.py                  # the Playwright flow

# SKILL.md
---
name: appd-replay                       # ≤64 · lowercase-hyphen
description: Capture AppD evidence for a service — log in, open the named service,
  screenshot the memory + error panels. Use when investigating a service in AppDynamics.
allowed-tools: [playwright, vault.read] # the capabilities this skill may use
---
# AppD replay
Steps the engine follows…              # the body — loaded only when the skill is triggered

How a user registers it: drop the folder in a skills directory — auto-discovered, no code change (~/.claude/skills, .claude/skills, or a plugin). New skills appear via the same sync that discovers any provider.

How the engine maps it (data model):

SKILL.mdOur model
the folder + nameProvider (kind: skill, connection:{path}) + namespace
descriptionDeclaredCapability.description — preloaded for selection (Level 1)
bundled scriptsa DeclaredCapability per script (input_schema introspected) — Level 3

Progressive disclosure (the standard's efficiency trick): L1 name+description always preloaded (~100 tok/skill) · L2 the body on trigger (<5k tok) · L3 scripts run via the adapter on demand (output only). Governance: allowed-tools scopes what it may call; a skill bundles executable code, so it's trusted only after review (supply-chain) — its capabilities default to ask / pending_review until granted. Invocation: model-invoked — the engine selects the skill by its description, loads the body, and runs its scripts through the governed call path (§4.1).

3.9 Worked example — an API capability (OpenAPI → operations)

What others do, and what we do: an api provider points at a REST service with an OpenAPI spec, and each operation becomes a capability — the same pattern every "tools-from-OpenAPI" generator uses. No per-endpoint code.

Register + discover. One provider row; the engine fetches the OpenAPI and generates a capability per operation — input_schema from the operation's parameters + requestBody, effect seeded from the HTTP verb (GET/HEAD → read; POST/PUT/PATCH/DELETE → write), auth from securitySchemes.

// Provider
{ "id":"appd-api", "kind":"api",
  "connection": { "base_url":"https://appd.internal/api", "openapi":"/openapi.json",
                  "auth": {"type":"bearer","ref":"vault:appd_api"} }, "trusted": true }

// discovered from the spec — one per operation:
{ "id":"appd-api__get_metrics",  "input_schema":{ …service,metric,window… }, "effect_hint":"read"  }   // GET  → read  → allow
{ "id":"appd-api__restart_node", "input_schema":{ …node_id… },              "effect_hint":"write" }   // POST → write → ask

// invoke:
invoke("appd-api__restart_node", { "node_id":"n-42" })
  → write → ASK → adapter builds  POST {base_url}/nodes/n-42/restart  + Authorization: Bearer …(vault)
  → call → validate response → Evidence + audit

Data model: Provider(kind:api) + a DeclaredCapability per operation (schema from the spec) + policy (verb-seeded default, human-overridable). The verb is a hint — we ratify the effect, exactly as for every other kind. No OpenAPI? Hand-declare the few capabilities; everything downstream is identical.

3.10 Worked example — an A2A agent capability (Agent Card → task)

Grounded in the A2A protocol (build to v0.3.0): the agent is reached by URL and publishes an Agent Card at /.well-known/agent-card.json whose skills are its callable actions; you invoke a task — two-way and streaming.

Register + discover. One provider row (URL + auth + governance_aware); the engine fetches the Agent Card and maps each skill → a capability. A2A declares no effect, so you classify it.

// Provider
{ "id":"network-triage-agent", "kind":"a2a_agent",
  "connection": { "url":"https://a2a.internal/net", "auth":{"type":"oauth2"} },
  "governance_aware": true, "trusted": false }

// discovered from the Agent Card skills (effect unknown → you ratify):
{ "id":"network-triage-agent__diagnose", "effect_hint":"unknown" }   // → read  → allow
{ "id":"network-triage-agent__failover", "effect_hint":"unknown" }   // → write → ask

// invoke = a two-way task:
invoke("network-triage-agent__diagnose", { "incident":"INC-2381" })
  → message/send (task created) → agent works, streams updates
  → may pause: input-required (asks the operator for INFO) | approval (gate a write)
  → completed → artifacts → Evidence + audit

The two pauses — both via interrupt() and one HumanInteraction (type: input | approval): input-required = the agent needs information (the operator answers, the task resumes); approval = a write meets the gate. A write the agent performs itself is gated by plan-then-approve when governance_aware, or kept read-only / least-privilege when opaque. 4 Interactions

How the components talk — the runtime flows.

4.1 The one call path — and how it loops

Every capability call — any kind — takes the same governed path. This is one step:

invoke(cap_id, input):
   cap     = resolve(cap_id)                  # → provider · kind · effect · access
   validate(input, cap.input_schema)          # reject bad args before anything runs
   gate    = govern(cap.effect, cap.access)   # allow · ask · deny
               deny → blocked
               ask  → interrupt() → operator approves / edits / denies (in the chat)
                      (reads usually allow → pass straight through)
   output  = adapter[cap.kind].call(cap, input)    # ← the only kind-specific step
   validate(output, cap.output_schema)        # if declared
   capture   output → Evidence ;  step → immutable audit
   return output

The boundary is identical for every kind; the transport is the only variable — swap an MCP tool for a skill and nothing in the engine changes.

It loops for a driven capability. The self-driving AppD browser (§3.6) wraps that one call in a loop — observe → reason → act → repeat until the goal — with the gate firing per call (the gate itself is unchanged):

drive(goal):                                   # "get payments-api memory from AppD"
   login(auth_ref)                             # trusted, once (vault)
   while not goal_met and steps < limit:
       view   = invoke("appd-browser__browser_snapshot", {})       # OBSERVE   read  → allow
       action = reason(goal, view)                                 # model picks next step + ref
       invoke(action.cap, action.input)                            # ACT   navigate→allow · type/click→ASK
       goal_met = check(goal, view)
   invoke("appd-browser__browser_take_screenshot", {})             # capture   read → Evidence
   close(); return RootCauseResult(evidence)

So the call path is the inner governed unit; the loop is the driven mode — reads flow through, writes pause for approval, every iteration audited. A predefined capability (§3.7) is the opposite: one invoke, with the loop inside the reviewed function — the engine doesn't loop at all.

4.2 Human-in-the-loop — one gate, plus A2A's mid-run ask

  • The ask — every kind. A capability with access = ask pauses before the call and surfaces in the chat, where an authorized user can approve, pick an option, or refine — not just yes/no — then it proceeds. allow flows straight through; the gate is host-side, the provider never decides it.
  • Mid-run input — A2A only. skill, mcp_*, and api are request→response: one shot, nothing to ask back. Only an A2A agent runs a multi-step task, so only it can hit input-required and ask the operator a question mid-run.

Both pauses flow through the one HumanInteraction record (type: approval | input) and the same interrupt()/resume machinery — so the engine and workbench handle them uniformly. The whole HITL story: one gate for writes across all five kinds; one extra mid-run "ask" that only A2A can trigger.

4.3 Incident intake & the run, end-to-end

From signal to run. The front door is an ingestion endpoint: a Signal arrives from monitoring (a webhook from AppD / Splunk / Prometheus / PagerDuty), normalized to {source, service, severity_hint, fingerprint, payload}. The engine dedups & correlates by fingerprint + service + time-window — a matching open incident attaches the signal (no duplicate); otherwise it creates one, sets an initial severity (rules) and owner (CMDB on-call), and starts a Run on the domain's playbook@version → Assess begins. This read-only intake is the only un-gated step; everything consequential downstream still passes the gate (§4.4).

One separate triage session per incident (no merging). A Session is the run: it moves through the phases, and each phase produces a PhaseResult whose steps record every capability call, the evidence it produced, and any approval — so the trail reads session → phases → steps → evidence → timeline. Incidents link to others (related-active / similar-past-with-fixes / caused-by) for leverage + learning, never merged. The full entities are in §5.5; a worked row-by-row run is in §5.9.

Why this shape. A prod-support team trusts the engine only when it can see the steps and citations — so the captured plan + cited evidence per phase is at once the work surface, the shift-handoff artifact, and the audit record.

4.4 Access & approval

Access is layered on what the bank already runs, plus the one gate the platform adds — three layers, nothing duplicated:

  • 1 · Domain access — an AD group. Membership in a domain's AD group (incident-triage, provisioning, …) is what lets a user in. The platform checks the group; it does not build its own RBAC — it leverages enterprise IAM.
  • 2 · Capability layer. The registry of what can be done; each capability carries an access — allow · ask · deny (§5.3).
  • 3 · The ask. When a capability's access is ask, the engine pauses and surfaces it in the chat, where an authorized user can approve, choose an option, or refine — not just yes/no. The AI proposes but never approves its own action; every proposal, decision, and outcome is audited.

The gate is the governance layer (what's permitted); the interaction — the asking, the options, the refining — lives in the chat (§3.2). Keep them distinct.

4.5 Session & resource lifecycle — clean, enforced teardown

Two things end, both deterministically — the engine session (the run) and the tool resource leases it held (the stateful ones: a Playwright browser, an open MCP connection). Cleanup is the host's job — a finally-style guarantee — never something the model has to remember.

Scoped leases + enforced teardown. A stateful tool is acquired as a lease bound to a scope (the phase, or a browser session inside the driven loop). The engine tracks open leases and tears them down when the scope exits — on success, error, cancel, or timeout:

lease = acquire(appd-browser)          # spawn MCP + browser context
try:
    drive(goal)                        # the loop
finally:
    lease.close()                      # ALWAYS — close tabs → context → browser → kill subprocess
                                       # idempotent · audited

Browser teardown order: pages (tabs) → context → browser → terminate the MCP subprocess — idempotent (safe if already gone) and audited. The model never calls close itself; the finally runs even if it forgets, errors, or the operator rejects a step.

How the session endsWhat the engine does
Normal — operator closes at verify-closeclose all leases · finalize audit · checkpoint · SessionStatus = completed
Canceled / failedsame teardown path · status = canceled | failed
Paused (HITL) — shorthold the lease (browser stays open) · resume in place
Paused — long / abandoneda reaper closes the lease past its TTL (scheduler, leader-elected) · re-acquire idempotently on resume (re-login from vault)

Scoped leases + finally teardown + idempotent close + a TTL reaper — tabs/browser/subprocess are always closed when the work or the session ends, abandoned sessions are reclaimed, and crashes are covered because the lease is tied to the session, not to the model. No orphaned browsers.

4.6 The engine — durable execution & recovery

The engine is the runtime that drives a playbook against an incident. It is built for durable execution: the run's state is persisted at every step, so a run survives pauses, crashes, and hours-long waits — and never double-executes a write. LangGraph is the framework (§7.3); the durability bar is Temporal/Conductor-grade.

How a run executes. Load playbook@version → build the phase graph → open a Run (a durable thread). At each step the engine writes a checkpoint — a snapshot of where it is, what's next, and what's pending. The persisted state, not memory, is the source of truth. An append-only event log records every transition — it is both the audit trail and the recovery log. A capability call (an Action) runs once; its result is recorded as Evidence, and on replay the recorded result is reused, never re-run.

Pause & resume (human-in-the-loop). A write — or an agent's question — interrupts the run; the engine persists the waiting state and releases held resources (§4.5), so a parked run holds nothing. A human signal — an approval or an answer (the HumanInteraction) — resumes it from the checkpoint.

Crash recovery. If a worker dies mid-run, another reloads the last checkpoint and continues. Completed actions are not re-executed (their results are in the log); everything else is guarded by idempotency.

Retry & idempotency — bank-grade. A failed action retries from itself, not from the top of the run. Every write carries an idempotency key, so a retry or a crash-replay executes it exactly once — no double restart, no double refund. This is the one place we go beyond stock LangGraph checkpoints, which alone do not guarantee exactly-once side effects.

Backtrack & time-travel. Because every step is checkpointed, re-entering an earlier phase (§5.7) restores from that phase's state — time-travel, not a cold restart.

Concurrency. Each incident is an independent run, isolated from the rest; many run in parallel. Per-run leases + the TTL reaper (§4.5) bound resources.

The use cases this has to cover, end to end:

Grounded in: Temporal (append-only event history, replay, activities run once, signals for external input); LangGraph (thread + checkpoint / StateSnapshot, interrupt/resume — our framework); Netflix Conductor (workflow = state-machine of tasks, central persisted state as source of truth, retry-from-task, exactly-once); idempotency / retry-safety patterns for distributed workflows. The one gap we close ourselves: stock checkpoints are not exactly-once → we add idempotency keys + the event log + the reaper.

5 Data model

Three clusters — the governed Registry, the versioned Playbook, and the Run — across two storage planes.

5.1 Storage planes

The control plane lives in Postgres: config/registry, providers, capabilities, policies, playbooks, session + checkpoint, audit, decisions, and the plan & execution record — everything you search, join, gate, and audit on. Inside Postgres, scalar/spine fields (ids, FKs, enums, status, timestamps) are relational columns, and variable documents are JSONB with their searchable fields projected to indexed columns — so you keep the full JSON and a fast relational view. The playbook keeps its MD + front-matter form in Postgres (body_md text + parsed columns + phases JSONB); large binaries (screenshots) go to an object store, referenced by id.

The two layers map, they don't compete. The incident state is the human status; the run/phase state is what the engine is doing. New = just ingested (§4.3) · In Progress = a run is active (Assess → Root cause → Remediation) · On Hold = the run is waiting_approval / waiting_input / blocked (carrying the reason) · Resolved = Verify passed · Closed = the operator confirmed resolution and closed it (Verify & close) — never automatic. One incident state can span several phases.

The transitions — and the scenarios to cover end-to-end:

ScenarioWhat happens
Normal advanceactive → done when the goal is met and the operator approves; the next phase goes pending → active, prior context carried forward.
Approval gateA write puts the phase in waiting_approval; approve → active proceeds, deny → canceled (or re-plan). Recorded in approval{actors[]}.
Mid-run inputAn agent needs information → waiting_input; the operator answers → back to active (A2A two-way tasks, §3.10).
Blocked after approvalApproved, but the action can't run — resource unavailable, agent can't act, quota, timeout → blocked. The error_handler fires → retry / escalate / hand to manual. Approval ≠ success — this is the gating gap you flagged.
Partial successA multi-target fix half-applies → partial; an action with an array input (e.g. nodes[]) splits per item, so the engine retries only the failed items (per_item_results[]), or compensates/undoes the applied ones (saga).
Sub-agent failsAn A2A sub-agent's task is an Action like any other — if it errors or times out it goes failed / blocked and the phase's on_failure + error_handler apply. No special path.
On-failure policyPer the playbook: fail-immediately (stop the phase) or run-remaining (finish independent steps, then report). Grounded in StackStorm Orquesta.
Mitigate before RCARemediation runs before Root cause finishes — mitigate now, diagnose after. Phases are re-enterable, so this is a legal transition, not an exception.
BacktrackA done phase re-opens an earlier one: Verify fails → reopen Root cause; Remediation shows the cause was wrong → back to Root cause. The earlier phase goes done → active again.

Every transition is recorded as a RunEvent (§5.8) — the audit trail is the state-machine history — and projected to the human-facing TimelineEvent (§5.5). Checkpoints make every paused state durable and resumable.

5.8 The engine runtime model — run, action, event, checkpoint

What the engine needs to run a playbook and to recover: the durable run, the per-phase execution, the action (a governed capability call), the append-only event log, and the checkpoint. This extends §5.5 with the execution-level entities. Control plane in Postgres; large payloads / evidence in Mongo + object store (§5.1).

RunEvent is the control-plane append-only log in Postgres — the engine's source of truth and recovery log. TimelineEvent (Mongo data-plane, §5.5) is the human-facing projection of it that the UI timeline and audit read: one is the machine record, the other the readable story.

How playbook, session, and action relate — the spine:

Playbook (the versioned program, §5.4) → Run (one execution against one incident) → PhaseRun (the run inside a phase) → Action (one governed capability call — gated, retried, idempotent, evidence-producing). A Checkpoint freezes the Run between steps (resume · recover · backtrack); a RunEvent records every transition (audit = recovery log).

The Action is the single chokepoint where everything the design promises converges: it names a capability_id (the registry resolves the provider + kind, §5.3), it passes the governance gate (§4.4) — which may open a HumanInteraction — it carries an idempotency key (writes are exactly-once, §4.6), it records attempts (retries are bounded), and it emits Evidence + a RunEvent (the audit is complete). Read one Action row and you can reconstruct what the engine did, why it was allowed, and how to undo it. Audit scope: every Action (capability · input · output · approval · evidence) and every RunEvent are captured, plus the operator chat (§6.2). The model's intermediate reasoning is summarized in the chat narration, not stored as a verbatim token trace — the audit answers what was done, why it was allowed, and how to undo it, which is the bank-grade bar.

6 The operator UI

The UI is the human surface of everything in §1–§5 — our own console, not Amelia's. One web workbench, role-gated by AD group (§4.4). Its defining property: it is read models projected over the data model (§5) — it stores no state of its own. The approval card is the HumanInteraction (§5.5); the timeline is the RunEvent log (§5.8); the incident context is the AssessResult (§5.6); the registry screen is the declared-vs-granted pair (§5.3). The UI therefore cannot drift from the engine — every screen is a view of something the design already defines, and every action it offers goes back through the engine + the governance gate, never around them.

7 Build, operate & open items

7.1 Engine topology & scale

v1 = a modular monolith, scaled by running N stateless replicas behind a load balancer. No microservice split, no Kafka/Redis in v1; modular seams allow later extraction without a rewrite. Backing stores: Postgres (control plane — checkpointer, work queue, registry, audit) + MongoDB (data plane — incidents, evidence, timeline). A thin repository layer hides the split.

7.2 Deployment

Kubernetes — one image, one Deployment + HPA, two managed DBs, object store (evidence), OTel → SIEM/Langfuse. The scheduler/timers module uses leader-election so timers fire once across replicas.

7.3 Framework & lock-in posture

LangGraph (MIT, open-source) for orchestration — most mature for stateful, checkpointed, human-gated pipelines. Use the OSS library + Postgres checkpointer + Langfuse; never adopt the proprietary LangSmith → zero license lock-in. Credible alternatives if the org pivots: Pydantic AI (lowest lock-in) and Microsoft Agent Framework (enterprise/Azure). Custom UI uses AG-UI over SSE. Principle: own the domain, rent the orchestrator — keep phases, incident model, capabilities, and governance framework-agnostic so the choice stays reversible.

7.4 Open items / next steps

  • Confirm: Slack/Teams bridge · the AD groups that gate each domain · the stack (queue tech, managed Postgres/Mongo, K8s distro).
  • Bank-grade hardening (its own pass — gates production writes) — secrets & encryption, immutable audit, failure-mode + DR (RTO/RPO), input / injection validation, the test strategy (dry-run / shadow).
  • From the design audit — flesh the feedback → autonomy promotion thresholds (org policy, §1.3); a consult-a-human TTL + escalation + ACL; specify the synthetic.replay capability + per-stack tool mappings (AppD → Datadog / Dynatrace, Splunk → ELK …); the relationship-detection mechanism (a correlation step in Assess); and standardize the capability_id notation across all examples (provider__action).
  • Later passes — multi-domain governance (one engine, many playbooks) · data portability / schema export.
  • Build: scaffold the engine + the UI (§6) to this design. Resync the Word doc / build.js (HTML is the source of truth).
