Incident Triage AIOps — System Brief

A governed, in-house engine for production incident support — one console, one capability layer, no vendor lock-in.

Infrastructure Resilience & Operations · for leadership · full design in the HLD + LLD

1 · The system

What it is, and how it connects

One engine drives the work and calls abstract capabilities — never tools directly, so it never changes when a source does. One governed capability layer sits over every source and binds each by its kind, in a clear priority: native MCP where a tool already speaks it; else register the API (its OpenAPI spec) as a dynamic MCP — no per-endpoint code, the way most REST tools onboard; else A2A for the deep, per-layer specialist agents — App, DB, Network root-cause, plus Amelia / BladeLogic wrapped as governed capabilities. Whichever kind, every call passes the same allow / ask / deny gate and is audited. The operator works in one console.

2 · How it runs

The engine drives a playbook — the process is a document, not code

The engine takes an incident through four re-enterable phases, gated and audited. The process lives in a versioned playbook; change how incidents are handled by editing the playbook — the engine never changes.

Assess→Root cause→Remediation→Verify & close↺ re-enterable
Every phase asks a few questions, calls the capabilities it's allowed, and records what it did + the evidence.
Every write pauses for approval, in the conversation; closing is always a human act.
Durable — checkpointed, exactly-once, resumable; it survives pauses and crashes.

The process is a real, versioned Markdown file — front-matter the engine executes, a body the agent follows. The essentials, in brief:

---
id: incident-triage   ·   version: 1.2.0   ·   domain: app-incident
defaults: { on_failure: run-remaining, retry: {max: 3, backoff: exponential} }
unknown_access: ask                    # default gate for unknown-effect capabilities
phases:
  - { id: assess,       capabilities: [servicenow.incident.read, cmdb.read, slo.read, …], output: AssessResult }
  - { id: root-cause,   capabilities: [appd.metrics.read, logs.search, traces.read],      output: RootCauseResult, min_confidence: 0.7 }
  - { id: remediation,  capabilities: [k8s.rollback, k8s.restart, runbook.run],           output: RemediationResult, gate_writes: true }
  - { id: verify-close, capabilities: [telemetry.read, synthetic.replay],                 output: VerifyResult }
---

3 · The playbook

The complete playbook — the format we build to

The whole incident process is one versioned Markdown file: front-matter the engine executes (the phases, the capabilities each may call, the gates and defaults) and a body the agent follows. This is the actual incident-triage.playbook.md — to change how incidents are handled, you edit this file; the engine never changes.

---
id: incident-triage
version: 1.2.0
domain: app-incident
status: active
owner: iro-sre
changelog:
  - { version: 1.0.0, date: 2026-06-17, by: iro-sre, note: "First published process." }
  - { version: 1.1.0, date: 2026-06-17, by: iro-sre, note: "Added retry + unknown_access defaults." }
  - { version: 1.2.0, date: 2026-06-18, by: iro-sre, note: "Multi-app topology in Assess; closing is human." }

# defaults are inherited by every phase unless the phase overrides them
defaults:
  on_failure: run-remaining                  # finish independent steps, then report
  retry: { max: 3, backoff: exponential }    # retry TRANSIENT errors only; permanent -> escalate

# this playbook's default gate for capabilities whose effect is unknown
unknown_access: ask                          # deny | ask | allow

phases:
  - id: assess
    name: Assess
    goal: Identify the affected service(s), how bad it is, what changed, and the context around it.
    capabilities: [servicenow.incident.read, cmdb.read, appd.flowmap.read,
                   slo.read, changelog.read, incidents.search]
    output: AssessResult

  - id: root-cause
    name: Root cause
    goal: Find the probable cause, with evidence and a confidence score (ranked candidates, never one).
    capabilities: [appd.metrics.read, logs.search, traces.read, changelog.read]
    output: RootCauseResult
    min_confidence: 0.7                        # below this -> gather more, or ask the operator

  - id: remediation
    name: Remediation
    goal: Apply the safest reversible fix (often, roll back the change).
    capabilities: [k8s.rollback, k8s.restart, k8s.scale, runbook.run, telemetry.read]
    output: RemediationResult
    gate_writes: true                          # force the approval gate on every write here

  - id: verify-close
    name: Verify & close
    goal: Confirm recovery from the user's side; the operator confirms and closes.
    capabilities: [telemetry.read, synthetic.replay]
    output: VerifyResult

# output contracts, versioned with the playbook (validated on author, publish, and load)
schemas:
  AssessResult:      { type: object, required: [affected_services, impact, priority] }
  RootCauseResult:   { type: object, required: [candidates, status] }
  RemediationResult: { type: object, required: [action, approval, execution] }
  VerifyResult:      { type: object, required: [verification_outcome, resolution] }
---

## assess
Identify the affected service(s) — it may span several apps and their dependencies. Establish
severity / impact + blast radius; pull recent changes; load the topology and check health across
nodes; link related / prior incidents; set the owner and an initial suspected locus. Derive
priority from impact × urgency (read-only, the operator can override).

## root-cause
Walk the topology from the symptom toward the suspected locus. Correlate metrics / traces / logs
with recent changes and unhealthy dependencies. Produce ranked candidates with evidence and a
confidence score — never a single asserted cause. Rule out alternatives and record why. If the top
candidate is below min_confidence, gather more evidence or ask the operator.

## remediation
Propose the safest reversible fix — usually rolling back the offending change. Show the blast
radius and the rollback plan. Every write is gated: the operator approves, refines, or denies
the plan before it runs. Execute with an idempotency key (exactly-once). Retry transient failures
per retry; a permanent failure or a blocked action goes to the error handler (escalate / manual).

## verify-close
Re-check health and replay the user journey (synthetic). Confirm the symptom is cleared and the SLO
is recovering; capture before / after. If recovery did not hold, backtrack to root cause with the
new evidence. Closing the incident is a human action — the operator confirms resolution and
closes; the engine never auto-closes.

The output each phase must produce — the data model

Every phase returns a typed result, validated against the schemas above — a small typed core (queryable) plus a JSONB extension for source-specific richness, so new sources need no schema change. Within a phase, every step is recorded in one universal shape — an Action (the capability called, its input and output) plus its Evidence — so the audit trail is identical across all phases.

PhaseOutputCore fields  (typed; + JSONB extension)
AssessAssessResultaffected_services[] · impact · priority · severity · topology · suspected_locus · links[]
Root causeRootCauseResultcandidates[] (cause · evidence[] · confidence) · status · ruled_out[]
RemediationRemediationResultaction (capability · input · output) · approval · execution (idempotency_key · status · rollback)
Verify & closeVerifyResultverification_outcome · resolution · recovery_confidence · slo_restored_pct · watch_window

4 · The operator UI

One workbench — watch the engine think, decide at the gates

The only UI, in the Claude-Desktop three-pane shape: Incidents on the left; in the center, Triage — the entire incident's chat, where the agent's steps, evidence, and suggestions appear and the approval gate is inline; and on the right, Playbook & Progress — the phases, their steps, and each step's output. It evolves with the operator as autonomy is earned.

5 · Delivery

Phased — value early, autonomy earned, the incumbent retired gradually

Five milestones over ~14 two-week sprints. Each is gated, audited, and reversible; Amelia runs alongside and is dialled down only as in-house actions prove themselves — no big-bang cutover.

M0 Foundation · S1–2→ M1 Assisted · S3–5→ M2 Remediation · S6–8→ M3 Coverage · S9–11→ M4 Autonomy · S12–14
M1 — assisted triage (read-only): faster diagnosis, no write risk.
M2 — governed remediation (gated writes): MTTR down, behind hardening.
M4 — earned autonomy for proven low-risk actions; human-on-the-loop.

Full by-sprint plan + the timeline (Gantt) + the parallel hardening track: the Delivery Plan.

Appendix · reference

What the artifacts actually look like

For the architecture and engineering review — how a capability is declared and governed, and the data a single run produces. These are the real shapes the platform reads and writes; the worked example follows one incident, INC-4821 on payments-api.

A · The capability layer — declared (machine-synced) vs granted (human-owned), and how govern() decides

# DeclaredCapability — synced from the provider's manifest (read-only, machine-owned)
- id: k8s.rollback
  kind: mcp_local            # skill | mcp_local | mcp_remote | a2a_agent | api
  provider: k8s-mcp
  effect_hint: write
  input:  { deployment: string, to_revision: int }
  output: { revision: int, status: string }

# CapabilityPolicy — human-owned, sticky (the grant that actually governs the call)
- capability_id: k8s.rollback
  effect: write              # may override effect_hint
  access: ask                # allow | ask | deny
  trust:  trusted

# govern() — resolved at call time, every time:
effect = policy.effect ?? declared.effect_hint ?? unknown
access = policy.access ?? (effect == unknown ? playbook.unknown_access
                                              : DEFAULT[trust][effect])
# -> here: effect = write, access = ask   =>   pause for approval, then invoke, then audit
