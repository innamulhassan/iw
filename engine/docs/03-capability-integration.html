Capability Integration

How the engine reaches your tools — a single, uniform, pluggable seam — and exactly what to request from each owning team to connect the real systems. Today every tool runs against a built-in simulator; wiring a real one is a small, isolated change.

01How a tool plugs in

Every tool the engine uses — monitoring, topology, logs, source control, the deployment platform — connects the same way: through a small adapter. An adapter has two halves, and only one of them changes when you go from the simulator to the real thing.

An adapter is a translator plus a connector. The translator turns whatever a tool returns into the engine's standard shapes (entities, facts, events). The connector is how it actually reaches the tool. The translator never changes; you only swap the connector from "simulator" to "real."

the capability layer is the same for every tool; only the adapter differs

When the engine wants data, the capability layer does three things in order: resolve the request to the right adapter, permit it (a change to a live system is refused unless a human has approved it), then fetch and translate. Every call is recorded for audit.

Safety is built into the seam. A read is always allowed. An action that changes something is checked before anything is contacted — so a fix can never run ahead of its approval.

02Three ways to connect

A tool is reached in one of three styles. The engine doesn't care which — it's just a property of the adapter — so a tool can move between styles without any other change.

MCP Model-tool protocol

The tool (or a thin wrapper) exposes a standard tool-server; the engine calls capabilities by name. Best for rich, evolving systems.

REST Plain HTTP

A simple HTTP/JSON endpoint. Best where the tool's API is straightforward and stable.

A2A Agent-to-agent

A governed action is handed to another system that executes it. Used for the one write capability, so execution and its record live on that system, not here.

03Identity resolution — one CI, many tool identities

A tool is only useful if you query it with the identifier it understands. The same service is "orders-api" in the incident, an app-id in APM, a repo in source control, a workload on the platform. iw makes that explicit and self-describing — so each tool is queried by its own id, never by reusing the display name.

The idea. The incident names a CI. From it, iw resolves the CI's per-tool identifiers, and each capability declares which id it needs. So the reasoner queries APM by app_id, source control by repo, the platform by k8s_workload — the realistic chain "get the app-id from the incident, then query APM with it."

CapabilityQueries byThe id it needs
APM (topology / traces)app_idthe application's APM id
Source control · build/artifactsrepothe repository
Deployment platformk8s_workloadthe namespace/workload
Metrics · logs · topologyservice_namethe service name/labels
ITSM · event correlationincident_idthe incident record

Two things make this composable:

  • Each capability is self-describing. It carries a small meta — a one-line purpose and a queries_by field naming the id it needs. The tool catalogue the reasoner sees is rendered from this, so nothing tool-specific is hardcoded; adding a capability makes the reasoner aware of it automatically.
  • The CI is the identity backbone. The service the incident affects carries app_id · sys_id · repo · k8s_workload, resolved off the incident record. The reasoner reads a capability's queries_by, picks that id off the CI, and passes it.
Proven live. Given only this metadata, the model queried APM with the resolved app_id and source control with the repo — each tool by its own id, with nothing about a specific tool written into the engine or the prompt.

05What to request from each team

To connect a real system, ask its owning team for read access plus the small mapping the adapter needs. Below is the checklist per capability.

Event correlation (AIOps)

MCPread

The layer above monitoring — it collapses an alert storm into one correlated incident with its affected-service blast radius and co-firing / related incidents. The first tool the investigation taps, so it starts de-noised and enriched.

Request from the AIOps / Event-Management team
  • Read access to the correlated incident, its member alerts, the affected services, and the related/co-firing incident cluster (a tool-server, or the platform's REST API to wrap).
  • The correlation confidence per related incident, and the mapping from a member alert to its service — so the investigation inherits the blast radius and the "N services saw the same" prior.

Incident & change record

MCPread

The origin of the investigation and the change context around it.

Request from the ITSM team
  • An integration user with read access to incidents, changes, and configuration items.
  • The endpoint (a tool-server or the table/read API) and its credentials.
  • The field mapping for change → affected asset, and how related incidents are correlated.

Metrics & alerts

RESTread

The quantitative symptom — and the proof it recovered.

Request from the Observability team
  • The query endpoint and an alerts endpoint, with a read token.
  • The metric names and labels for the standard signal families (rate / errors / latency, and utilisation / saturation for pools, queues, hosts).

Application topology (APM)

MCPread

Service maps and the call boundaries that separate a caller's fault from a callee's.

Request from the APM team
  • Read access to the app's service map, traces, snapshots, and health rules.
  • The identifiers for the relevant applications and transactions.

Configuration & dependencies (CMDB)

MCPread

The declared dependency spine that seeds the topology.

Request from the CMDB / Platform team
  • Read access to configuration items, their relationships, and their attributes.
  • Where available, the discovery source and last-seen time per item (this feeds provenance).

Logs & signatures

MCPread

The textual evidence — error signatures, traces, and policy-denial records.

Request from the Logging / SIEM team
  • A read account scoped to the relevant indexes.
  • The index map for application errors and for security/policy denials.

Deployment platform

MCPread

Rollout and workload state — deployment and infrastructure faults.

Request from the Container Platform team
  • A read-only credential (get / list / watch) on workloads, events, and rollouts, scoped to the relevant namespaces.
  • Access to workload logs and rollout history.

Source control

RESTread

Diffs, blame, and pull-request context — a change resolved to the exact line.

Request from the source-control admins
  • A read token (contents + pull requests) scoped to the relevant repositories, and the API base.
  • The mapping from a service to its repository.

Build & artifacts

MCPread

The build → artifact → promotion lineage.

Request from the Build / Release team
  • Read access to builds, artifacts, and promotion history, with the registry base and a token.

Remediation action

A2Awrite · human-gated

The only capability that changes anything. The reasoner proposes a concrete, reversible action; the run stops for a person to approve it; on approval the action is handed to the executing system — the engine never runs it directly.

Request from the Change / Automation team
  • An endpoint (runbook automation or an orchestration hook) that accepts a typed action and returns a recorded result.
  • The approved catalogue of reversible actions (roll back a deploy, restore an index, revert a policy, reschedule a job, scale a pool) with the authorization and audit contract.
  • Confirmation that each action is reversible and idempotent, and logged on that system.
Boundary. The engine proposes and records; the executing system and the human own the consequence. It never enters credentials or performs a destructive action on its own.

06Going from simulator to real — one at a time

Because each adapter is isolated behind the same seam, you connect real systems independently. One capability can go live while the rest stay simulated — nothing else is affected.

StepWhat happens
1 · Keep the translatorThe part that turns a tool's response into the engine's shapes is already written and tested. It does not change.
2 · Swap the connectorPoint that one adapter at the real endpoint instead of the simulator.
3 · Map any differencesIf the real response differs from the simulated one, adjust that adapter's field reads — the single place a vendor quirk lives.
4 · VerifyThe simulated suite keeps running for repeatable checks; a small live check exercises the real connection for that one capability.
