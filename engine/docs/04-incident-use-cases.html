01What this page walks through

Docs 01–03 describe the machine; this page shows it running. The subject is the first packaged incident, INC-4821 — a code regression — driven by its deterministic scenario twin through the real engine, the real reducer, the real belief arithmetic, the real write gate, and the real journal. The twin is the same script the golden suite replays in CI, so everything below is reproducible to the byte.

Plain version. A payments service starts throwing errors thirteen minutes after a deploy. The investigation frames the symptom, puts up two competing explanations — the deploy shipped a bug vs the database is drowning — and then goes looking for evidence that could kill each one. The database turns out to be healthy (rival refuted); the error traces to one line of the deployed commit (leader earns its confidence). Only then does the engine let the run propose a fix — and the fix stops at a human, who sees the action, the theory, and the evidence before approving. After the rollback the symptom clears, the theory is confirmed, and the whole story — every step, question, answer and rejection — is one append-only journal you can replay.

Provenance of every number. The scores (0.6 → 0.933), the weights (0.4921), the seq numbers (1–10), the gate payload, the quarantine spellings — all were produced by running tests/e2e/scenario_code_regression.py through Engine.run() / SessionManager.create() against playbooks/incident.yaml on this tree, plus one reducer probe for the airlock section (§05). Nothing is invented; where the walk says a fact id, that id is what registry.fact_id() minted.

02The cast — one incident, three artifacts

A runnable incident is three pieces of data, no code: a catalog entry (what the start selector shows, plus the remediation the gate will propose), a scenario twin (the deterministic script + fixtures), and the playbook every incident shares. The registry that binds them is runtime/scenarios.py.

    return subject, [frame, triage, hypothesize, investigate, remediate, verify, close]
  • The subject is the identity. SubjectRef{domain, id, kind}; subject.key = "app-incident:INC-4821" is the session id and (via safe_key) the on-disk folder name. Deliberately never a bare incident id.
  • The twin is a list of PlanOutputs authored with the terse helpers in tests/e2e/_helpers.py — node() · fact() · event() · edge() · propose() · update() · no_evidence() · phase(). fact() emits the native AddAssertion atom (species inferred by the same §9.1 classifier the reducer uses), so the scripted path exercises exactly the grammar a live model emits.
  • Two planners, one engine. The same incident runs under ScriptedPlanner (this page’s numbers, and CI’s) or under the real-LLM LivePlanner with the richer runtime/live_fixtures.py blobs (golden_root = "code_commit:abc123" — the convergence check, never shown to the model). Engine, reducer, hypothesis store and journal are identical either way; only the judgment author changes.
  • The gate is injected, not scripted. _with_gate(script, remediation) adds an apply_remediation WRITE call to every REMEDIATE step for interactive sessions only — so the operator gets an Approve / Refine / Deny card. The batch/golden path never sees it and only proposes the fix (§04, ACT).

Frame — seed the symptom, the topology, the candidate change

The frame turn lands the anomaly node (onset_value 0.40, severity_score 2), the service’s RED snapshot (red_errors 0.40, red_rate 820 rpm, red_latency_p50 58 ms — throughput holding, p50 sane: a code-fault shape, not a saturation one), and the candidate change with a correlated_with edge at med. In the live path the same shapes arrive through adapters instead of direct ops — and the ServiceNow blob carries the piece that makes a cross-tool investigation possible, the identity backbone:

The adapter copies those per-tool identifiers onto the SERVICE node’s props, so every later tool can be asked in its own vocabulary — and the reducer’s alias backbone (sys_id→servicenow · app_id→appd · repo→git · k8s_workload→k8s) resolves any re-arrival of the same entity under a different display name onto one node instead of minting a twin.

Because correlate_timeline is in this playbook’s allowed intents, the engine also hands the planner a computed hint — deterministic, replay-invisible, never a graph mutation:

R-J2 in action: the change led onset by 780 s; the combined clock-skew bound of the two sources involved (servicenow 300 + prometheus 30) is 330 s. Only because 780 > 330 may “the change came first” be asserted — inside the bound the correlation would surface with the ordering claim explicitly withheld.

The loop — two rivals, then evidence that can kill one

Hypothesize puts up a genuine differential, not one idea (the doctrine requires a rival rooted at a different node):

# the twin, verbatim (helpers: propose/update from tests/e2e/_helpers.py)
propose("h1", "v4.12.0 (commit abc123) introduced a NullPointerException in TaxCalculator",
        "med", root=COMMIT)                       # band 0.6 · root_candidate=code_commit:abc123
propose("h2", "payments-ora connection-pool exhaustion", "low", root=DB)   # band 0.3

Each ProposeHypothesis mints both a store record and a graph node (hyp:h1, hyp:h2 — R-G2: causal edges need a node to attach to). The investigate turn then gathers evidence against both theories, and updates them through the only channel belief moves on — a delta with a basis:

db_fact  = fid(DB, "conn_pool_util", T_INV)       # fact ids are deterministic: sha1(subject|predicate|valid_from)
err_fact = fid(ERRSIG, "count", T_INV)

update("h2", status="refuted",   add_refuting=[db_fact],
       basis="pool util 28%, 56/200 connections, replication current — DB healthy, not the cause")
update("h1", status="supported", level="high", add_supporting=[err_fact],
       basis="NPE stack traces to TaxCalculator.java:88, only present in v4.12.0; 152 occurrences since onset")

Earned belief — the three factors, executed

The LLM band survives only as the prior; the score the gate reads is computed from the graph (hypothesis/belief.py). Each evidence fact weighs w = reliability × temporal_proximity × topological_specificity:

fact (real id)subjectreliabilityproximityspecificityw
fact:da70219d53e3f4c5
count = 152 (supports h1)		 error_signature:npe-taxcalc 0.98 (splunk) 0.7846 0.64 (2 hops → 0.8²) 0.4921
fact:f09f2c94be57f5d3
conn_pool_util = 0.28 (refutes h2)		 database:payments-ora 0.99 (prometheus) 0.7756 0.64 (2 hops → 0.8²) 0.4914
  • Reliability is the fact’s own belief channel — a measured fact’s source_reliability, filled by the reducer from the playbook’s per-source map when the adapter states none (INV-9: adapters carry no constants).
  • Proximity is skew-tolerant (R-J2): the reads are at 14:12, onset at 14:00 — 720 s apart. Subtract the combined skew bound first (splunk 60 + prometheus 30 = 90 s for the count fact), then decay the 630 s excess: 0.5^(630/1800) = 0.7846. Inside the bound the factor is exactly 1.0 — never a false-precision penalty.
  • Specificity decays per structural hop from the anomaly (0.8^hops, floor 0.25) — and hops are counted over observed structure only: the BFS never crosses an inferred edge and never enters a hypothesis node, so a theory’s own causal claims can never shorten the distance of its own evidence.
# hypothesis/belief.py weighted_score — the blend, with this run’s numbers
score = round((prior_weight·band + Σw_sup) / (prior_weight + Σw_sup + Σw_ref), 4)

h1 = round((1.0·0.9 + 0.4921) / (1.0 + 0.4921 + 0), 4) = 0.933    # band 0.9 (high), 1 supporting fact
h2 = round((1.0·0.3 + 0)      / (1.0 + 0 + 0.4914), 4) = 0.2012   # band 0.3 (low),  1 refuting fact

Note the asymmetry doing its job: supporting evidence pulls the score toward 1, refuting evidence sits only in the denominator and drags it toward 0 — h2 ends below its own band. And a hypothesis with no resolvable evidence scores exactly its band: at seq 3 both rivals scored 0.6 and 0.3, nothing more.

The promotion gate — INV-8, the engine’s decision

The investigate exit is the strictest gate in the playbook: require_confidence_gate + require_refutation. After the fold (fold runs before the gate) the controller checks store.promotion_ok(tunables):

promotion_ok — executed at seq 4
earned ≥ gate0.933 ≥ confidence_gate 0.8 ✓

∧
delta floormargin ≥ delta 0.15 ✓ (sole survivor)

ADVANCE honouredroute: investigate → remediate

Plus require_refutation: a REFUTED rival exists (h2) — genuine differential work, not one theory re-supported in a loop.

What breaks without it. Rivals must be refuted, not merely out-scored: an unrefuted rival at 0.75 under a 0.8 gate still blocks — at any band. And an LLM-set CONFIRMED status grants no bypass; the gate reads only the engine-earned score. On failure the verdict downgrades to REPEAT carrying the exact reason — "confidence gate not met (lead below gate, margin < delta, or an unrefuted rival is still alive)" — which is fed to the next plan as gate_feedback, and the repeat cap (max_retries: 2) keeps a stalling planner from looping the phase forever. The twin even ships the other path: build(refuted_variant=True) has investigate refute h1 (“the NPE predates v4.12.0”), verdict backtrack, and the run re-enters hypothesize — proving the reasoning path, not just the answer.

ACT ⚑ — the write stops for a human

In the interactive session, remediate is where the machine stops being autonomous. The wrapped planner peeks at the plan: the phase is writes_allowed and the plan carries a call whose per-intent effect is WRITE (layer.effect_for) — so the session opens a gate before the write exists and raises _GateSuspend out of plan(). Nothing is claimed, folded or journaled for the phase yet; the question itself is durable:

                "root_candidate": "code_commit:abc123"},

The operator sees action + theory + evidence together, and answers one of three ways — each a durable gate_decision journal entry (source: human, actor recorded):

approve

The cached plan resumes intact; the layer serves the write under allow_write=spec.writes_allowed. The planner is never re-called — no second LLM round-trip, and a scripted cursor stays consistent.

refine

Operator params merge into the write call ({**call.params, **decision.params}); narrative gains " [params refined by operator]". Same plan, adjusted blast radius.

deny

Write calls are stripped; an UpdateHypothesis(basis="operator DENIED the proposed remediation: …") is appended so the denial lands on the store and the next plan re-plans — a genuinely divergent journal.

Verify, close — and the proof

Verify is a fresh read of the world, not a victory lap: the live transport is phase-scoped (ScenarioSource fixtures key provider → {phase|"*": blob}, and the drive loop sets source.phase before each step), so prometheus’s verify blob returns the post-rollback state — red_errors 0.01, degraded=false. Only then does the leader go confirmed. Close routes done; close_outcome = RESOLVED because the terminal phase ran and store.confirmed() is non-None (a run that ends without a confirmed theory closes MITIGATED — R-P4 keeps “fixed” and “made quieter” distinct).

What the run left behind

Graph: 12 nodes · 13 edges · 44 assertions (facts, events and node props are read views over the one assertion store). Hypothesis store: h1 confirmed @ 0.933 earned, h2 refuted @ 0.2012 — refuted records are kept, ranked; they are evidence. Journal: 10 entries. Postmortem (runtime/postmortem.py): root_cause = h1’s statement + root_candidate code_commit:abc123 + the earned score; ruled_out = h2 with its refutation basis — folded from journal + store, never authored separately.

The replay proof (R-J1)

Every e2e run asserts assert_replay_equivalent: rebuild(journal) must reproduce the graph byte-for-byte (g2.to_dict() == graph.to_dict()) and the hypothesis store record-for-record. The projections are caches; the journal is the truth. Eleven golden bundles (tests/e2e/golden/*.json) pin the whole output byte-stable across refactors.

05Mid-run unknown — the airlock

Real incidents do not stay inside the catalog. Suppose, mid-INC-4821, a discovery pass returns a CI ServiceNow can name but not classify — a legacy ESB bridge — plus a vendor metric the dictionary has never seen. The closed vocabulary must not silently accept garbage; the old failure was worse — rejected observations were erased as if they never existed. The airlock is the third way: everything lands, nothing lands at full trust, everything is journaled. Why three lanes and a penalty, not a dumpster or a wall. Silent acceptance opens the closed vocabulary to vendor drift (garbage in). Silent rejection erases discovery — the audit finding that motivated this design was exactly that rejected observations left no trace. So: unknown names land under a deterministic quarantine spelling (x.<source>.<native>) so repetition itself becomes the promotion signal; off-shape values land provisional plus a journaled rejection notice; an unclassified CI participates in structure and can even be blamed (caused_by), but its evidence is multiplied by discovery_penalty 0.75 — provisional knowledge is admitted, never at full weight. Every rejection rides PhaseResult.rejections into the journal and into the next PlanContext (R-K2) — the planner is told exactly what dropped and why, instead of re-emitting into silence.

06The gallery — eleven incident shapes, one engine

Same engine, same playbook, same grammar — eleven packaged incidents (runtime/scenarios.py _CATALOG → tests/e2e/scenario_*.py), each chosen to force a different discriminator: the fact pattern that separates the true root from its most tempting rival. Each card names the rival that gets refuted and the action the ACT gate proposes.

INC-4821 · Application code

payments-api 5xx after v4.12.0 deploy. Discriminator: pods Ready but throwing; p50 sane, error tail huge. Rival refuted: the healthy database. Root: code_commit:abc123 via error signature → change → commit.

Gate proposes: roll back to v4.11.3 (revert abc123).

INC-7731 · Deployment

checkout-api CrashLoopBackOff after rev43. The deploy dropped ConfigMap key DB_HOST; the pod never reaches Ready — a downstream-dependency rival would flap intermittently instead. Blame → PR #482 → commit 9f2a1e0.

Gate proposes: roll the Deployment back rev43 → rev42.

INC-9001 · Network

checkout-svc → pricing-svc timeouts after an MTU change. Retransmits spike on the segment (SEG-EDGE-12) while the callee’s health check comes back clean — a transport fault, so the segment’s change is the root, not either endpoint.

Gate proposes: revert the MTU/uplink change (CHG-77).

INC-7734 · Database

orders-api p99 after CHG-9 dropped an index. The slow exit call is JDBC-classified while the service’s own p50 stays flat; the git diff carries the literal DROP INDEX line. Roots at the CHANGE_EVENT, not the pool it saturates.

Gate proposes: re-create the index on orders.order_items.

INC-7702 · Firewall / Security

fraud-scoring egress blocked after CHG-3311. The link is healthy (packet_loss 0.0, no retransmits) yet traffic is cleanly DENIED on FW-EGR-118 — so the root is the policy change, never the healthy segment the denied traffic crosses.

Gate proposes: revert CHG-3311 (restore the prior ACL).

INC-9100 · No-change / Saturation

checkout-api pool saturation with an EMPTY change log. The change-first method must fall back honestly: the twin emits no_evidence("find_recent_changes", …), which materializes the engine fact no_evidence:find_recent_changes — an honest null that satisfies gates (R-P2). Root: organic 3.4× traffic vs a fixed pool limit.

Gate proposes: scale the pool + add a read replica.

INC-8801 · Messaging

…and the pattern

Every scenario is a differential: the obvious suspect vs the actual root, separated by one discriminating fact pattern the evidence turn must find. That is the doctrine (“confirmation is Popperian, not vibes”) made executable — and each run ends with assert_replay_equivalent and a byte-stable golden bundle.

One nuance worth knowing: in the batch twins a security-sensitive fix (e.g. the firewall revert) is only ever proposed as a hypothesis update — the packaged batch scripts execute no writes. The interactive registry injects the matching apply_remediation WRITE call (_with_gate), so in a session every one of the eleven stops at the Approve / Refine / Deny card. The write path exists only where a human is present to answer for it.

07UI actions & future ideas — ideation, not built

Everything above this section is executed, shipping behavior. Everything below is parked ideas — examples of where the substrate could go. None of it exists in the tree; where an idea leans on real machinery, the hook is named honestly, including whether that hook is wired today.

The line, drawn precisely — built today: the Approve / Refine / Deny gate card (gate_opened payload: actions + hypothesis + evidence; decisions durable); the typed SSE event stream (phase_started · reasoning · capability_call — whose outcome field is the honesty axis — graph_delta · hypotheses_delta · gate_opened · gate_decision · session_state); operator steer messages (add_message → journaled, fed into the next plan); durable persistence after every step and read-only reopen after a restart; the postmortem folded from the journal.
