The three projections — Graph · Journal · Hypothesis — the single fold that writes them, the one sequence space that orders them, replay as the proof of truth, and the file-backed data layer that makes an investigation durable. 01The data plane at a glance

Everything a run learns flows through one seam into three stores. The Journal (journal/journal.py) is the append-only, typed event log — the source of truth. The Graph (graph/graph.py) and the Hypothesis store (hypothesis/store.py) are projections: rebuildable caches of the journal, written only by graph/fold.py apply_delta(). That asymmetry is the whole design: if the projections burned down, rebuild(journal) reproduces them bit-for-bit (R-J1) — so nothing anywhere is allowed to depend on state the journal doesn’t carry.

Plain version. The journal is the save file; the graph and the hypothesis store are the loaded game. One function writes the loaded game, every write it makes is recorded in the save file first-class, and reloading the save file always produces the same game — byte for byte. Everything else on this page is the discipline that keeps that sentence true.

02Graph — the typed, bi-temporal projection

The graph is the model of the system under investigation: typed nodes and edges from the closed registry (43 node types · 36 edge types — see 02 · Data Model), carrying assertions in the Entity-Assertion model. Since the P6 store-flip there is exactly one assertion collection; facts, events and node props are read views over it.

The view discriminator is two predicates, not two stores: the facts property returns every assertion with species is not Species.EVENT and channel is not Channel.DECLARED (observed knowledge), events returns species is Species.EVENT (occurrences), and a node’s props are its channel = DECLARED assertions flattened to {name: value} — Node.props is a materialized read view rebuilt on every upsert, never dual-authored. Because channel_for_source() never returns DECLARED, a converted fact can never leak into the props view. to_dict() persists only the assertions list — “no dual write to drift” — and the shim pairs are exact inverses, which is why the goldens stayed byte-identical across the store-flip.

Bi-temporal, and never mutated

Two time axes

Valid time: valid_from / valid_to — the window a state was true (open window = still true). Observation time: observed_at — when a source reported it (occurred_at for events). The species fix the shape: identity carries no time, readings carry stat + window, events carry occurred_at (domain/assertion.py _time_shape).

Supersede, don’t overwrite (R-G5)

A newer value closes the prior open fact: add_fact sets the old assertion’s valid_to and state SUPERSEDED. A back-dated correction is clamped — closed_at = max(new.valid_from, old.valid_from), a zero-length window — never rejected, because add_fact is the shared live-fold and replay seam with no rejection channel: raising here could brick crash-resume replay of a journal already written.

Retract, don’t delete (R-J3)

retract_fact(fact_id) flips state to RETRACTED — a tombstone, the record survives. The Fact model carries no invalidated_by; the cause of a fact’s retraction lives in the journaled Retraction record. retract_event / retract_edge do carry invalidated_by (and the edge closes its window).

Point-in-time (INV-5)

facts_valid_at(ts) answers “what was true at ts” with a half-open window test valid_from ≤ ts < valid_to. SUPERSEDED facts are included — a superseded fact is still the truth for every instant inside its now-closed window; only RETRACTED is excluded. Excluding superseded facts broke as-of-incident-start reconstruction the moment anything was superseded (2026-07-22 review, finding 1).

The governed spine, and identity that never re-keys

structural_distances(anchor) is the graph’s one belief-facing traversal: an undirected, level-order BFS over ACTIVE edges whose origin is not INFERRED, never entering a HYPOTHESIS node. Topological specificity (§04) must come from observed structure — a hypothesis’s own CAUSED_BY claims can never shorten the distance of its own evidence. Level order makes hop counts independent of iteration order, so replay reproduces them exactly.

Identity is write-once. When the reducer graduates an id (a provisional twin merges, a generic_ci retypes, an alias resolves), the fold applies remap_id(old, new): the old id enters the chain-compressed id_remaps table (resolvable forever), DECLARED prop assertions re-key, one pass rewrites subject_ref — fact and event ids never move, so provenance ordering, supersession chains and hypothesis fact-id references stay byte-stable — edges are re-minted (their ids embed endpoints) with identical-edge collisions collapsing first-writer-wins, and the alias index rebinds. History re-homes; nothing orphans.

03Journal — the typed, append-only source of truth

One class, one file: journal/journal.py, SCHEMA_VERSION = 2. Everything that happens in a run — phases, tool calls, gate questions, human answers, operator messages, terminal outcomes — is one typed event log. Phase entries carry the full PhaseResult as their delta; that is what makes replay possible and the journal the truth (R-J1).

# journal/journal.py — the wire/record shape
SCHEMA_VERSION = 2      # typed entry kinds + append-at-event one-seq (v1 read-only supported)

class JournalEntry(BaseModel):
    model_config = ConfigDict(extra="ignore")   # tolerant-additive: unknown FIELDS load;
                                                # an unknown KIND still fails the Literal
    seq: int = 0                      # 0 = unassigned — Journal.append stamps it
    ts: datetime
    kind: Literal["phase", "step", "invocation", "gate_opened", "gate_decision",

    reasoning: str | None = None      # the narrative / thought

One seq space, append-at-event

def reserve_seq(self) -> int:            # the engine claims its phase seq HERE — after
    with self._lock:                     # plan() returned, past the suspension point,
        self._seq += 1                   # because materialized records stamp created_by
        return self._seq                 # before the phase entry can be appended

def append(self, entry: JournalEntry) -> JournalEntry:
    with self._lock:
        if entry.seq <= 0:               # append-at-event: unassigned entries are numbered
            self._seq += 1               # here, under the ONE lock (the background-drive
            entry.seq = self._seq        # duplicate-seq race closes)
        else:                            # a claimed seq — a phase, or its shared-seq
            self._seq = max(self._seq, entry.seq)   # invocation annotations — is accepted
        self.entries.append(entry)                  # as-is and advances the watermark
    return entry

  • Header validated on load. from_ndjson refuses a non-int or future schema_version loudly — “misreading a future journal would corrupt the source of truth.” Version 1 loads read-only; a headerless file is accepted.
  • Tail-only tolerance. An unparseable line is skipped only when it is the trailing line (a crash mid-append leaves at most one torn tail — appends are fsync’d); mid-file corruption raises — silently dropping an interior entry would corrupt the truth.
  • Watermark restore. After load, _seq = max(surviving seqs): a truncated tail entry’s seq is re-issued, a clean roundtrip resumes after the last entry, and v1 reserve-era gaps are preserved, never compacted.
  • Who reads what. rebuild() consumes phase_entries() only. The bundle’s "journal" view (api/bundle.py) is seq-sorted and filtered to phase entries plus step · gate_decision · message — invocation, gate_opened and lifecycle are record kinds excluded from that view so batch goldens stay byte-stable. The postmortem narrative is phase_entries()’ seq + phase + reasoning.
  • Code over design. kind still defaults to "phase" (the design said “kind never defaults”); refs and the narrative duplication are still stored (the design wanted them recomputed on read); rejection / repair have no writers. The code is the truth; these pages follow it.

04Hypothesis — earned belief, ranked once

The hypothesis store is a projection of the PhaseResult.hypotheses_updated stream: competing causal explanations, each carrying its own refuting evidence (anti-confirmation-bias), ranked in exactly one order that the planner and the UI both consume. Belief is earned: the number the engine ranks and promotes on is computed from graph evidence — the model’s band survives only as the prior and as the Confidence.basis text.

# hypothesis/store.py — the ranking key and the terminal wall
_STATUS_RANK = {CONFIRMED: 5, SUPPORTED: 4, INVESTIGATING: 3, PROPOSED: 2, REFUTED: 0, SUPERSEDED: -1}
_TERMINAL   = frozenset({REFUTED, CONFIRMED, SUPERSEDED})   # a reached verdict is indestructible

def _key(self, h):  return (_STATUS_RANK.get(h.status, 1), self.score(h))
def ranked(self):   return sorted(self.hypotheses.values(), key=self._key, reverse=True)
def score(self, h): # unbound → the raw LLM band; bound → belief.weighted_score(h, graph, tunables)

apply() — CREATE never destroys

First CREATE of a hid inserts (stamping created_by=seq). Re-CREATE of a live hid is a merge-update: statement/confidence refresh, evidence lists grow by sorted set-union, the chain appends, status and created_by are preserved. Re-CREATE of a terminal hid (REFUTED / CONFIRMED / SUPERSEDED) is a silent no-op — a live planner re-proposing the same id in a repeat loop can never silently resurrect a refuted rival (Track-4 #1).

One ranked order, two consumers

ranked() — status rank, then earned score, insertion-stable — feeds exactly two places: PlanContext.hypotheses (the planner sees confidence: store.score(h), the engine-earned number, not its own band) and export_bundle’s "hypotheses" array. Refuted hypotheses stay ranked-but-kept — they are evidence; the store has no deletion path at all.

# hypothesis/store.py — promotion, verbatim shape (INV-8)
def promotion_ok(self, tunables: Tunables) -> bool:
    alive = self.alive()                    # ranked, minus REFUTED + SUPERSEDED
    if not alive:
        return False
    lead = alive[0]
    if self.score(lead) < tunables.confidence_gate:   # the EARNED score, never the band
        return False
    rivals = alive[1:]                      # ALL alive unrefuted rivals block — no band filter
    if rivals:
        return False
    runner = self.score(alive[1]) if len(alive) > 1 else 0.0
    return (self.score(lead) - runner) >= tunables.delta

Read the code truth precisely: because any alive rival already returns False, the runner-up expression only ever executes when the leader stands alone — runner is always 0.0, so delta acts as a second absolute floor on a sole leader, not a live margin race (the docstring’s “beats the runner-up” phrasing predates the strengthened rival rule). This is Popperian on purpose: an unrefuted 0.75 rival under a 0.8 gate still blocks — rivals must be refuted, not out-scored. Enforcement lives in runtime/controller.py check_gate: only a proposed ADVANCE is gated, require_confidence_gate ⇒ store.promotion_ok(tunables), and an LLM-set CONFIRMED status grants no bypass. Both directions are pinned by tests: a 0.9-band leader dragged under the gate by refuting evidence stays blocked; a 0.6-band leader with strong support earns its way over.

Evidence edges are derived, only. The canonical evidence record is Hypothesis.supporting_facts / refuting_facts — fact-id lists. The graph’s SUPPORTS/REFUTES edges are a projection recomputed inside the fold (_project_evidence_edges): stale edges whose backing fact left the list are tombstoned first (sorted edge-id order, invalidated_by=h.id), then backed edges are re-asserted idempotently — a previously retracted edge revives if its fact returns. The reducer independently rejects planner-emitted SUPPORTS/REFUTES (EdgeSpec.derived=True), so the graph view can never disagree with the store.

05The fold — apply_delta, the one writer

One function owns every mutation of Graph + Hypothesis. Its in-delta order is not style — it is the replay contract: the same journal must always produce the same projections.

# graph/fold.py — THE single graph+hypothesis-store mutation seam
def apply_delta(result: PhaseResult, seq: int, graph: Graph, store: HypothesisStore) -> None:
    _apply_to_graph(result, graph)              # nodes → facts → events → edges →
                                                #   retractions → remaps LAST
    store.apply(result.hypotheses_updated, seq)
    touched: list[str] = []                     # dedup in delta order — never a set:
    for d in result.hypotheses_updated:         # iteration order must be deterministic
        hid = d.hypothesis.id if d.hypothesis else d.hypothesis_id
        if hid and hid not in touched:
            touched.append(hid)
    for hid in touched:
        h = store.hypotheses.get(hid)
        if h is not None:
            _project_evidence_edges(h, seq, graph)   # SUPPORTS/REFUTES, fully reconciled

def fold(result: PhaseResult, seq: int, graph: Graph, store: HypothesisStore,
         journal: Journal) -> None:
    apply_delta(result, seq, graph, store)
    journal.append_phase(seq, result)

The monopoly is verifiable, not aspirational: no caller of upsert_node / add_fact / add_event / add_edge / retract_* / remap_id exists outside graph/graph.py + graph/fold.py anywhere in src/iw_engine (grep-verified, 2026-07-22). HypothesisStore.apply has exactly two callers: fold.apply_delta and one sanctioned side door — InvestigationStore.load_result’s cache-hit branch, which replays hypothesis deltas directly while the graph is served from the watermark-verified cache (§08); the graph is untouched on that branch.

Why fold() exists but the engine doesn’t call it: journaling is kept separate from applying so an interactive write-gate can hold a computed-but-unapplied delta pending human approval. The production engine inlines the pair around the gate — apply_delta(…), then check_gate + the repeat cap, then swaps only the verdict via model_copy, then journal.append_phase(seq, result) — so the journaled verdict is the gated one. Replay equivalence holds because the verdict is a controller field that folds into no projection.

If the gate suspends (interactive)

On a writes_allowed phase whose plan carries a WRITE-effect call (per-intent: layer.effect_for(adapter, intent)), _GatePlanner stashes the plan, journals gate_opened — gate_id, proposed actions, the serving hypothesis and its evidence fact ids: what the human was asked, durably — and raises _GateSuspend. The step unwinds with nothing: no seq, no ops, no delta, no phase entry.

When the human answers

answer_gate(approve | refine | deny) journals gate_decision (source=human, actor, decision), then re-drives. The resume step gets the cached decided plan — the model is never re-called: APPROVE serves the plan intact; REFINE merges operator params into the write calls; DENY strips the writes and appends an UpdateHypothesis whose basis records the denial — a genuinely divergent journal (test_session_deny_diverges).

07Replay — rebuild() and what counts as state

Replay is not a debugging trick; it is the load path, the crash-recovery path and the proof that the journal is the truth. One rule decides everything: state is kind == "phase" entries with a delta; every other kind is record.

# graph/fold.py — replay, verbatim
def rebuild(journal: Journal, *, tunables=None) -> tuple[Graph, HypothesisStore]:
    """Replay the journal → (graph, hypothesis store). Must equal the live projections."""
    graph, store = Graph(), HypothesisStore()
    for entry in journal.phase_entries():        # kind == "phase" and delta is not None
        apply_delta(entry.delta, entry.seq, graph, store)
    if tunables is not None:
        store.bind_scoring(graph, tunables)      # a replayed store RANKS as the live one did —
    return graph, store                          # scoring is a pure query-time projection

Equivalence is pinned end-to-end: test_fold_and_replay_equivalence (g2.to_dict() == g.to_dict() + store equality), the session approve and deny branches each replay to their own journal’s graph, test_rejections (a rejection mutates no projection — rebuild still equals live), and e2e test_persistence (the on-disk journal replays to the live graph + ranked hypotheses).

The determinism ledger — what keeps replay bit-for-bit

  • Everything mutating rides the delta: retractions, identity remaps, reducer-derived transition events, rejections-as-record — nothing depends on engine memory.
  • Fixed fold order (§05) — violate it and the same journal produces a different graph.
  • Rejections are record, not state: excluded from is_empty_delta() and every gate floor; journaled with the phase; fed to the next plan. Counted as state, they would inflate progress and move gates on replay-inert data.
  • List-dedup, never sets for touched hypothesis ids; evidence lists stored sorted; stale-edge retraction in sorted-id order; level-order BFS everywhere hop counts matter.
  • No wall clocks in state: DECLARED prop declarations carry no observed_at; invocation timing (started_at / duration_ms) lives only on the in-memory Invocation, never in the journal or bundle — goldens stay byte-deterministic.
  • Scores round to 4 decimals and belief never crashes on untrusted time (INV-7: a naive/aware datetime mix degrades a factor to neutral, never kills the fold).

What breaks if the rule breaks: a record kind that mutated projections would double-apply on replay; a fold that read engine memory would make crash-resume diverge from the live run; either way R-J1 — “the journal is the source of truth” — becomes a lie, and every consumer built on it (reopen, bundle, postmortem, goldens) inherits the lie.

08The data layer — InvestigationStore

One module owns every byte of investigation disk I/O: runtime/store.py. By owner directive it is a simple, concrete file store — no StorageBackend Protocol, no second implementation, files not a DB. The engine is pluggable exactly where pluggability pays — planner, tools, domain — and storage is deliberately not one of those seams. It is contained instead: save is persist / reset, load is load_result, reopen is load_bundle, list is list_disk / has — so a later DB swap is a one-module change.

class InvestigationStore:

persist() runs after every fold/step, at gate open, on every operator message and at every terminal (the session’s _persist() cadence) — so disk is never more than one in-flight step behind. It appends journal.entries[prior:] with flush + os.fsync, computes head = max(e.seq), writes graph.json stamped with that watermark, and atomically rewrites meta.json. Why tunables ride in meta: a disk reopen re-binds the rebuilt store’s belief arithmetic with exactly the knobs the run scored under, so the reopened bundle’s earned confidences equal the live ones field-for-field. The two crash models unify (R-J4): the journal is append-crash-tolerant; the caches are atomic-rewrite, disagreement-checked on load.

Cache hit

Serve the cached Graph; the hypothesis store is still replayed from the journal’s phase deltas (store.apply(entry.delta.hypotheses_updated, entry.seq) — graph-independent), then bind_scoring(graph, tunables-from-meta). The cache is an optimization, never an alternate truth.

Miss / stale / corrupt

graph, store = rebuild(journal, tunables) — the §07 path — and the cache is rewritten (self-heal). A stale or gutted graph.json can never poison a reopen; for any deeper doubt, delete it and it is reprojected.

Restart-resume is a read, by design

After a backend restart, SessionManager.reopen(id) lazy-hydrates: memory hit → the live snapshot; miss → store.load_bundle(id) — export_bundle of the disk-rebuilt run plus {read_only: true}. Only GET /sessions/{id} uses it; advance / gate / messages / events / stream go through memory-only get() and 404 — reopened runs are served durably, not re-driven (continue-driving is out of scope). On reopen, RunResult.rejections / invocations rehydrate empty as objects — the bundle derives its rejections from the journaled deltas, so the list survives anyway.

No zombies — every stop is diagnosable

Every terminal path writes a lifecycle entry before the final persist: closed (with the outcome), max_steps_exhausted (the P6 zombie fix — exhaustion used to leave RUNNING forever with no trace), or error (a dead background drive). The session closes, the SSE stream terminates, and the run stays reopenable read-only from disk. A fresh session of the same id calls reset() first — a journal file never interleaves two runs.

09The reasoning loop — projections drive the planner (P7)
