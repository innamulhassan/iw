How Incidents Are Investigated

This document explains the method — the way the engine takes an incident from first alert to a confirmed, evidence-backed root cause, and the kinds of failures it can reason about across the whole application stack.

01The method

Every incident is worked the same disciplined way — the way a senior engineer would, but made explicit and repeatable. The investigation starts from a correlated incident and moves through seven phases, gathering evidence and narrowing to a cause.

Where it starts — the correlated incident. In production a single failure sets off a storm of alerts across many services. The first tool the investigation taps is the event-aggregation / AIOps layer (e.g. BigPanda, Moogsoft): it has already collapsed that storm into one incident, computed the affected-service blast radius, and clustered the co-firing / related incidents. So the reasoner begins from a de-noised, enriched picture — and picks up the related incidents automatically as a hypothesis prior ("these N services reported the same in the same window") rather than staring at a wall of raw alerts.

1 · Frame the symptom

Pull the correlated incident from the aggregation layer — the affected service(s), the alert cluster, the onset time, the related incidents — and pin the immediate topology around it.

2 · Triage

Is it still bleeding? Decide whether to mitigate immediately or investigate first, and narrow the set of suspects.

3 · Hypothesize

Form competing theories — change-first (what recently changed?), and fallbacks (saturation, dependency failure). Keep the strongest as the lead and at least one rival.

4 · Investigate

Gather the evidence that would confirm the lead and the evidence that would refute the rival. This is where the root cause is proven.

5 · Remediate ⚑

Propose the safest reversible fix. Because it changes a live system, the run pauses here for a person to approve, refine, or decline it.

6–7 · Verify & Close

Confirm the symptom cleared and the causal chain still holds, then produce a postmortem — the confirmed chain and timeline, assembled from the log.

02The core discipline: differential diagnosis

Finding a plausible cause is easy and often wrong. The engine insists on telling apart look-alike causes by their discriminators — the signal that separates the real cause from the obvious impostor.

The idea. Two very different faults can produce almost the same symptom. The investigation is not allowed to conclude until it has found the specific evidence that rules one out. A theory that only has evidence for it — and none against its rival — is treated as unproven.

Concretely, the engine can require, before an investigation advances, that a rival theory was ruled out or the leading theory was actively challenged — genuine differential diagnosis, not confirmation of the first guess.

03What it can investigate — the whole stack

The graph models a modern application end to end, so incidents can be reasoned about at every layer. Each layer has a characteristic failure and a characteristic way of telling it apart from its neighbours.

🟦 Application code

A code change introduces a fault — an exception, a bad code path — that surfaces as errors.

Told apart by: the fault traces to a specific change and file, while the service's dependencies stay healthy.

🟩 Deployment / configuration

A rollout ships a broken configuration and the workload fails to start or stay up.

Told apart by: the workload never becomes healthy after the change — a dependency problem would look intermittent, not a hard failure at start-up.

🟪 Database

A schema or query change degrades the datastore, so calls to it slow down or queue.

Told apart by: the slowdown lives entirely in the call to the database while the service's own processing stays fast.

🟫 Messaging / streaming

A consumer or producer change makes a queue back up, breaching a downstream deadline.

Told apart by: the backlog is one-sided — the broker and the other side are healthy, so the deficit is in processing rate, not the transport.

🟥 Network

A change on the network path introduces loss or latency between two services.

Told apart by: the loss is on the path itself — both endpoints are healthy, so it is neither the caller's code nor the callee's load.

🟧 Firewall / security policy

A policy tightening blocks traffic that used to be allowed.

Told apart by: the failures are clean policy denials — the link is healthy and packets flow; they are simply being rejected — not packet loss.

🟨 Infrastructure / platform

A host or platform condition — resource pressure, a noisy neighbour — starves a workload.

Told apart by: the pressure is at the host level while the affected workload's own usage is normal — a co-tenancy fault, not the application itself.

⬜ Saturation with no change

Organic demand outgrows a limit — a pool, a thread budget — with nothing changed to blame.

Told apart by: the change log is empty and the whole latency distribution shifts up under load. With nothing to revert, the correct outcome is to mitigate (scale), not "resolve."

Same engine, every layer. None of these are special cases in code. They are all the same seven-phase method over the same shared graph vocabulary and the same tools — the only thing that differs is which evidence the reasoner gathers and which discriminator settles it.

04How the incident domain is defined

"Incident investigation" is not hard-coded — it is a curated domain: a lean configuration on top of a shared core. That is what lets the same engine serve other investigation domains without new code.

The playbook — per domain

A small file names the phases, each phase's goal, its allowed intents, and its gate (what evidence is required before it may advance). For incidents that is the seven-phase method; a different domain is simply a different playbook.

The shared core — all domains

The graph vocabulary (the entity and relationship types) and the capabilities (the tools, including event correlation) live in the core and are reused by every domain — curated once, not per incident.

So curating the incident domain is three declarative things:

  • Phases & gates — the disciplined method and the bar each phase must clear (the playbook).
  • Vocabulary — the types the reasoner may use to describe the system (the shared registry).
  • Tools — which capabilities are wired (event correlation, monitoring, topology, logs, …).

The reasoner is never free-form: it works only within the curated phases, classifies only into the curated vocabulary, and calls only the wired tools — which is exactly what keeps every investigation consistent and auditable across thousands of incidents.

05What an investigation produces

A confirmed root cause

The single explanation that the evidence supports and whose rival was ruled out — pinned to the specific entity that caused it.

An evidence chain

For that conclusion, the exact facts that back it and the facts that refuted the alternative — each traceable to the tool and time it came from.

A human-approved action

A concrete, reversible fix that a person reviewed and approved before it touched anything live — or a mitigation, when there is no cause to revert.

A postmortem

The confirmed causal chain and timeline, assembled automatically from the append-only log — reproducible, not hand-written after the fact.

06Where the human sits

The person is not a spectator. Throughout the investigation they can steer it — add context, redirect a line of inquiry, ask a question — and at the one moment that matters, they hold the decision.

  • Watch it reason. Each phase streams its thinking, the tools it calls, and what it found.
  • Steer any time. A message from the operator becomes input the reasoner weighs on its next step.
  • Own the change. Before any fix runs, the investigation pauses and presents the proposed action with the evidence behind it: approve it, adjust it, decline it, or say what to do instead.
