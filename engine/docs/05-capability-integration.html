01The boundary at a glance

The capability layer is the only place the engine touches the outside world, and it is built around one idea (capability/layer.py, module docstring): an Adapter is a pure pair — a deterministic normalize(raw) → Operation[] plus data descriptors — while the live fetch is the ONE side-effecting seam, behind Source.fetch(). Everything else in the engine is deterministic; this seam is where non-determinism is quarantined.

Plain version. The planner says “check the error logs for payments-api.” The layer looks up which tool owns that verb, checks whether the call is even allowed (writes need an approved human gate), fetches the vendor’s raw JSON through one transport seam, and hands it to that tool’s adapter — a pure function that folds vendor fields into typed AddAssertion/AddNode/AddEdge ops the reducer then validates like any other ops. Swap the transport and nothing else changes; that is why a fixture-driven test run and a live MCP run exercise identical adapter and engine code.

Where this sits in the pluggability story. The engine is pluggable in exactly three places — planner, tools, domain. This page is the tools seam: an Adapter is a pure fold, the transport is one injectable Source, and adding or replacing a tool touches only its adapter (plus a fixture, or two env vars for a live endpoint). Nothing else — not the engine, not the prompt, not the reducer — changes.

02Verbs, intents, and the identity backbone

There are three vocabularies in play, and being precise about them explains half the design. Abstract verbs — a playbook phase’s allowed_intents (“correlate_timeline”, “fetch_metrics” as a category) — are prompt-advisory: the engine consults them for exactly one thing (whether to compute the correlate_timeline hint); they are not enforced at resolve time. Concrete intents — bt_health, search_errors, ocp__restart — are what the planner actually emits and what CapabilityLayer.resolve() maps to an adapter. Vendor tool names are the third vocabulary: today McpSource sends the concrete intent verbatim as the MCP tool name — a registry-driven intent→vendor-name mapping is design work, not code (§11).

The planner learns the concrete vocabulary from a catalogue rendered from adapter metadata — nothing tool-specific is hardcoded in the engine or the prompt. domain/catalog.py render_tools() walks each adapter’s meta (a CapabilityMeta) and prints its summary, its intents, and — the load-bearing field — queries_by: the identifier that tool needs. Write intents render under a [WRITE — human-gated] block, and only when include_writes (so a live planner can propose a remediation); tool_intents() is the same walk used to reject off-catalog calls. Adding a capability makes the reasoner aware of it with zero prompt edits.

The catalogue tells the model exactly this, verbatim: “Resolve that id off the target CI … Do NOT reuse the display name for a tool that queries by a different id (AppD queries by app_id, git by repo, the platform by k8s_workload).” The backbone is expressed as data (CapabilityMeta.queries_by), not code — one string per adapter today; per-intent queries_by is a design row that has not landed (§11).

03The contracts — Adapter, the call, the audit record

Three small shapes carry the whole boundary. They are worth reading verbatim, because every governance and honesty property below hangs off one of their fields.

# capability/layer.py — the tool contract (verbatim shapes)
@runtime_checkable
class Adapter(Protocol):
    provider: str                 # "prometheus", "servicenow", ...
    intents: frozenset[str]       # the concrete verbs this tool owns
    effect: Effect                # READ | WRITE — the default across the intents set
    binding: Binding              # MCP | REST | A2A — which transport family serves it
    meta: CapabilityMeta          # the self-describing catalogue record

    def normalize(self, raw: dict) -> list[Operation]: ...

# optional, not in the Protocol — discovered via getattr(a, "effects", None):
#   effects: ClassVar[dict[str, Effect]]     # PER-INTENT effect override
#   e.g. ServiceNowAdapter: {"ingest_alert": Effect.WRITE}
#        OcpAdapter:        {"ocp__restart": Effect.WRITE}

@dataclass(frozen=True)
class CapabilityMeta:
    summary: str          # one line: what this capability is FOR
    queries_by: str       # the target identifier it needs: app_id | repo | service_name | ...
    returns: str = ""     # what it contributes to the graph (optional, for the catalogue)

class CapabilityCall(BaseModel):  # extra="forbid" — what the planner emits
    intent: str
    params: dict = Field(default_factory=dict)

And the output side — one Invocation per call, the audit record the engine journals, the session streams, and RunResult.invocations collects:

class Invocation(BaseModel):      # extra="forbid" — one audit record per capability call
    intent: str
    provider: str
    effect: Effect
    op_count: int
    blocked: bool = False
    reason: str | None = None
    outcome: str = "data"         # "data" | "empty" | "error" | "blocked"  (§05 — load-bearing)
    params: dict = Field(default_factory=dict)   # what went IN (the query, journaled in full)
    summary: str = ""             # one line of what came OUT
    kind: str = "tool"            # tool | workflow | llm | handoff (agent-trace span kind)
    started_at: str | None = None       # ISO wall-clock — EPHEMERAL, never journaled
    duration_ms: float | None = None    # ephemeral too — goldens stay byte-deterministic

Why a Protocol, not a base class

An adapter is data plus one pure function. There is nothing to inherit: no shared transport code (that lives behind Source), no shared constants (INV-9 — reliabilities come from the playbook’s Tunables, filled by the reducer). The Protocol is runtime_checkable so the layer can be handed anything with the right shape.

Why per-intent effects

Real tools mix reads and writes on one surface. The old workaround was a split placeholder class per write (OcpRestartAdapter); the effects override retires it — ocp__restart is WRITE on the very adapter serving pod_status as READ, and ingest_alert is WRITE because it creates a vendor-side record while its six sibling ServiceNow intents stay reads.

Why timing is ephemeral

started_at / duration_ms are stamped by the engine around serve() (model_copy) for the live trace UI, and never journaled, never exported in the bundle. Wall-clock in the record would diff every golden on every run.

Why params ride in full

The journaled invocation carries params verbatim — “hashing them is a live-privacy knob for a later phase” (runtime/engine.py). A deliberate code-over-design choice: audit fidelity first, redaction when live tenancy demands it.

04serve() is gate-first — allow / ask / deny

One method is the production path: serve(call, *, allow_write) → tuple[list[Operation], Invocation]. Its order is the invariant: resolve → gate → fetch → normalize. A blocked call returns before Source.fetch is ever reached, so a denied write cannot cause a vendor-side effect — this is the fix for the audited write-before-gate bug, and it is structural, not conventional.

serve() — the gate ladder, with every early exit and its exact reason string
resolve a = self._by_intent.get(call.intent) — miss ⇒ blocked Invocation(provider="?", outcome="blocked", reason="no capability for intent '<intent>'"). An unknown verb is provably refused, never guessed at.
↓
policy (opt-in) Only when a CapabilityRegistry is wired: spec_for(intent) → DENY ⇒ blocked ("capability pending review - deny" for a synthesised unknown, else "policy: deny for intent '…'"); ASK without allow_write ⇒ blocked ("policy: ask - human approval required for '…'"). ASK is released by the same allow_write boolean — there is no per-call token yet (§11).
↓
read/write gate effect_for(a, intent) — resolution order: registry-declared effect → the adapter’s effects dict → the adapter default. WRITE and not allow_write ⇒ blocked "write blocked - no approved gate" — the human-approval invariant, enforced per intent, never per adapter.
↓ only now may a side effect happen
fetch raw = self.source.fetch(a.binding, call.intent, call.params) if self.source else {} — any exception (vendor 4xx/5xx, timeout, parse blow-up) is caught ⇒ a recorded error Invocation. A live transport failure degrades the read; it never raises through serve().
↓
normalize _fold(): a.normalize(raw) inside try/except — a raising adapter is an error Invocation (summary="tool error - no evidentiary weight"), never a crash. Success: outcome = "data" if ops else "empty".

invoke(intent, raw, *, allow_write) is the same governance over a pre-fetched raw — the seam the unit tests feed directly; no production caller. One known gap, stated in §11: intent collision when two adapters claim one verb is silent last-wins over the adapters list (layer.py ctor) — the design’s ranked multi-source is unbuilt.

The human gate above the boolean

The layer’s gate is a boolean — allow_write comes from the playbook: spec.writes_allowed, true only for the write-armed ACT stage (in the shipped seven-id incident.yaml that is the remediate phase — today’s declaration of the frame → investigate-loop → gated ACT → verify → close shape, see doc 01 §05). Who flips the boolean? In an interactive session, a human, and only after seeing the evidence:

The write gate, end to end (runtime/session.py — composition over the engine, no engine edits)
plancarries a WRITE-effect call
→
_GatePlanner intercepts_is_write_call: effect_for per intent
→
gate_openedjournaled: action + hypothesis + evidence
→
SUSPENDED ⚑_GateSuspend — nothing claimed, no seq burned
→
approve / refine / denygate_decision journaled, source=human
→
resumecached plan — serve(allow_write=True)

APPROVE serves the cached plan intact; REFINE merges operator params into the write calls; DENY strips the writes and feeds an UpdateHypothesis(basis="operator DENIED the proposed remediation: …") back into the ops — a genuinely divergent journal. The question (gate_opened) and the answer (gate_decision) are both durable journal kinds: the journal proves consent and execution — the approved write’s invocation entry shares the executed phase’s seq.

Two honest caveats, design-acknowledged. (1) One approve releases every write call in that plan — per-call approval-token binding (a gate_id tied to one write Invocation) is deferred substrate work (registry.py SCOPE NOTE). (2) In batch mode (Engine.run() with a write-capable layer), a writes_allowed phase passes allow_write=True with no human in the loop — the interactive gate lives in the session layer. Both are on the convergence list, not surprises.

05Outcome honesty — data · empty · error · blocked

Every call ends in exactly one of four boundary outcomes, and the distinction is the layer’s most load-bearing honesty property. The field’s own comment (layer.py Invocation.outcome) says it plainly: a downstream reader keys off this, “never off op_count == 0 alone (which conflates clean-empty with error — the silent-empty poison this field exists to kill).”

data

op_count > 0 — the tool returned facts that folded into the graph. summary is a one-line human digest of the ops.

empty (“clean-empty”)

An honest no-data read: the provider answered, nothing to fold. This may feed the refutation path — it is the only outcome that can back an honest NoEvidence null-result fact (R-P2). The code literal is "empty"; the design prose calls it clean-empty — same thing.

error

A transport or normalize failure. No evidentiary weight — it must never be read as a refuting result. reason="{ExcType}: {msg}", summary="tool error - no evidentiary weight". Error ≠ empty is the honesty line.

blocked

The gate refused the call — unknown intent, policy deny/ask, or an ungated write. blocked=True plus the exact reason string from §04. Recorded, never silently swallowed.

What makes this real is that the outcome is consumed, end to end:

One outcome, three consumers — how honesty propagates
engine self._intent_outcomes[inv.intent] = inv.outcome — the last outcome per intent (a later clean call restores an intent’s evidentiary standing). Then no_weight = frozenset(i for i, o in … if o in ("error", "blocked")) → materialize(ops, seq, …, no_weight_intents=no_weight).
↓
reducer A NoEvidence op naming a no-weight intent is rejected: an errored or blocked call observed nothing — “only a clean-empty read can become null evidence.” This is the fabricated-negative-evidence killer: without it, a vendor outage becomes refuting evidence and the run converges wrong-but-confident.
↓
journal + UI Every call — data, empty, error, blocked alike — becomes a kind="invocation" journal entry sharing its phase’s seq (an annotation, not a numbered step; replay ignores it): action={capability, provider, params, effect}, observation={outcome, reason, blocked, op_count}, decision=outcome. The session streams the same record as a capability_call event with outcome explicit — the UI must never infer “clean” from zero ops.

What breaks without this. Pre-fix, an unguarded boundary had two failure modes the audits named: one vendor 500 killed the session (now: a caught error Invocation), and a shape mismatch read as “we checked and it was clean” (now: error ≠ empty, enforced where evidence is minted). Both fixes are downstream of one field.

One display-level staleness, so you don’t chase it: _summarize_ops() counts legacy op class names (AddFact, AddEvent, …) and does not count AddAssertion — which is what all ten adapters now emit. A fold of pure assertions can therefore summarize as "no new data" even though outcome="data" and op_count > 0. The honest outcome field is unaffected; only the cosmetic summary lags.

MCP envelope tolerance — the part of “live” that is already real

A real MCP endpoint answers in several shapes, and none of them may crash the boundary (INV-7). _mcp_result(reply) prefers result.structuredContent; falls back to the first parseable JSON content[].type=="text" block; raises RuntimeError only on a genuine JSON-RPC error or result.isError (which serve() catches into an error Invocation); and degrades a merely-unrecognised shape to {} — never an AttributeError. Underneath, _parse_sse_or_json(body) handles a plain JSON string and SSE / Streamable-HTTP frames (event: message + data: lines) — the last parseable data: payload wins, because a streaming server may emit progress frames before the result; [DONE] markers are skipped; an undecodable body degrades to {}. What is not built: the MCP initialize handshake, session ids, tools/list discovery, OAuth (§11).

# sources.py — the wire shape McpSource actually sends (intent verbatim as the tool name)
payload = {"jsonrpc": "2.0", "id": self._id, "method": "tools/call",
           "params": {"name": intent, "arguments": params or {}}}
headers = {"Content-Type": "application/json"}          # + Authorization: Bearer <token>
return _mcp_result(self._http(self.endpoint, payload, headers))

07Vendor JSON → the documented shape — mapping.py

Adapters expect a fixed raw shape — the de-facto schema documented by runtime/live_fixtures.py. Fixtures are that shape and bypass mapping; a real vendor envelope is translated here, so a live MCP/REST tool works without touching any adapter. Translation is response-side only (request-side builders — PromQL, SPL, sysparm, AQL — are unbuilt, §11).

  • MappingSource(inner, intent_provider) composes the translation over any Source (and passes .phase through for ScenarioSource) — built and unit-tested, wired nowhere in production: fixtures don’t need it, and no live transport is constructed in src/ yet.

08Governance as data — registry.py

The domain dictionary governs the vocabulary of the graph; this module applies the same closed-core + governed-onboarding grammar to the vocabulary of the tools. It is substrate: built, tested, opt-in — and not yet passed by any production wiring (registry=None everywhere today).

# capability/registry.py — one governance row per intent
class Policy(StrEnum):
    ALLOW = "allow"; ASK = "ask"; DENY = "deny"

@dataclass(frozen=True)
class CapabilitySpec:
    provider: str
    intent: str
    effect: Effect = Effect.READ       # PER-INTENT — where the split-adapter workaround retires
    policy: Policy = Policy.ALLOW
    pending_review: bool = False       # True only on synthesised rows for unknown intents

CapabilityRegistry.from_adapters(adapters, *, policy=Policy.ALLOW)
    # mirrors the registry off the wired adapters, honoring each adapter's
    # per-intent `effects` override (ocp__restart lands "write")
CapabilityRegistry.spec_for(intent)
    # unknown intent → CapabilitySpec(provider="?", policy=DENY, pending_review=True)
    # — NEVER an implicit allow

The property this buys

An unregistered tool call is provably refused at resolution, never silently executed. A partial hand-authored registry denies everything it doesn’t list; set_policy() is the explicit, auditable mutation (mark a write ask, quarantine a provider deny).

The scope note, verbatim intent

Enforcement is at the CapabilityLayer boundary, opt-in. Per-call approval-token binding (a gate_id linking one approval to one write Invocation) and playbook allowed_intents enforcement at resolve time are engine-side and deferred — this module is the substrate they will consume. Earned autonomy (an ask→allow ladder) is likewise deferred.

09The roster — one adapter per tool

Ten adapters, one per tool. ALL_ADAPTERS lists the nine readers (adapters/__init__.py — its docstring still says “the 8 tool adapters,” stale since BigPanda landed); default_adapters() instantiates them; the write-effect RemediationAdapter is wired only by the scenario registry and the live manager. The reference template, verbatim

PrometheusAdapter is the template the other nine follow — descriptors plus one pure fold. Note what an adapter doesn’t do: no HTTP, no retries, no constants (source_reliability=m.get("reliability") — None lets the reducer fill the playbook default, INV-9), no invented names (the vendor’s metric name survives on source_native_name; an off-dictionary name will land quarantined as x.prometheus.<native>, not erased):

# capability/adapters/prometheus.py — the whole read side of a tool integration
class PrometheusAdapter:
    provider = "prometheus"
    intents = frozenset({"active_alerts", "instant_query", "range_query", "fetch_metrics"})
    effect = Effect.READ
    binding = Binding.REST   # no first-party MCP server — trivial raw REST
    meta = CapabilityMeta(
        summary="Time-series metrics (RED/USE) and firing alerts",
        queries_by="service_name", returns="alerts + metric facts")

    def normalize(self, raw: dict) -> list[Operation]:
        ops: list[Operation] = []
        svc = raw.get("service")
        # ... AddNode(SERVICE) → svc_id = registry.node_id(...)
        for al in raw.get("alerts", []):
            # AddNode(ALERT) + AddAssertion(name="fired", species=EVENT,
            #                               occurred_at=al["at"], ...) + AddEdge(FIRED_ON)
            ...
        for m in raw.get("metrics", []):
            ops.append(AddAssertion(subject=subject, name=m["predicate"], value=m["value"],
                                    unit=m.get("unit"), species=Species.READING,
                                    stat=Stat.GAUGE, window=Window(at=m["at"]),
                                    valid_from=m["at"], observed_at=m["at"],
                                    source=Source.PROMETHEUS,
                                    source_reliability=m.get("reliability"),
                                    source_native_name=m["predicate"]))
        return ops
Adding or replacing a tool. Write one class like the above (descriptors + normalize), add it to the layer’s adapter list, and give it data — a fixture blob for hermetic runs, or IW_CAP_<PROVIDER>_URL/_TOKEN once the provider-routed live path is wired. The catalogue re-renders itself from meta (the reasoner learns the tool with zero prompt edits), tool_intents() starts accepting its verbs, and nothing else in the engine changes. Swapping vendors behind a capability — Moogsoft for BigPanda, say — is the same move: the intents and the fold shape stay, only the adapter (and, live, a translator in mapping.py) changes.

10The first tap — de-noising the storm

The event-correlation / AIOps adapter (BigPandaAdapter — BigPanda, Moogsoft, and kin) is the layer above raw monitoring, and the investigation taps it first. Monitoring tools fire many individual alerts; the aggregation platform deduplicates and correlates them into a single high-level incident. Instead of a wall of raw alerts, the engine gets: “these N alerts across these M services are ONE incident — and here are the co-firing, related incidents.” That correlated cluster is a strong hypothesis prior, folded in the same shapes the engine already reasons over.

11The cross-tool joins the adapters draw

Each adapter folds its own tool honestly; the investigation works because a few of those folds deliberately join tools. These are the seams worth knowing by name.

The identity backbone — servicenow

The CI’s per-tool identifiers (app_id · sys_id · repo · k8s_workload) are copied onto the Service node, so every later tool can be queried by its own id (§02) — and the reducer’s alias backbone resolves cross-tool arrivals onto one entity instead of twinning it.

Discovered topology — appd exit-calls

_fold_exit_calls is a branch switch on the trace boundary: type=="JDBC" → Database ({db_id, engine}) + DEPENDS_ON origin=discovered; type=="HTTP" → ExternalService (when target_external) else Service + DEPENDS_ON; anything else is an in-process hop — no edge. Telemetry-discovered topology, kept distinct from the CMDB-declared spine.

The terminal join — git blame

{sha, file, line, snippet} pins an ErrorSignature to a CodeCommit: a CAUSED_BY edge at ConfidenceLevel.HIGH with EvidenceRef(kind="blame", ref="file:line", label=snippet), plus a blame_line content DESCRIPTOR (live-only — hermetic fixtures carry no snippet). A CAUSED_BY from hyp:<id> to the commit is drawn only when the payload names a hypothesis id — causal linkage stays the planner’s job; the adapter just draws the edge to an already-proposed hypothesis.

Supply chain by digest — artifactory

The artifact’s digest is its identity; properties["git.revision"] draws BUILT_FROM to the CodeCommit (the Artifactory→Git join) and promoted.to/.at draws RELEASED_AS to the Release — commit → artifact → release becomes walkable structure, which is exactly what the deployment and code-regression scenarios traverse.

Closed vocabulary at the platform — ocp

k8s reasons map through fixed tables — _POD_REASON_EVENTS (OOMKilling/OOMKilled → OOMKilled, BackOff/CrashLoopBackOff → restarted, Killing → terminated, …), _DEPLOYMENT_REASON_EVENTS (ScalingReplicaSet → rollout_started, ProgressDeadlineExceeded → rollback, …), _LOG_SIGNATURES for pod_logs (a second, independent path to the same signal). The raw reason survives on source_native_name; an unmapped reason is skipped, never minted into the vocabulary.

The declared spine — cmdb

Structure only, treated as truth: every edge origin=Origin.DECLARED (R-G8). The raw relationship string (“Runs on::Hosts”) maps to the first registry-legal EdgeType candidate for the concrete type pair (_edge_type_for); no legal candidate ⇒ the relationship is dropped — the adapter’s own guard against the reducer’s rejection path. Uniquely threads source=Source.CMDB onto AddNode, so each declared prop is attributed to the tool that supplied it.

12Real vs aspirational — the honest ledger of this boundary

The design docs (part 4 of the redesign) describe more than the tree contains. Here is the split, verified against code — so nobody reads a roadmap as a feature list.

Real — shipped and tested

  • Gate-first serve(); guarded transport and normalize — one vendor 500 can no longer kill a session.
  • Four distinguishable outcomes, consumed end to end: engine no_weight_intents → the reducer rejects tainted NoEvidence.
  • Every call journaled (kind="invocation", sharing the phase seq); durable gate_opened / gate_decision entries.
  • Per-intent effect: effects overrides + effect_for; ingest_alert and ocp__restart are WRITE on mixed adapters; the split-adapter workaround class is retired.
  • Registry-as-data with allow/ask/deny and pending-review→deny — as substrate: built + tested, not wired into any production layer.
  • Provider-arity routing + env config (ProviderRoutedSource, IW_CAP_<P>_URL/_TOKEN, build_provider_transports) — substrate, unit tests only.
  • Tolerant MCP envelope + SSE / Streamable-HTTP parsing (_mcp_result, _parse_sse_or_json).
  • Response-side translators for prometheus / servicenow / splunk / git (4 of 10; the rest pass through by design).
  • ScenarioSource: provider-keyed, phase-scoped fixtures — the live-LLM product path.
  • Ten adapters emitting the Assertion atom natively, with source_native_name preserved and ids from registry.node_id.

Aspirational — design prose, no or partial code

  • MCP initialize handshake, session ids, tools/list discovery, OAuth; intent→vendor-tool-name mapping (the intent goes verbatim today).
  • Request-side builders — PromQL / SPL / sysparm / AQL templating, REST POST; RestSource is GET-only.
  • Per-call approval-token binding (gate_id → one write Invocation); today one approve releases every write in the plan, and batch mode self-approves on writes_allowed.
  • Playbook allowed_intents enforced at resolve time — today abstract, prompt-advisory (the engine consults it only for the correlate_timeline hint).
  • Ranked multi-source per intent — today an intent collision is silent last-wins over the adapters list (the page’s one anti-invariant).
  • Per-vendor hardening: ServiceNow {value, link} reference-field shapes; BigPanda against the real v2 API (fixture-shaped today).
  • A live A2A transport for RemediationAdapter — deliberately reserved; an approved routed write degrades to an error Invocation rather than crashing, but the transport is fictional.
  • Earned autonomy — the ask→allow ladder — explicitly deferred.
  • Per-intent queries_by and registry-carried request/response templates (CapabilitySpec has no such fields).

13What breaks if you break it

The boundary’s guarantees, each with its enforcement point and its failure mode — the fastest way to review a change to capability/.

InvariantEnforced atIf violated
Gate before fetch — a blocked call returns before Source.fetch layer.py serve(): resolve → _gate → fetch → normalize A denied write causes a vendor-side effect — the audited write-before-gate bug returns: production mutation without consent.

The one-sentence summary of this page. Judgment picks the verb; the layer gates it; one seam fetches it; a pure fold types it; and the record of what happened — including nothing happened and it failed — is kept distinct, durable, and honest, so the evidence arithmetic downstream never stands on a lie.
