Investigation Workbench

Design & Architecture

A governed, human-in-the-loop engine that investigates an incident and produces an evidence-backed root cause. It is built to be generic and pluggable: the reasoning is done by a large language model behind a swappable interface, the tools are swappable adapters, and the problem domain is a small configuration file — so the same engine serves many kinds of investigation without change.

01The idea, in one minute

Think of the system as a disciplined investigator working alongside a person. The investigator (an LLM) is good at judgment — reading signals, forming theories, deciding what to check next. But judgment alone is risky: it can hallucinate, contradict itself, or act without proof. So the LLM is placed inside an engine that enforces the rules of good investigation.

Plain version. The LLM decides — which theory, which tool, what the root cause is. The engine checks and records — it validates every step against a fixed vocabulary, keeps one tamper-proof log, and blocks anything that would change a live system until a human approves it. You get the creativity of a model with the accountability of a system.

🧠 The LLM decides

Forms theories, picks tools, weighs evidence, names the root cause. Any capable model can be plugged in — it sits behind one interface.

⚙️ The engine governs

Validates each step, keeps the single source of truth, applies the rules for advancing, and never lets a fix run on its own.

🙋 The human approves

Before anything changes a real system, the investigation pauses and shows the proposed action plus the evidence behind it.

03How one step works — the phase loop

An investigation moves through a fixed sequence of phases. Every phase runs the same five-step loop, which is why the whole engine stays small and predictable.

1 · Plan

The LLM reads the current picture and produces a structured plan: which tools to call and what it concludes. Output is typed, never loose text.

2 · Reduce

The plan is checked against the fixed vocabulary — is this a known entity type? a legal relationship? a valid measurement? Anything illegal is rejected; the rest is accepted.

3 · Fold

The validated result is written into the picture, the theories, and the log — through one function. Nothing else in the system is allowed to write.

4 · Gate & 5 · Journal

The engine decides whether the phase may advance (enough evidence? a rival ruled out?), then appends the whole step to the log. The next phase is chosen from the verdict.

04How the planner is designed

The planner is the heart of the "pluggable" design. It is defined as a small contract, not a specific implementation, so the reasoning engine can be swapped freely.

The planner is an interface with interchangeable implementations
Enginecalls plan(context)
→
Planner contractplan(context) → PlanOutput
↓ any implementation that satisfies the contract
LLM plannerprompts a model, repairs its output
Scripted plannerfixed steps, for testing
your planneranything conforming

What the planner receives — a read-only snapshot of the investigation: the incident, the current picture of the system, the theories so far, the goal of this phase, and any feedback from the last step. What it returns — a typed PlanOutput: the tools to call, any conclusions as typed operations, a short narrative, and a proposed verdict (advance, repeat, or done).

# the contract — the engine depends only on this shape
class Planner:
    def plan(self, context) -> PlanOutput: ...

# PlanOutput is structured, never free text
PlanOutput(
    calls   = [ ... ],   # tools to invoke this phase
    ops     = [ ... ],   # typed conclusions (entities, facts, theories)
    narrative = "...",   # a human-readable summary
    verdict = advance | repeat | done,
)
Why this matters. Because the engine only knows the contract, the model behind it is a plug. The LLM planner and a deterministic scripted planner are used interchangeably — the model for real investigations, the script for repeatable tests — with zero change to the engine, the rules, or the data model.

Making a model's output safe: reject-and-repair

An LLM can produce a plan that references something outside the allowed vocabulary. Instead of crashing or blindly trusting it, the LLM planner runs a small set of guards that correct or drop the offending part and keep the rest. So a slightly-wrong plan becomes a valid one, and a good plan passes untouched. A hard guardrail also caps how many times a phase may repeat, so an investigation always makes progress.

05The data model — a typed picture of the system

Everything the investigation learns is stored as a small, strict graph. Its vocabulary is closed — a fixed catalogue of entity types and relationship types — so the LLM classifies into known shapes rather than inventing them, and every choice can be validated.

Entities & relationships

The things in the system (services, databases, hosts, changes…) and how they connect (depends-on, runs-on, caused-by…). This is the picture the investigation reasons over.

Facts

Timestamped, sourced observations — "this was true of that thing over this window." Facts are never edited; a newer value supersedes the old one, so you can always ask "what did we know, and when?"

Events

Point-in-time occurrences — a deploy, a restart, a config change. The moments that anchor a timeline.

Provenance — where each thing came from

Every entity, fact, and relationship carries a small, uniform envelope: which tool reported it and when it was first and last seen. This is what lets the interface show, for any node, "fetched from monitoring · first seen 14:05" — trust you can see, not take on faith.

Theories — the reasoning ledger

Competing explanations live in a ledger. Each theory keeps both the evidence for it and the evidence against it, plus the causal chain it implies. The engine can require genuine differential diagnosis — a rival must be ruled out, or the leading theory challenged — before an investigation is allowed to conclude.

06Design principles

Pluggable reasoning

The model sits behind one contract. Swap it, and the engine, rules, and data model are untouched.

Pluggable tools

Every tool is a small adapter with a uniform shape. Add, remove, or replace one without touching the others.

Pluggable domain

A problem domain is a lean configuration — the phases, goals, and gates. The graph vocabulary and tools are shared across domains.

One source of truth

The append-only journal is authoritative; the picture and the ledger are rebuilt from it. Replay must reproduce the exact same state.

One write path

A single function performs every change to state, so the system can pause with a change computed-but-not-applied while a human decides.

Human-gated consequences

Reading is free; changing a live system is not — it always stops for explicit approval, with the evidence shown.
