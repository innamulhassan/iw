01The idea, in one minute

A capable model is good at judgment — reading signals, forming theories, deciding what to check next. Judgment alone is risky: it hallucinates, contradicts itself, and acts without proof. So the model is placed inside an engine that enforces the mechanics of good investigation, and the engine is configured by a playbook that owns the domain method.

Plain version. The LLM decides — which theory, which tool, what the root cause is — but only through a closed grammar of typed operations. The engine checks, folds and records — it validates every op against a fixed vocabulary, applies the survivors through one write path, weighs the evidence itself, and keeps one append-only journal that can rebuild the whole run bit-for-bit. The human approves — nothing write-shaped touches a live system until a person sees the proposed action and the evidence behind it.

🧠 The planner judges

Forms hypotheses, picks tools, proposes a verdict. Live LLM or deterministic script — interchangeable behind one Planner Protocol (runtime/planner.py).

⚙️ The engine governs

Validates (materialize), writes once (apply_delta), gates the verdict (check_gate), journals the full delta, routes the next phase. Owns arithmetic, never judgment.

🙋 The human approves

Every write-effect capability stops at a gate: the session suspends before the write exists, shows action + hypothesis + evidence, and records approve / refine / deny durably.

Underneath, everything the run learns is one kind of record: a typed Assertion about an entity or edge — five species (identity · descriptor · state · reading · event) on one provenance envelope, held in a bi-temporal graph. Facts, events and node properties are read views over that one store, not separate stores. That is the Entity-Assertion model, and it is what doc 02 · Data Model covers in depth.

02The three authors

Every behavior in the system belongs to exactly one of three authors. The rule is stated in the code itself (domain/playbook.py): “playbook = WHAT/WHEN; engine owns only mechanics/arithmetic — no tuning constant lives in engine code.”

Playbook — WHAT & WHEN

playbooks/incident.yaml → Playbook: the phases (PhaseSpec), their gates (GateSpec), the transitions (on_verdict), every tunable (Tunables, extra="forbid" — a stale key is a loud load error), role bindings (symptom_node, terminal_phase, writes_allowed), and the live prompt Doctrine — nine verbatim prompt fragments. A method change is a playbook edit, not an engine release.

Engine — MECHANICS

runtime/ · graph/ · journal/ · hypothesis/: validation, folding, replay, gate arithmetic, belief arithmetic. INV-9: the engine holds zero tuning constants — reliability defaults, confidence bands, decay curves, ceilings all ride in Tunables; the reducer fills a measured fact’s missing reliability from the playbook’s per-source map.

Planner — JUDGMENT

Emits PlanOutput: capability calls + typed ops + a narrative + a proposed verdict. Its only structural output is the 11-op grammar (domain/operations.py) — “prose is a field, not the medium.” The verdict is a proposal; the controller is the routing authority (R-C1).

Pluggability, stated precisely. Three seams swap: the planner (anything satisfying Planner.plan(ctx) → PlanOutput), the tools (an Adapter is a pure normalize(raw) → ops pair; the one side-effecting seam is Source.fetch() — fixtures and live MCP/REST swap under identical adapters), and the domain (a playbook file). Storage is deliberately not pluggable (owner directive): runtime/store.py InvestigationStore is a simple, concrete file store — journal.ndjsonl (append-only, fsync’d) + graph.json (watermark-verified cache) + meta.json — no StorageBackend Protocol, no second implementation. It is contained: every byte of investigation I/O goes through this one module, so a later DB swap is a one-module change. Likewise the reducer and the fold are law, not extension points.

04How one phase runs — the loop

Every phase, batch or interactive, runs the same six-step loop. That uniformity is why the whole controller is a few lines and why adding a phase — or a whole playbook — needs no new plumbing (R-P1: one uniform PhaseResult contract).

1 · Plan — planner.plan(ctx)

Input is a read-only PlanContext: subject, phase spec + goal, graph_view (render_slice(graph, anomaly_ref) — the full graph capped at 40 nodes, symptom ordered first, focus a highlight never a filter), the ranked hypotheses with their engine-earned scores, tunables, last gate’s gate_feedback, last step’s rejections, engine-computed correlations. Output is PlanOutput{calls, ops, narrative, verdict, next_actions}, extra="forbid". This is the only suspension point: an interactive write gate suspends by raising out of plan() — nothing claimed, nothing to roll back.

between 1 and 2 — serve the tools

The engine claims the phase seq (journal.reserve_seq() — after plan(), so a suspended gate burns nothing), then serves each call: layer.serve(call, allow_write=spec.writes_allowed), gate-first — a blocked write returns before any fetch. Every call — data, empty, error, blocked — is journaled as a kind="invocation" entry sharing the phase’s seq. Tool ops precede planner ops; the combined list is truncated to the phase’s op_ceiling.

2 · Reduce — materialize(ops, seq, graph, tunables, …)

The authoritative validator (graph/reducer.py): names canonicalize through the dictionary (identity / N→1 merge / unit-keyed split); an unknown name lands provisional under x.<source>.<native> instead of vanishing; applies_to is enforced; edges must be registry-legal (generic-CI airlock: structural spine only, origin forced to discovered, confidence × discovery_penalty); identity resolves before it twins (alias backbone, provisional mints, degenerate ids refused); a NoEvidence op naming an intent whose last call errored or was blocked is rejected — an error observed nothing. Partial-accept: each illegal op becomes a Rejection{op_index, op_kind, reason}; the rest apply.

3 · Fold — apply_delta(result, seq, graph, store)

The only writer of the projections (graph/fold.py). In-delta order is load-bearing for replay: nodes → facts → events → edges, then retractions (after additions), then remaps last. Then HypothesisStore.apply(hypotheses_updated, seq), then SUPPORTS/REFUTES evidence edges are recomputed from each touched hypothesis’s fact-id lists — full reconciliation, stale edges tombstoned — so the graph can never disagree with the store. The engine folds before it gates, so the gate judges the updated store.

4 · Gate — check_gate(spec, result, store, tunables)

Only a proposed ADVANCE is gated; first failure wins: every produces_required field non-empty (an honest NoEvidence fact counts — R-P2) → min_facts over facts_added → require_confidence_gate ⇒ store.promotion_ok(tunables) (INV-8: earned score ≥ confidence_gate, zero alive unrefuted rivals, the delta floor — an LLM-set CONFIRMED grants no bypass) → require_refutation ⇒ a refuted rival exists or the leader carries refuting facts. Failure downgrades to REPEAT with the exact gate_reason, fed to the next plan. The engine’s repeat cap (max_retries) then forces an ADVANCE after too many consecutive REPEATs, so a live model cannot stall the run.

5 · Journal — journal.append_phase(seq, result)  ·  6 · Route — next_phase(spec, verdict)

The phase entry stores the full PhaseResult as its delta — the replay payload (R-J1); rebuild(journal) replays phase entries only, every other kind is record, not state. Then the gated verdict routes through the playbook’s on_verdict table: DONE ends the run; an unmapped REPEAT re-enters the same phase; the planner’s next_actions never routes.

05The phase model

A phase is data, not code: PhaseSpec{id, goal, allowed_intents, produces_required, gate, on_verdict, writes_allowed} in the playbook, plus role bindings — symptom_node (what frame anchors), terminal_phase (what closes the run), writes_allowed (where the human gate arms). The engine references roles and transitions, never phase names. The shape of an investigation:

The investigation shape — five stages, one gated write
Frameseed symptom · topology · changes
→
Investigate — the loophypothesize ⇄ evidence ⇄ refute
→
ACT ⚑human-gated writes
→
Verifydid it clear? chain holds?
→
Closepostmortem from the journal

The loop exits only through the evidence gate — require_confidence_gate + require_refutation ⇒ promotion_ok (INV-8). ⚑ ACT is the one write-armed stage: every write-effect call stops for approve / refine / deny, and the decision is a durable journal entry.

Today in code. The shipped playbooks/incident.yaml still declares seven phase ids — frame, triage, hypothesize, investigate, remediate, verify, close — and Phase is still an engine enum. They collapse onto the shape above: triage dissolves into frame and the loop’s first turns; remediate is the gated ACT (the only writes_allowed: true phase). Finishing that dissolution — the Phase enum leaving the engine so phases are fully playbook-declared — is the P7 step now settling. Read the five-stage strip as the model; read the seven ids as today’s declaration of it.

08The planner contract — and reject-and-repair

The engine depends on one shape. Anything that satisfies it is a planner; the live LLM and the deterministic script are used interchangeably, which is why the golden suite proves things about production.

# runtime/planner.py — the contract, verbatim shapes
@runtime_checkable
class Planner(Protocol):
    def plan(self, ctx: PlanContext) -> PlanOutput: ...

@dataclass
class PlanContext:                    # read-only judgment input
    subject: SubjectRef
    phase: Phase
    phase_spec: PhaseSpec
    goal: str
    graph_view: dict                  # render_slice(graph, anomaly_ref)
    hypotheses: list[dict]            # ranked; "confidence" = the ENGINE-earned score
    tunables: Tunables
    gate_feedback: str | None         # WHY the last gate downgraded (GAP 3)
    messages: list[dict]              # operator steering (session-injected)
    rejections: list[Rejection]       # last step's dropped ops + reasons (R-K2)
    correlations: list[dict]          # engine-computed change→onset hints (R-J2)

class PlanOutput(BaseModel):          # extra="forbid"
    phase: Phase
    calls: list[CapabilityCall]       # tool invocations -> data ops
    ops: list[Operation]              # planner-direct typed ops (the 11-op grammar)
    narrative: str                    # the ONLY prose channel
    verdict: PhaseVerdict             # a PROPOSAL — the controller gates it
    next_actions: list[str]           # advisory; journaled, never routes

ScriptedPlanner — the deterministic twin

Replays a fixed PlanOutput list. Fails loudly: exhaustion raises, and a scripted step whose phase mismatches the engine’s raises — a mis-ordered script can’t silently drift. Drives the scenario goldens.

LivePlanner — the real model

System prompt = the playbook’s Doctrine fragments interleaved with engine hard rules, closed by validity lists derived from the enums (op kinds, sources, statuses) — a derived list cannot drift the way a hand-restated one did. User prompt carries the catalog, wired tools, gate spec, dropped-op feedback, steering, and the graph view.

Reject-and-repair — making untrusted output safe

The model’s JSON is untrusted; the planner is the first guard and the reducer the second, authoritative one. Repairs never raise (INV-7 — one malformed op must never kill a live session):

  • a non-dict payload is repaired to an empty plan; every repair is recorded on LivePlanner.repairs and the per-phase trace;
  • calls with off-catalog intents are dropped with a reason; valid ones become CapabilityCalls;
  • each op is parsed to a typed Operation (add_node, add_fact, propose_hypothesis, merge, retype, …); any exception becomes a drop-with-reason, never a crash;
  • _belief_channel repairs the belief invariant (R-C4) before the reducer would raise: an LLM-sourced fact carries confidence only, a measured one reliability only;
  • an unknown verdict status is coerced (recorded), and the coarse confidence_level band maps through tunables.confidence_band — the planner holds no constants of its own (INV-9).

What slips past the planner is rejected by the reducer as a journaled Rejection and fed back into the next PlanContext.rejections — the bounded repair loop (R-K2): the model is told why each op vanished instead of re-emitting into silence. There is deliberately no planner-local repair cap; progress is bounded where it belongs — the engine repeat cap (max_retries forces ADVANCE), the per-phase op_ceiling, max_steps (default 60), and the session’s diagnosable max-steps close.

09The invariants that carry the design

The journal is the source of truth (R-J1)

Every phase entry stores its full delta; rebuild(journal) reproduces Graph + Hypothesis bit-for-bit; the on-disk graph cache is served only when its seq watermark matches the journal head, else rebuilt and self-healed.

One mutation seam

Only fold.apply_delta writes the projections; in-delta order (nodes → facts → events → edges → retractions → remaps last) is fixed so replay is order-deterministic. No graph mutator has a caller outside graph.py/fold.py.

Promotion is earned (INV-8)

promotion_ok: the leader’s engine-earned score ≥ confidence_gate, zero alive unrefuted rivals (any band), and the delta floor. Popperian: rivals must be refuted, not out-scored. The model’s CONFIRMED is a proposal, never a bypass.

No constants in the engine (INV-9)

Every knob — bands, reliability defaults, skew bounds, decay, ceilings, penalties — lives in Tunables (extra="forbid"). The reducer fills a measured fact’s missing reliability from the playbook’s per-source map; adapters carry none.

Nothing observed is erased (the airlock)

An unknown name lands provisional as x.<source>.<native>; an off-shape known name lands provisional plus a journaled rejection; a generic-CI edge is admitted on the structural spine only, origin forced to discovered, confidence × discovery_penalty.

Error ≠ empty ≠ blocked

The boundary outcome is load-bearing: only a clean-empty read may become the honest null fact no_evidence:<intent>; a NoEvidence op naming an errored/blocked intent is rejected — fabricated negative evidence is the failure this kills.

One seq space, append-at-event

The phase seq is claimed only after plan() returns — a suspended gate burns nothing; invocation entries annotate their phase’s seq; the journal is gap-free under suspend / approve / deny.

Writes stop for a human

serve() is gate-first — a blocked write returns before any fetch; the session suspends before the write delta exists; the question (gate_opened) and the answer (gate_decision) are durable journal kinds. (Per-call approval tokens are deferred work: batch runs gate on writes_allowed alone.)
