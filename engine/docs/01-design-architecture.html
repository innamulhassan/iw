Investigation Workbench · iw

Design, Architecture & Data Model

A governed, human-in-the-loop, LLM-orchestrated incident root-cause engine. The LLM owns all judgment; the engine owns all mechanics. Every fact, hypothesis, and decision is typed, sourced, and reconstructable from an append-only journal.

01What it is

iw turns a raw incident signal (a ServiceNow major incident) into a confirmed, evidence-backed root cause — the way a strong SRE would, but as a governed state machine. An LLM plans each phase and chooses tools; a deterministic engine validates, folds, gates, and journals every step. Side-effecting fixes are always paused for human approval.

LLM owns judgment

Which hypothesis, which tool, when to advance, what the root cause is — authored live by the model, never scripted.

Engine owns mechanics

Phase routing, type validation, the single write seam, the confidence gates, the audit journal — deterministic and testable.

Human owns consequence

Any WRITE (a remediation) suspends at an approval gate carrying the proposed action + its evidence chain.

02The engine — a phased orchestrator

One phase = one loop iteration: plan (typed ops) → reduce (validate + materialise) → fold (apply to graph + ledger) → gate (guard the verdict) → journal → route to the next phase. No LangGraph, no agent sprawl — the uniform contract makes the whole controller a few lines and fully deterministic given a scripted planner.

FRAME→ TRIAGE→ HYPOTHESIZE→ INVESTIGATE→ REMEDIATE ⚑→ VERIFY→ CLOSE

⚑ REMEDIATE is the one writes_allowed phase — a WRITE-effect call there suspends the run at the human approval gate.

Both planners implement one Planner protocol, so they are interchangeable. The engine, reducer, ledger, and journal are identical on both paths — only the judgment author changes.

03The data model — a typed incident graph

The graph is the reasoning substrate. Its vocabulary is a closed, typed registry (43 node types, 36 edge types) so the LLM classifies into known types rather than inventing them — and the reducer validates every choice against the same specs the model was shown.

Three storage shapes (the core ruling)

Static property

Immutable / identity attributes on a node (app_id, sha, env). Never time-varying.

Fact bi-temporal

A reified, sourced observation that was true of a thing over a window. Never mutated — a newer value supersedes (closing valid_to); a wrong one is retracted.

Event

A point-in-time occurrence (deployed, evicted, index_dropped) with an occurred_at.

Anything time-varying is a Fact, not a mutable property — this is what makes "reconstruct the graph as of incident-start" answerable.

The provenance envelope (the W's)

One shared envelope — source + two clocks — is attached identically to Node, Fact, and Edge. A Fact already answers who (source), what/where (subject_ref, optional where), when (valid_from / valid_to valid-time, observed_at transaction-time) and why (evidence). Node provenance is a pure projection — a node's source and first_seen are derived from its earliest fact (the AppDynamics/Dynatrace entity model), so it is deterministic under journal replay with no redundant storage.

# domain/fact.py — the reified, bi-temporal, sourced observation
Fact(subject_ref, predicate, value, unit, where?,
     valid_from, valid_to?, observed_at,          # two clocks: valid-time vs transaction-time
     source, confidence? | source_reliability?,   # exactly one belief channel (R-C4)
     evidence[], supersedes?, state)

Belief-channel invariant: an inferred fact (source=llm) must carry a confidence; a measured fact (any tool) must carry a source_reliability. Never both, never neither.

Causation as a refutable layer

Structural edges (depends_on, runs_on, consumes_from…) are the durable spine. Causal edges (caused_by, supports, refutes, correlated_with) are a separate inferred layer that carries mandatory confidence + evidence and can be tombstoned when refuted — inferred relationships can be wrong, and the model says so. Evidence edges (supports/refutes) are derived projections the fold recomputes from each hypothesis's canonical fact-id lists, so the graph view can never disagree with the ledger.

04The hypothesis ledger (Popperian)

A hypothesis is both a ledger entry and a graph node, so causal edges can reference it. It holds its supporting and refuting facts and a typed causal chain. Investigation is genuinely differential: a phase gate can require that a rival was ruled out or the leader was challenged before advancing. A status the reasoner marks CONFIRMED clears the gate even at a MED confidence band — conviction is signalled by status, the band is only the fallback.

Hypothesis(statement, status, confidence, root_candidate,

05Interactive session & workbench

The interactive layer is pure composition over the deterministic fold — no engine edits:

  • Write-gate by composition. A wrapped planner computes the plan, and if a writes_allowed phase's plan carries a WRITE call it raises a suspend before the write is served — the engine unwinds with nothing applied. On approve it re-runs with the write intact; on deny it records the denial as feedback and the next plan replans.
  • Background drive. For the live path a phase is seconds of LLM latency, so the drive runs off the HTTP thread; POST /sessions returns immediately and the SSE stream shows incremental progress.
  • An ordered event stream (phase_started · reasoning · capability_call · graph_delta · ledger_delta · gate_opened · user_message · session_state) derived purely from what the engine recorded — the React workbench folds it into the panes.

The workbench (Vite / React / TS) renders a two-way investigation chat, a layer-laned live graph with per-node origin/source/provenance badges, an expandable hypothesis ledger whose facts cross-highlight the graph, a decision-prompt approval gate, and a deep journal — all fed by the SSE store, nothing invented.

06Design invariants

Journal is truth

Graph + ledger are projections; replay must equal the live state, byte-for-byte.

One write seam

apply_delta is the only mutator; an interactive gate holds a computed-but-unapplied delta.

Closed, typed vocabulary

The LLM classifies into 43/36 known types; an off-catalog op is rejected or repaired, never silently accepted.

Append, never overwrite

Facts/edges supersede or retract; two clocks stay distinct (valid-time vs observed-time).

Human-gated side effects

A WRITE cannot execute outside an approved gate — enforced in the capability layer, gate-first.

Judgment ≠ mechanics

Swap the planner (mock ⇄ live) with zero engine change; the CI net stays deterministic.
